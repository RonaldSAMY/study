## The hook: there isn't time to be perfect

A delivery van has 30 stops. The number of possible routes is `30!` — more than the number of atoms in
your body, by a wide margin. No computer will ever enumerate them. Yet the driver needs a *good* route in
under a second. So we give up on the guaranteed-best answer and instead **search smartly**: start with any
route and keep nudging it toward something better. That's the whole philosophy of heuristics — trade a
proof of optimality for an answer that's fast and usually within a few percent of the best.

<Callout type="intuition" title="The one-sentence idea">
Local search starts at some solution and repeatedly moves to a better neighbour; the art is escaping the
"local peaks" where every neighbour looks worse but the global best lies across a valley.

</Callout>

## Intuition: climbing a foggy hill

Picture the space of solutions as a landscape: each point is a candidate, its height is how good it is.
You want the highest peak. **Hill climbing** is the obvious strategy — look at nearby solutions, step to
the best one that's higher, repeat. It's fast and simple. Its fatal flaw: it stops at the first peak it
reaches, even if a taller mountain stands just beyond a dip. That trap is a **local maximum**.

The cure is to *sometimes go downhill on purpose*. **Simulated annealing** borrows from metallurgy:
heat a metal and its atoms jiggle freely; cool it slowly and they settle into a low-energy crystal. The
algorithm keeps a "temperature" that starts high — early on it readily accepts worse moves, letting it
cross valleys — and cools over time, so it gradually stops wandering and settles near the best peak it
found. The acceptance rule is beautifully simple: an improving move is always taken; a worse move by `Δ`
is accepted with probability `exp(Δ / T)`, which is high when `T` is large and shrinks as it cools.

Other members of the family vary the same theme: **tabu search** keeps a short memory of recent solutions
so it won't cycle; **genetic algorithms** evolve a whole population with crossover and mutation; **beam
search** keeps the best `k` partial solutions at each step.

## How it works

Hill climbing on a one-dimensional bump function, maximizing:

```
current = x0, score = f(x0)
repeat:
    neighbours = { x0 - step, x0 + step }
    best = the highest-scoring neighbour
    if f(best) <= score:  STOP   ← no neighbour is better: local maximum
    else: current = best
```

Simulated annealing changes only the accept rule:

```
T = T0
repeat:
    next = random neighbour of current
    Δ = f(next) - f(current)
    if Δ > 0 or random() < exp(Δ / T):   accept → current = next
    T = T * coolingRate         ← e.g. 0.99 each step
until T is tiny
```

When `T` is large, `exp(Δ / T)` is close to 1 even for a bad move, so the walker explores. As `T → 0`, only
improving moves survive, so it converges.

function anneal(f, x0, opts) {
  const { T0 = 5, cool = 0.97, steps = 400, rng = Math.random } = opts || {};
  let x = x0, best = x0, T = T0;
  for (let i = 0; i < steps; i++) {
    const next = x + (rng() * 2 - 1) * 1.2;       // random neighbour
    const d = f(next) - f(x);
    if (d > 0 || rng() < Math.exp(d / T)) x = next; // accept (maybe downhill)
    if (f(x) > f(best)) best = x;                   // remember the best seen
    T *= cool;                                       // cool down
  }
  return { x: best, value: f(best) };
}

## The real implementation

The course file's hill climbing and simulated annealing are generic over any solution type:

```ts
// /dsa/17-advanced-algorithms/heuristic-algorithms.ts
function simulatedAnnealing<T>(
    initial: T,
    evaluate: (s: T) => number,
    getRandomNeighbor: (s: T) => T,
    opts: { initialTemp?: number; coolingRate?: number; minTemp?: number; maximize?: boolean } = {}
) {
    const { initialTemp = 1000, coolingRate = 0.995, minTemp = 0.01, maximize = false } = opts;
    let current = initial, currentScore = evaluate(current);
    let best = current, bestScore = currentScore;
    let temperature = initialTemp;

    while (temperature > minTemp) {
        const neighbor = getRandomNeighbor(current);
        const neighborScore = evaluate(neighbor);
        const delta = maximize ? neighborScore - currentScore : currentScore - neighborScore;

        // accept if better, or probabilistically if worse
        if (delta > 0 || Math.random() < Math.exp(delta / temperature)) {
            current = neighbor; currentScore = neighborScore;
            if (maximize ? currentScore > bestScore : currentScore < bestScore) {
                best = current; bestScore = currentScore;
            }
        }
        temperature *= coolingRate;   // cool down
    }
    return { solution: best, score: bestScore };
}
```

Run the full file: `npx tsx 17-advanced-algorithms/heuristic-algorithms.ts`. It also includes hill
climbing, tabu search, genetic algorithms, beam search, and TSP via nearest-neighbour plus 2-opt local
search.

## Complexity

Heuristics trade the *guarantee* of optimality for a controllable running time. There is no "n" that
bounds quality — only the iteration budget you choose:

$$
T = O(\text{iterations} \times \text{cost of evaluating a neighbour}), \qquad \text{quality: approximate}.
$$

| Method | Escapes local optima? | Memory | Determinism |
|--------|----------------------|--------|-------------|
| Hill climbing | no | $O(1)$ | deterministic |
| Simulated annealing | yes (via randomness) | $O(1)$ | randomized |
| Tabu search | yes (via memory) | $O(\text{list})$ | mostly deterministic |
| Genetic algorithm | yes (via population) | $O(\text{pop})$ | randomized |

The headline contrast: hill climbing is fast but gets stuck; annealing spends extra iterations buying the
ability to cross valleys.

## Common pitfalls & interview tips

- **Cooling too fast.** Drop the temperature too quickly and annealing degenerates into hill climbing,
  trapped at the first peak. Too slow and you waste iterations. The schedule is the key tunable.
- **Forgetting to track the best-so-far.** Annealing's *current* solution can wander downhill at the end;
  always keep a separate record of the best solution ever seen and return that.
- **A neighbourhood that's too big or too small.** Tiny steps stall; huge steps turn the search into random
  guessing. The move operator must make meaningful but local changes.
- **Expecting reproducibility.** Randomized heuristics give different answers each run unless you fix the
  seed — essential for debugging and for fair benchmarking.
- **Interview cue:** if a problem is NP-hard or the input is huge and "approximate is fine", propose a
  greedy construction *plus* local search, and mention annealing to escape local optima.
