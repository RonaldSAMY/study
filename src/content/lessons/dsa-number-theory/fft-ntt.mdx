## The hook: multiply two 10,000-digit numbers before lunch

Grade-school multiplication of two $n$-digit numbers takes $n^2$ single-digit products — a million for
1000 digits, a hundred million for 10,000. Yet your computer multiplies enormous integers almost
instantly. The secret is a change of viewpoint borrowed from signal processing: a number (or polynomial)
is easier to *multiply* when you describe it by its values at a handful of magic points rather than by its
digits. The Fast Fourier Transform switches between those two views in $O(n \log n)$, and that single idea
collapses the whole $n^2$ wall.

<Callout type="intuition" title="The one-sentence idea">
A degree-$n$ polynomial is pinned down by its values at $n+1$ points, and in *value form* multiplication
is just pointwise products — so evaluate, multiply, and interpolate back, each in $O(n \log n)$.
</Callout>

## Intuition

There are two ways to describe a polynomial: by its **coefficients** $[a_0, a_1, \dots]$, or by its
**values** at some chosen points $[(x_0, A(x_0)), (x_1, A(x_1)), \dots]$. Multiplying in coefficient form
is a convolution — $O(n^2)$. But in value form, $C(x_i) = A(x_i) \cdot B(x_i)$: one multiplication per
point, $O(n)$ total. The catch is converting between the two views cheaply.

The FFT makes both conversions $O(n \log n)$ by evaluating at very special points — the **$n$-th roots of
unity** $\omega^k = e^{2\pi i k / n}$, evenly spaced around the unit circle (the clock, again!). Their
magic is a self-similar symmetry: splitting a polynomial into its even- and odd-indexed coefficients lets
you compute the size-$n$ transform from two size-$n/2$ transforms, combined by a cheap **butterfly**
$A(x) = E(x^2) + x \cdot O(x^2)$. That divide-and-conquer recursion is the entire algorithm.

The **NTT** plays the identical game but over integers modulo a special prime, using a primitive root in
place of $\omega$. Same butterflies, no floating-point error — the right choice when you need *exact*
results.

## How it works

Multiply $A = 1 + 2x + 3x^2$ by $B = 4 + 5x$ via the transform round-trip:

```
resultLen = 3 + 2 - 1 = 4  ->  pad both to n = 4 (next power of two)
1. FFT(A), FFT(B): evaluate both at the 4th roots of unity   [O(n log n)]
2. pointwise multiply the value vectors                       [O(n)]
3. inverse FFT: interpolate back to coefficients              [O(n log n)]
result = [4, 13, 22, 15]  =  4 + 13x + 22x^2 + 15x^3
```

Check by hand: $(1 + 2x + 3x^2)(4 + 5x) = 4 + 13x + 22x^2 + 15x^3$. The FFT never "expanded" the product
directly — it went out to value space and back.

  }

## The real implementation

The iterative in-place FFT and NTT from the course file share the exact same butterfly structure — the
NTT just swaps roots of unity for modular arithmetic so results are integer-exact:

```ts
// /dsa/22-number-theory/fft-ntt.ts
export function fft(a: Complex[], invert = false): void {
  const n = a.length;
  for (let i = 1, j = 0; i < n; i++) {          // bit-reversal permutation
    let bit = n >> 1;
    for (; j & bit; bit >>= 1) j ^= bit;
    j ^= bit;
    if (i < j) [a[i], a[j]] = [a[j], a[i]];
  }
  for (let len = 2; len <= n; len *= 2) {       // stage by stage
    const ang = (2 * Math.PI / len) * (invert ? -1 : 1);
    const wlen = Complex.fromPolar(1, ang);
    for (let i = 0; i < n; i += len) {
      let w = new Complex(1);
      for (let j = 0; j < len / 2; j++) {       // the butterfly
        const u = a[i + j];
        const v = a[i + j + len / 2].mul(w);
        a[i + j] = u.add(v);
        a[i + j + len / 2] = u.sub(v);
        w = w.mul(wlen);
      }
    }
  }
  if (invert) for (let i = 0; i < n; i++) a[i] = a[i].div(n);
}
```

## Complexity

The transform recurses over $\log n$ stages of $n$ work each; multiplication is three transforms plus a
pointwise pass:

$$
T_{\text{FFT}}(n) = O(n \log n), \qquad T_{\text{multiply}}(n) = O(n \log n).
$$

| Task | Naive | FFT / NTT |
|------|-------|-----------|
| Polynomial / big-int multiply | $O(n^2)$ | $O(n \log n)$ |
| Convolution of two length-$n$ signals | $O(n^2)$ | $O(n \log n)$ |
| One transform | $O(n^2)$ (naive DFT) | $O(n \log n)$ |

FFT wins only once $n$ is large enough (roughly $n \ge 64$) to pay off its constant factor — below that,
schoolbook multiplication is faster. Use NTT when you need exact integers modulo a prime; use FFT for
real-valued signals where small floating-point error is fine.

</ExampleCard>

<ExampleCard domain="Machine learning" icon="🧠">

$O(n^2)$ DP into $O(n \log n)$ with FFT, a trick that shows up in divide-and-conquer DP optimization.
</ExampleCard>

## Common pitfalls & interview tips

- **Pad to a power of two, and to at least $\deg A + \deg B + 1$.** Too small and the result wraps
  (circular convolution) instead of the linear product you wanted.
- **Floating-point error.** FFT accumulates rounding; `Math.round` the real parts at the end, and switch
  to NTT when the numbers are large or must be exact.
- **Bit-reversal and twiddle direction.** The inverse transform negates the angle *and* divides by $n$.
  Forget the $1/n$ and every coefficient is scaled wrong.
- **Interview cue:** "multiply two big numbers," "count pairs/sums by convolution," or "match a pattern
  with wildcards" are FFT/NTT problems — recognize the convolution and reach for the transform.
