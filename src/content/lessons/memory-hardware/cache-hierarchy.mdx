## The hook: two loops, a 10x difference

You write a game and it stutters. A profiler shows the hot spot: a loop walking through every
enemy in the level. You swap one detail — how the enemy data is laid out in memory — and the *exact
same math* now runs many times faster. Nothing about the work changed. What changed is whether the
CPU could find the data **nearby** or had to go fetch it from far away. That gap between "near" and
"far" is the most important performance story in modern hardware, and it has a name: the **memory
hierarchy**.

<Callout type="intuition" title="The one-sentence idea">
Your CPU keeps small, blazing-fast copies of recently and nearby used data close by; if the data you
ask for is already close it is nearly free, and if it isn't you pay a long trip out to main memory.
</Callout>

## Intuition: a stack of stores, each bigger but slower

From the previous lesson you know RAM is one giant addressable array. But the CPU does not read RAM
directly for every operation — RAM is *slow* compared to how fast the CPU can compute. So hardware
designers build a ladder of storage, each rung bigger but slower than the one above it:

- **Registers** — a handful of slots inside the CPU. Instant.
- **L1 cache** — a few kilobytes, almost instant.
- **L2 cache** — hundreds of kilobytes, still very fast.
- **L3 cache** — a few megabytes, shared between cores.
- **Main memory (RAM)** — gigabytes, but far away and slow.

Here are rough, illustrative latencies (in CPU cycles) so the shape sinks in:

| Level | Size (rough) | Latency (~cycles) |
| --- | --- | --- |
| Register | bytes | ~1 |
| L1 | tens of KB | ~4 |
| L2 | hundreds of KB | ~12 |
| L3 | a few MB | ~40 |
| RAM | many GB | ~200 |

The jump from L1 to RAM is roughly **50x**. That is the cliff your stuttering loop fell off.

Two more ideas make the whole thing work. When the CPU fetches one byte from RAM it does not grab
just that byte — it grabs a whole **cache line**, a contiguous block of about **64 bytes**, and stores
the copy in cache. And it bets on **locality**:

- **Temporal locality** — if you used something just now, you will probably use it again soon.
- **Spatial locality** — if you used one address, you will probably use its neighbors soon.

A request the cache can satisfy is a **hit** (fast). One it cannot is a **miss** (you pay the slow
trip and a fresh line gets loaded).

## The formal definition

Treat each access as either a hit or a miss. If the fraction of accesses that hit is the **hit rate**
$h$, then the miss rate is $1 - h$, and the **average cost per access** is a simple weighted average:

$$
T_{\text{avg}} = h \cdot T_{\text{hit}} + (1 - h) \cdot T_{\text{miss}}.
$$

With the illustrative numbers $T_{\text{hit}} = 4$ and $T_{\text{miss}} = 200$ cycles, a sequential
loop with $h = 0.9$ costs about $0.9 \cdot 4 + 0.1 \cdot 200 = 23.6$ cycles per access, while a random
pattern with $h = 0.25$ costs about $0.25 \cdot 4 + 0.75 \cdot 200 = 151$ cycles. Same instructions,
roughly $6\times$ the time — purely from layout.

Spatial locality has a clean knob too. If a cache line holds $B$ elements and you read an array
straight through, you take **one** miss every $B$ elements, so the miss rate floor is about $1 / B$.
Bigger lines amortize each slow trip over more useful data — *as long as you actually use the
neighbors you dragged in*.

```c
// Cache-friendly: strides by 1, uses every byte of each line it loads.
long sum = 0;
for (int i = 0; i < n; i++)
    sum += data[i];        // neighbors data[i+1], data[i+2]... already in cache
```

<Callout type="why">
This is why "the same algorithm" can be fast or slow in a game engine. Looping over objects packed
contiguously (an array) streams cache lines and stays near 90% hits; chasing pointers scattered across
the heap (a linked list of objects) misses constantly. Big-O counts operations; the cache decides what
each operation actually costs.
</Callout>

## The same idea, four different worlds

A hierarchy of "fast and near" versus "slow and far," plus the trick of grabbing a whole batch at
once, shows up far outside computers.

<ExampleCard domain="Cooking" icon="🧑‍🍳">
A chef's *mise en place* is a cache. The cutting board and the small bowls within arm's reach are
**L1** — grabbing salt is instant. The pantry across the kitchen is **RAM**: a slow walk. A good chef
pulls everything a recipe needs to the counter first (loads the "line"), then cooks without a single
trip. A disorganized cook walks to the pantry for every pinch — constant misses.
</ExampleCard>

<ExampleCard domain="Woodworking" icon="🪚">
On the bench sits a tool tray with the chisels and the marking gauge you are using right now — a hit
is a quick reach. Anything not on the tray means a drive to the **hardware store**: the worst-case
miss. So you lay out the few tools this joint needs before starting, the same way the CPU pulls a
cache line in before it computes.
</ExampleCard>

<ExampleCard domain="Logistics" icon="📦">
A warehouse picker filling an order is access-pattern bound. When the items sit in **adjacent bins**,
one walk down the aisle grabs them all — sequential access, high hit rate. When the order is items
**scattered** across the building, the picker zig-zags everywhere, walking far more than they carry.
Smart warehouses *re-shelve* popular items together, exactly the way a cache keeps hot lines close.
</ExampleCard>

<ExampleCard domain="Office work" icon="🗄️">
The papers in your desk's top drawer are reachable in a second; documents in the **offsite archive**
take a day to retrieve. You naturally keep the file you are working on in the drawer (temporal
locality) and pull the whole folder, not one page, when you fetch it (spatial locality) — so you do
not order a retrieval again tomorrow.
</ExampleCard>

<Callout type="note">
Notice the universal trade-off: storage that is fast is always small and storage that is huge is
always slow. Nobody can give you "huge and instant," so every level — kitchens, warehouses, CPUs —
keeps a small fast copy of what is hot.
</Callout>
