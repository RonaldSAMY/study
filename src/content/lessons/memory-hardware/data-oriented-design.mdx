## The hook: the same loop, ten times faster

Two game studios write the *exact same* feature: move ten thousand enemies one step to the right each
frame. Same algorithm, same number of additions, same language. One runs at a smooth 120 FPS; the other
stutters at 12. The difference is not the code that runs — it is the **order the data sits in memory**.
One studio laid out its bytes so the cache stays full of useful numbers; the other drags mostly-unused
data through the cache on every pass. Welcome to **data-oriented design**.

<Callout type="intuition" title="The one-sentence idea">
Group the data your loop actually touches so it arrives together in a cache line — instead of storing
each object's fields side by side and paying to load the parts you never read.

</Callout>

## Intuition: the cache moves whole lines, not single fields

From the cache lesson, remember the golden rule: memory does not travel one byte at a time. When the
CPU reads an address, it pulls the entire surrounding **cache line** (typically 64 bytes) into fast
memory. If the next bytes you need are right there, you get a cache *hit*; if not, you pay a slow trip
to main memory — a cache *miss*.

Now picture a game entity with several fields: position `x`, `y`, `health`, and a `sprite` pointer.
The obvious layout is an **Array of Structs (AoS)**: store each entity's fields together, then the next
entity's, and so on — `x y health sprite | x y health sprite | ...`. It reads naturally. But suppose a
system loops over every entity and updates *only* `x`. Each cache line you load is mostly `y`,
`health`, and `sprite` — fields this loop never touches. You are paying full bandwidth to move data you
throw away.

The fix is **Struct of Arrays (SoA)**: store all the `x` values in one contiguous array, all the `y`
values in another, and so on — `x x x x ... | y y y y ... | ...`. Now the `x`-only loop walks a tight,
packed array. Every cache line is nearly all `x` values: few misses, full utilization. Same data, same
math, dramatically better cache behavior. This is the core idea behind **Entity Component Systems**
and data-oriented engines.

## The formal definition

Say each entity has $F$ fields, every field is $b$ bytes, and a cache line holds $L$ bytes. A loop reads
**one** field across $N$ entities. The useful data is always

$$
\text{useful} = N \cdot b \ \text{bytes.}
$$

With **AoS**, that field's values sit $F \cdot b$ bytes apart, so almost every line you load contains
just one wanted value surrounded by $F - 1$ unwanted fields. The bytes actually moved are about

$$
\text{loaded}_{\text{AoS}} \approx N \cdot F \cdot b, \qquad
\text{utilization} \approx \frac{1}{F}.
$$

With **SoA**, the field is contiguous, so a line of $L$ bytes carries $L / b$ useful values and you load
only

$$
\text{loaded}_{\text{SoA}} \approx N \cdot b, \qquad
\text{utilization} \approx 1.
$$

The two layouts in code are just a transpose of each other:

```cpp
// AoS: fields interleaved, one struct per entity
struct Entity { float x, y, health; Sprite* sprite; };
Entity entities[N];          // x y h s | x y h s | ...

// SoA: each field is its own packed array
struct World {
    float x[N], y[N], health[N];
    Sprite* sprite[N];       // x x x ... | y y y ... | ...
};
```

<Callout type="why">
A modern CPU can do billions of additions per second, but only if the numbers arrive in time. The real
bottleneck in game loops, physics, and ML preprocessing is usually **memory bandwidth**, not
arithmetic. Choosing SoA for hot, single-field loops can cut the bytes moved by the field count $F$ —
often a several-times speedup with zero change to the math.
</Callout>

## The same idea, four different worlds

Storing things "by attribute" instead of "by object" pays off any time you read one attribute across
many items.

<ExampleCard domain="Sports analytics" icon="🏟️">
Full player cards (one card per athlete, all stats together) are AoS. But to compute the league's
average points, you only want the points column. A spreadsheet that keeps each stat in its own column
(SoA) lets you sum points without flipping through every card's height, weight, and team.
</ExampleCard>

<ExampleCard domain="Spreadsheets" icon="📊">
Exporting one *column* from a million-row sheet is instant when the file stores data column-by-column,
but slow when it stores complete rows — you must skip past every other field of every row. This is
exactly why analytics databases store data in columns rather than rows.
</ExampleCard>

<ExampleCard domain="Library archives" icon="🗂️">
An index drawer filed by a single attribute — say, publication year — lets a librarian sweep one
contiguous run of cards to answer "what came out in 1998?" The full catalog cards (title, author,
shelf, year together) would force reading everything to extract just the years.
</ExampleCard>

<ExampleCard domain="Music library" icon="🎵">
To total the listening time of 50,000 tracks, you want only each track's *duration*. If durations live
in one packed array, the sum flies; if every track is a fat record bundling title, artist, artwork, and
lyrics, you drag megabytes of cover art through the cache to add up a few numbers.
</ExampleCard>
