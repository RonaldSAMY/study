## The hook: why does a screenshot of a blank wall weigh almost nothing?

Save a photo of a plain white wall and it is a few kilobytes; save a photo of a bustling market and it is megabytes — same pixel count. A text file of one letter repeated a million times zips down to a whisper, while a million random bytes barely shrink at all. Compression is not magic ink that makes bytes smaller; it is a hunt for the two things real data is full of and random data is not: **repetition** and **imbalance**.

<Callout type="intuition" title="The one-sentence idea">
Compression removes two kinds of waste — **redundancy** (patterns that repeat, which you can replace with a pointer) and **skew** (some symbols are far more common than others, so they deserve shorter codes) — down to a hard limit called entropy.
</Callout>

## Intuition: two levers, one floor

Every lossless compressor pulls on the same two levers:

1. **Redundancy.** If the string `abcabcabc` appears, you do not need to store `abc` three times — store it once and say "repeat that." This is what run-length and LZ methods exploit, and it is the subject of the next three lessons.
2. **Skew.** In English text, `e` and space are everywhere while `q` and `z` are rare. Giving `e` a 3-bit code and `z` a 10-bit code beats giving everything a flat 8 bits. This is **entropy coding** (Huffman, arithmetic, FSE), which you met in [Huffman coding](/lessons/dsa-greedy/huffman-coding).

Both levers have a shared limit. The **entropy** of a source — the same quantity from [Information and Entropy](/lessons/information-theory/entropy) — is the average number of bits per symbol that no lossless coder can beat. Redundancy and skew are just the gaps between your data's size and that floor.

And there is a line you must not cross carelessly: **lossless** compression reconstructs the original bit-for-bit (text, code, executables), while **lossy** compression throws away detail humans barely notice (JPEG, MP3) to shrink far more. This track is about lossless.

## How it works

Run-length encoding is the simplest redundancy remover. Fold `WWWWWWBWWWWWW`:

```
scan:  W W W W W W  B  W W W W W W
runs:  (W,6)        (B,1) (W,6)
out:   W6 B1 W6      →  6 bytes for 13 characters
```

Entropy is the other half. For a source where symbol i has probability p_i, the average bits per symbol is

$$
H = -\sum_i p_i \log_2 p_i .
$$

A file of `n` symbols cannot be squeezed below about `n · H / 8` bytes without losing information — that is Shannon's limit, and it is why already-random or already-compressed data will not shrink.

<CodeRunner

  code={`// Run-length encoding: fold repeats into (char, count) tokens
function rle(s) {
  let out = '';
  for (let i = 0; i < s.length; ) {
    let j = i;
    while (j < s.length && s[j] === s[i]) j++;
    out += s[i] + (j - i);
    i = j;
  }
  return out;
}

// Shannon entropy in bits per symbol
function entropy(s) {
  const freq = {};
  for (const c of s) freq[c] = (freq[c] || 0) + 1;
  let h = 0;
  for (const k in freq) {
    const p = freq[k] / s.length;
    h -= p * Math.log2(p);
  }
  return h;
}`}

## The real implementation

Before compressing anything, real systems decide *whether it is worth it* — small files, already-compressed formats, and encrypted blobs have no exploitable waste. And they guard against **compression bombs**, tiny files that explode on decompression. Straight from the course file:

```ts
// /dsa/30-compression/compression-practice.ts
export function shouldCompress(file: { name: string; size: number; type: string }): boolean {
  // Already-compressed formats have no redundancy left to remove
  const compressed = ['.jpg', '.png', '.webp', '.mp4', '.mp3', '.zip', '.gz'];
  if (compressed.some((ext) => file.name.endsWith(ext))) return false;
  // Too small: header/overhead outweighs any savings
  if (file.size < 1024) return false;
  return true;
}

// Defend against zip bombs: a 42 KB file that expands to 4.5 PB
export function compressionBombDetection(
  compressedSize: number,
  decompressedSize: number,
  maxRatio = 1000
): boolean {
  const ratio = decompressedSize / compressedSize;
  return ratio > maxRatio; // true = suspiciously explosive
}
```

## Complexity

RLE and a single entropy pass both scan the data once:

$$
T(n) = O(n), \qquad S(n) = O(1)\text{–}O(k),
$$

where `k` is the alphabet size. The deep result is the *lower bound*: no lossless coder averages fewer than `H` bits per symbol.

| Method | Exploits | Time | Best on |
|--------|----------|------|---------|
| Run-length (RLE) | runs of one symbol | $O(n)$ | scanned images, silence |
| Entropy coding | symbol skew | $O(n \log k)$ | text, any skewed source |
| LZ (next lesson) | repeated substrings | $O(n)$–$O(nw)$ | logs, code, natural text |

The trade-off that runs through this whole track: **ratio versus speed**. Cheap methods (RLE, LZ4) are blazing but leave bits on the table; heavy methods (Brotli, LZMA) approach the entropy floor but cost far more CPU.

## The same idea, different worlds

<ExampleCard domain="Fax & scanned documents" icon="📠">
A fax page is mostly white with thin black text. Group 3 fax encoding run-length codes each scan line — long white runs become a couple of bytes — which is why a mostly-blank page transmits in seconds while a dense one takes far longer.
</ExampleCard>

<ExampleCard domain="Genomics" icon="🧬">
DNA uses a 4-letter alphabet (A, C, G, T). Its entropy is at most 2 bits per base, so genome tools pack each base into 2 bits instead of a full 8-bit character — a 4x win before any fancier modeling of repeats even begins.
</ExampleCard>

<ExampleCard domain="Analytics databases" icon="🗄️">
Column stores (Parquet, ClickHouse) keep one column together, so a "country" column reads `US US US US FR FR ...`. Run-length and dictionary encoding on that sorted column turns millions of rows into a handful of run tokens — the reason columnar scans are so fast.
</ExampleCard>

<ExampleCard domain="Music streaming" icon="🎵">
Silence and sustained notes are low-entropy stretches. Lossless audio codecs (FLAC) predict each sample from the last few and store only the tiny surprise, spending almost no bits on quiet passages and more on busy ones — entropy in action.
</ExampleCard>

## Common pitfalls & interview tips

- **Compressing twice does not help.** Once data is near its entropy floor it looks random; a second pass usually *grows* it by a few header bytes. Never gzip a JPEG.
- **RLE can expand data.** With no runs, `(char, count)` doubles the size. Real formats add an escape/flag so worst case is bounded — always cap your output at "store raw."
- **Entropy is a floor, not a promise.** `H` tells you the best *possible* size; a specific coder may not reach it, and reaching it needs a good model of the data.
- **Lossless vs lossy is a correctness decision, not a tuning knob.** You can lossy-compress a photo; you cannot lossy-compress a bank ledger or a binary.
- **Interview cue:** if asked "how small can this get?", compute the entropy first. If asked "why won't this file shrink?", the answer is almost always "it is already near its entropy — no redundancy or skew left."
