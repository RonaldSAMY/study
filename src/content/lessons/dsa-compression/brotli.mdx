## The hook: the compressor that already knows what the web looks like

Every website starts the same way — `<!DOCTYPE html>`, `<meta charset="utf-8">`, `Content-Type: text/html`, `function`, `return`, `https://`. An LZ compressor has to *see* a string before it can point back at it, so the first copy of each of these is stored the expensive way, as raw literals. Google's engineers asked: what if the compressor shipped with a phrasebook of the web already memorized? That is **Brotli** — and it is why the fonts and scripts loading this very page arrived 15-25% smaller than gzip could manage.

<Callout type="intuition" title="The one-sentence idea">
Brotli is LZ + Huffman like the others, but with a **120 KB built-in dictionary** of common web strings baked in — so patterns like `Content-Type` or `<div class="` compress on their *first* appearance, with no prior copy required.
</Callout>

## Intuition: solving the cold start once and for all

In [Zstandard](/lessons/dsa-compression/zstandard) you saw a *trained* dictionary fix LZ's cold start for small, similar payloads — but you had to build and ship that dictionary yourself. Brotli takes the idea further: it standardizes a single dictionary of ~13,000 common strings and word-transforms (RFC 7932) that every Brotli encoder and every browser already has. Nothing to train, nothing to send.

Brotli stacks three techniques:

1. **Built-in dictionary** — check the 120 KB phrasebook first; a hit is a 2-byte reference even on first use.
2. **LZ back-references** — the familiar sliding window for repeats that *aren't* in the dictionary (your own content).
3. **Huffman + context modeling** — entropy-code the result, and switch Huffman tables based on *context* (HTML tags vs. text vs. numbers each get their own), squeezing skew harder than a single table can.

The result is the best ratio of the three codecs on typical web assets — at the cost of much slower compression, which is why Brotli is used at **build time**, not on every dynamic request.

## How it works

At each position, Brotli asks three questions in order: is this in my dictionary? If not, did I see it earlier in this file? If not, emit a literal.

```
input:  <div class="a"><div class="b">
step 1: "<div class=\"" → dictionary entry #6   (2 bytes, FIRST time!)
step 2: 'a'             → literal
step 3: '">'            → literals
step 4: "<div class=\"" → LZ back-reference OR dictionary again
step 5: 'b'             → literal ...
```

The first `<div class="` cost 2 bytes via the dictionary where LZ would have spent 12 literal bytes. Brotli's **quality levels** run 0-11: low levels (0-4) are fast enough for dynamic content, while 10-11 search exhaustively and are reserved for compressing static assets once at build time. WOFF2, the web font format, is literally Brotli applied to font tables.

const DICT = ['<div class="', '</div>', 'Content-Type', 'text/html', 'https://', 'the '];

function brotli(s) {
  const out = []; let pos = 0;
  while (pos < s.length) {
    // 1. built-in dictionary (works on first use)
    let best = '';
    for (const d of DICT) if (s.startsWith(d, pos) && d.length > best.length) best = d;
    if (best) { out.push('DICT:' + best); pos += best.length; continue; }
    // 2. LZ back-reference into earlier text
    let bLen = 0, bStart = -1;
    for (let k = 0; k < pos; k++) {
      let l = 0;
      while (pos + l < s.length && s[k + l] === s[pos + l]) l++;
      if (l > bLen) { bLen = l; bStart = k; }
    }
    if (bLen >= 4) { out.push('MATCH:' + (pos - bStart) + ',' + bLen); pos += bLen; continue; }
    // 3. literal
    out.push(s[pos]); pos++;
  }
  return out;
}

// bytes: dict ref = 2, match = 3, literal = 1
function size(tokens) {
  return tokens.reduce((n, t) =>
    n + (t.startsWith && t.startsWith('DICT:') ? 2 : t.startsWith && t.startsWith('MATCH:') ? 3 : 1), 0);
}`}

## The real implementation

The course file builds a simplified web dictionary and tries it before falling back to an LZ back-reference — the exact priority order that gives Brotli its edge on small, web-shaped data. From the file:

```ts
// /dsa/30-compression/brotli.ts
private buildWebDictionary(): Map<string, number> {
  const dict = new Map<string, number>();
  const commonStrings = [
    '<html>', '</html>', '<body>', '</body>', '<div class="', '<script>',
    'Content-Type', 'application/json', 'text/html', 'charset=utf-8',
    'the ', 'and ', 'for ', 'https://', 'www.', '.com', '.js', '.css',
  ];
  commonStrings.forEach((str, idx) => dict.set(str, idx));
  return dict;
}

compress(input: string): Uint8Array {
  const encoded = [];
  let pos = 0;
  while (pos < input.length) {
    const dictMatch = this.findDictionaryMatches(input, pos);   // try phrasebook FIRST
    if (dictMatch) {
      encoded.push({ type: 'dict', data: dictMatch.id, length: dictMatch.pattern.length });
      pos += dictMatch.pattern.length;
      continue;
    }
    const backRef = this.findBackReference(input, pos);         // then LZ window
    if (backRef && backRef.length >= 4) {
      encoded.push({ type: 'match', data: backRef.offset, length: backRef.length });
      pos += backRef.length;
    } else {
      encoded.push({ type: 'literal', data: input[pos] });      // else raw byte
      pos++;
    }
  }
  return this.encodeSequences(encoded);   // then Huffman with per-context tables
}
```

Run the full file: `npx tsx 30-compression/brotli.ts`.

## Complexity

High quality levels search the window hard, so compression can approach quadratic in the window size; decompression stays linear and fast:

$$
T_{\text{comp}}(n) = O(n \cdot w)\ \text{to}\ O(n \cdot w^2), \qquad T_{\text{decomp}}(n) = O(n).
$$

The dictionary is a fixed 120 KB regardless of input.

| Algorithm | Ratio (vs gzip) | Comp speed | Decomp speed | Where it runs |
|-----------|-----------------|------------|--------------|---------------|
| LZ4 | ~0.9x | ~10x | ~15x | hot path, real time |
| Zstd | ~1.1x | ~3.5x | ~4x | dynamic + archival |
| **Brotli (lvl 11)** | ~1.2x | ~0.4x | ~1.2x | static assets, build time |

The ratio-vs-speed trade reaches its logical end here: Brotli spends the *most* CPU of the three to win the *best* ratio, which only pays off when you compress **once** and serve **many** times — the definition of a static web asset.

## Common pitfalls & interview tips

- **Never Brotli-compress dynamic responses at level 11.** The encoder is far too slow for per-request work; use level 4-5 for dynamic content, or reach for zstd. Level 10-11 is strictly for build-time static assets.
- **The dictionary only helps web-shaped data.** On genomic reads or binary blobs the built-in phrasebook rarely fires, and Brotli loses its edge over zstd. Match the codec to the content.
- **Serve the right variant.** Precompressing to `.br` is useless if the server ignores `Accept-Encoding` and ships the raw file anyway. Wire up content negotiation and set `Vary: Accept-Encoding`.
- **Do not double-compress.** Brotli over an already-gzipped or already-minified-and-compressed asset wastes CPU and can grow the file — compress the original once.
- **Interview cue:** "best ratio for static web assets" is Brotli. Say "built-in 120 KB dictionary + LZ + context-modeled Huffman, compressed once at build time."
