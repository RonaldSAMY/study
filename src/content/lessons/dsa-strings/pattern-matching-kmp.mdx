## The hook: stop re-reading what you already read

You're searching a long text for the pattern `abcaby`. The naive way lines the pattern up at position 0,
compares until it fails, then shrugs and slides the pattern one step right — **throwing away everything it
just learned** and re-reading the same text characters again and again. That redundant backtracking is what
makes brute force $O(n \cdot m)$. Knuth-Morris-Pratt's idea: when a mismatch happens after matching a few
characters, those matched characters already tell you exactly how far you can safely jump — so the text
pointer **never moves backward**.

<Callout type="intuition" title="The one-sentence idea">
Precompute, for each prefix of the pattern, the longest proper prefix that is also a suffix; on a mismatch
that number tells you how far to slide the pattern without rescanning any text.

</Callout>

## Intuition: the failure function (LPS)

Brute force wastes work because it forgets matched characters. KMP remembers them with a tiny table called
the **LPS array** (Longest proper Prefix that is also a Suffix), sometimes called the *failure function*.

For each position `j` in the pattern, `lps[j]` is the length of the longest string that is **both** a proper
prefix and a suffix of `pattern[0..j]`. Why does that help? Suppose we've matched `j` characters of the
pattern against the text and then character `j` mismatches. The last `lps[j-1]` matched characters are, by
definition, also a prefix of the pattern — so we can slide the pattern forward to reuse them and resume
comparing at `j = lps[j-1]`, **keeping the text pointer where it is**. No character of the text is ever read
twice.

Two pointers drive the search: `i` over the text (only ever moves forward) and `j` over the pattern (jumps
back via the LPS table on a mismatch). When `j` reaches `m`, the whole pattern matched and we record an
occurrence — then jump `j` to `lps[m-1]` to keep finding overlapping matches.

## How it works

First the LPS table for `abcaby`. No prefix repeats until the second `a`/`b`, which echo the start:

```
pattern:  a  b  c  a  b  y
index:    0  1  2  3  4  5
lps:      0  0  0  1  2  0
                   ^  ^
              "ab" is both a prefix and the suffix ending at index 4 → lps=2
```

Now search `abxabcabcaby`. At the mismatch on `x`, instead of restarting we consult the table and slide:

```
text:    a b x a b c a b c a b y
pat:     a b ↑                      mismatch at text[2]='x', pat[2]='c'
                                    j was 2 → j = lps[1] = 0, i stays at 2
text:    a b x a b c a b c a b y
pat:         a b c a b y            re-anchor, keep matching forward → match at index 6
```

  const m = pattern.length;
  const lps = new Array(m).fill(0);
  let len = 0, i = 1;
  while (i < m) {
    if (pattern[i] === pattern[len]) { len++; lps[i] = len; i++; }
    else if (len !== 0) len = lps[len - 1];     // fall back, don't reset to 0
    else { lps[i] = 0; i++; }
  }
  return lps;
}

function kmpSearch(text, pattern) {
  const matches = [];
  const n = text.length, m = pattern.length;
  if (m === 0 || m > n) return matches;
  const lps = buildLPS(pattern);
  let i = 0, j = 0;                              // i over text, j over pattern
  while (i < n) {
    if (text[i] === pattern[j]) { i++; j++; }

    else if (i < n && text[i] !== pattern[j]) {
      if (j !== 0) j = lps[j - 1];               // jump, i stays put
      else i++;
    }
  }
  return matches;
}`}

## The real implementation

From the course file — the failure function and the linear-time search, exactly as the demo runs them:

```ts
// /dsa/02-strings/pattern-matching.ts
function buildLPS(pattern: string): number[] {
    const m = pattern.length;
    const lps = new Array(m).fill(0);
    let length = 0;
    let i = 1;
    while (i < m) {
        if (pattern[i] === pattern[length]) {
            length++;
            lps[i] = length;
            i++;
        } else {
            if (length !== 0) {
                length = lps[length - 1];   // reuse a shorter prefix instead of resetting
            } else {
                lps[i] = 0;
                i++;
            }
        }
    }
    return lps;
}

function kmpSearch(text: string, pattern: string): number[] {
    const matches: number[] = [];
    const n = text.length, m = pattern.length;
    if (m === 0 || m > n) return matches;
    const lps = buildLPS(pattern);
    let i = 0, j = 0;
    while (i < n) {
        if (text[i] === pattern[j]) { i++; j++; }
        if (j === m) {
            matches.push(i - j);            // occurrence found
            j = lps[j - 1];                 // continue (allows overlaps)
        } else if (i < n && text[i] !== pattern[j]) {
            if (j !== 0) j = lps[j - 1];    // slide pattern; i never decreases
            else i++;
        }
    }
    return matches;
}
```

## Complexity

Building the table is linear in the pattern; the search is linear in the text because `i` only moves forward
and `j` decreases at most as often as it increases:

$$
T(n, m) = O(n + m), \qquad S(m) = O(m).
$$

| Algorithm | Time | Space | Note |
|-----------|------|-------|------|
| Naive search | $O(n \cdot m)$ | $O(1)$ | re-reads text after each mismatch |
| **KMP** | $O(n + m)$ | $O(m)$ | failure function, no text backtracking |
| Rabin–Karp | $O(n + m)$ avg, $O(n m)$ worst | $O(1)$ | rolling hash, great for multi-pattern |
| Boyer–Moore | $O(n/m)$ best, $O(n m)$ worst | $O(\sigma)$ | skips ahead, fast in practice |

## The same idea, different worlds

<ExampleCard domain="Bioinformatics" icon="🧬">
Aligning a short gene probe against a chromosome of billions of bases needs linear-time search — quadratic
rescanning is hopeless at that scale. KMP-style exact matching (and its hashed cousins) underpins the
seed-finding stage of read aligners.
</ExampleCard>

<ExampleCard domain="Network security" icon="🛰️">
An intrusion-detection system scans every passing packet for known attack signatures at line rate. It can't
afford to back up over bytes it already saw, so it uses failure-function automata (Aho–Corasick generalizes
KMP to many patterns at once).
</ExampleCard>

<ExampleCard domain="Version control" icon="🔀">
Diff and `grep` engines locate a query inside huge source files repeatedly. Linear, no-backtrack matching
keeps an editor's "find in files" responsive even across a multi-gigabyte repository.
</ExampleCard>

## Common pitfalls & interview tips

- **"Proper" prefix matters.** `lps[j]` excludes the whole substring itself — the prefix must be strictly
  shorter — otherwise every `lps` value would trivially equal its index.
- **On a mismatch, fall back with `lps[length-1]`; don't reset to 0.** Resetting is the bug that quietly
  turns KMP back into something slower and sometimes wrong. The fall-back chain is what reuses partial matches.
- **The text pointer `i` must never decrease.** If you find yourself moving `i` backward, you've reverted to
  brute force; only `j` jumps.
- **Record the match at `i - j`, then set `j = lps[m-1]`** to catch overlapping occurrences (e.g. `"aa"` in
  `"aaaa"` matches at 0, 1, 2).
- **Interview cue:** "find all occurrences in linear time" or "no extra rescanning" points straight at KMP;
  if they say "multiple patterns at once", reach for Aho–Corasick.

2. **Easy:** Why is the naive matcher $O(n \cdot m)$? Give a text/pattern pair that hits the worst case.
3. **Medium:** Use KMP to detect whether string `a` is a **rotation** of string `b` (hint: search `b` inside

4. **Medium:** Solve *Implement strStr()* — return the index of the first occurrence — using `kmpSearch` and

5. **Hard:** Find the **shortest** string that, when repeated, yields the pattern (LeetCode 459, "Repeated
   Substring Pattern"), using only the LPS array — no extra search.
