## The hook: a one-way street that becomes two-way

A singly linked list is a one-way street: you can drive forward from the head, but to back up by a
single node you must return to the start and drive the whole way again. That's painful when you hold a
node and just want to delete it — you have no way to reach the node *before* it to fix its `next`
pointer. The fix is almost embarrassingly simple: give every node a second pointer, `prev`, aimed at the
node behind it. Now the street is two-way. You can walk backward, you can delete the tail in one move,
and — crucially — given any node, you can splice it out in O(1) because both of its neighbours are one
hop away.

<Callout type="intuition" title="The one-sentence idea">
A doubly linked list adds a prev pointer to every node, so each node knows both its neighbours — enabling
backward traversal and O(1) removal of any node you already have a reference to.
</Callout>

## Intuition: two pointers per node, four to re-link

Every node now carries `value`, `next`, **and** `prev`. The list tracks a `head` (whose `prev` is
`null`) and a `tail` (whose `next` is `null`). The cost is one extra pointer of memory per node and a
little more bookkeeping: an insertion or deletion has to fix up to **four** pointers instead of two,
because both neighbours must be told about the change.

What you buy is significant. `removeLast` is now O(1) — `tail` knows its `prev`, so there's no walk to
find the second-to-last node. Better still, `removeNode(node)` is O(1) for *any* node you hold: set
`node.prev.next = node.next` and `node.next.prev = node.prev`, and the node is spliced out without a
search. That single property is why a doubly linked list, paired with a hash map, is the backbone of an
**LRU cache**: the map finds a node in O(1), and the list moves or evicts it in O(1).

## How it works

To remove an interior node `X` with neighbours `A` (before) and `B` (after):

```
before:  A  <->  X  <->  B
step 1:  A.next = X.next      // A.next now points to B
step 2:  B.prev = X.prev      // B.prev now points to A
after:   A  <->  B            // X is spliced out, two writes
```

Inserting works the same way in reverse — a new node sets its own `prev` and `next`, then its two
neighbours point back at it. The bookkeeping is symmetric, which is exactly why drawing both lanes makes
it click.

## The real implementation

From the course's doubly linked list — the O(1) node splice and the symmetric `removeLast` that a
singly linked list cannot match:

```ts
// /dsa/05-linked-list/doubly-linked-list.ts
interface DoublyListNode<T> {
    value: T;
    prev: DoublyListNode<T> | null;
    next: DoublyListNode<T> | null;
}

class DoublyLinkedList<T> {
    head: DoublyListNode<T> | null = null;
    tail: DoublyListNode<T> | null = null;
    length: number = 0;

    // O(1) — the tail knows its prev, so no walk
    removeLast(): T | null {
        if (this.isEmpty()) return null;
        const value = this.tail!.value;
        if (this.length === 1) { this.head = null; this.tail = null; }
        else { this.tail = this.tail!.prev; this.tail!.next = null; }
        this.length--;
        return value;
    }

    // O(1) — splice out a node you already hold
    removeNode(node: DoublyListNode<T>): T | null {
        if (node === this.head) return this.removeFirst();
        if (node === this.tail) return this.removeLast();
        node.prev!.next = node.next;
        node.next!.prev = node.prev;
        this.length--;
        return node.value;
    }

    isEmpty(): boolean { return this.length === 0; }
}
```

Run the full file: `npx tsx 05-linked-list/doubly-linked-list.ts`.

<Callout type="why">
The same file builds an <code>LRUCache</code> from this list plus a <code>Map</code>. The map turns a
key into a node reference in O(1); <code>removeNode</code> then unlinks it and <code>prepend</code> moves
it to the front — so every <code>get</code> and <code>put</code> is O(1). Without the backward pointer,
that move-to-front would cost an O(n) search for the predecessor each time.
</Callout>

## Complexity

You trade one pointer of memory per node for cheaper end-and-node operations:

$$
T_{\text{removeLast}} = O(1), \qquad T_{\text{removeNode}} = O(1), \qquad S = O(n) \ \text{with a 2}\times \ \text{pointer overhead.}
$$

| Operation | Singly | Doubly |
|-----------|--------|--------|
| Remove first | $O(1)$ | $O(1)$ |
| Remove last | $O(n)$ | $O(1)$ |
| Remove a held node | $O(n)$ | $O(1)$ |
| Traverse backward | impossible | $O(n)$ |
| Pointers per node | 1 | 2 |

If you never walk backward and never delete from the tail, the singly list is leaner. The moment either
appears — think LRU caches or browser history — the doubly linked list earns its extra pointer.

## Common pitfalls & interview tips

- **Updating only one direction.** Every splice touches *both* lanes. Fix `next` but forget `prev` (or
  vice versa) and the backward traversal silently breaks while the forward one looks fine.
- **Forgetting the head/tail edge cases.** Removing the head must null the new head's `prev`; removing
  the tail must null the new tail's `next`. Skipping these leaves dangling pointers.
- **Order of assignment on insert.** Wire the new node's `prev` and `next` first, then redirect the
  neighbours — otherwise you can overwrite a link you still need.
- **Interview cue:** "move-to-front in O(1)" or "evict least-recently-used" is a flashing sign for a
  doubly linked list plus a hash map. Name the LRU cache and explain why `prev` makes the splice O(1).

## Practice

1. **Easy:** Draw `A <-> B <-> C` and write the four pointer assignments to insert `X` between `A` and `B`.
2. **Easy:** Implement `toArrayReverse` by walking from `tail` via `prev`.
3. **Medium:** Implement `removeNode(node)` handling the head, tail, and interior cases.
4. **Medium:** Reverse a doubly linked list in place by swapping each node's `prev` and `next`, then
   swapping `head` and `tail`.
5. **Hard:** Build an LRU cache: a `Map` from key to node plus a doubly linked list, with O(1) `get` and
   `put` that evict the tail when full.
