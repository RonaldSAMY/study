## The hook: a treasure hunt, not a bookshelf

An array is a bookshelf — every book sits in a numbered slot, side by side, and you can grab slot 47
instantly. But shoving a new book into the middle means sliding every later book over by one. A
**linked list** is a treasure hunt instead: each clue holds a prize *and* tells you where the next clue
is hidden. The clues can be scattered anywhere in memory. To insert a new clue you just rewrite one
"go here next" note — no sliding, no shelf-shuffling. The price you pay: there are no slot numbers, so
to reach the 47th clue you must follow the trail from the very first one.

<Callout type="intuition" title="The one-sentence idea">
A singly linked list stores each value in a node that also holds a pointer to the next node, so the
list is reached entirely through a single head reference and walked one hop at a time.
</Callout>

## Intuition: nodes, the head, and the null at the end

A **node** is a tiny package with two fields: a `value` and a `next` pointer. The list itself is just a
**head** reference to the first node. Follow `head.next`, then that node's `next`, and so on, until you
hit `null` — the signal that the trail has ended.

Because nodes are linked by pointers rather than packed into one block, two things flip compared to an
array. Inserting at the front is now trivially cheap: make your new node point at the old head, then
move the head — nothing else moves. But indexing is now expensive: there is no arithmetic that jumps to
"element 47", so reaching it costs a walk of 47 hops. A linked list optimizes for *restructuring* at the
cost of *random access*. Many implementations also cache a **tail** pointer so that appending to the end
is O(1) instead of an O(n) walk.

## How it works

Suppose `head -> 7 -> 3 -> 9 -> 5 -> null` and we prepend `4`:

```
new = Node(4)
new.next = head        // 4 -> 7 -> 3 -> 9 -> 5 -> null
head = new             // head now points at 4
result: 4 -> 7 -> 3 -> 9 -> 5 -> null
```

Two pointer writes, done — the rest of the list never moved. Traversal to collect every value is the
mirror image: start at the head and keep following `next` until you fall off the end into `null`.

function node(value) { return { value, next: null }; }

function fromArray(arr) {
  let head = null, tail = null;
  for (const v of arr) {
    const n = node(v);
    if (!head) { head = n; tail = n; }
    else { tail.next = n; tail = n; }
  }
  return head;
}

function prepend(head, value) {
  const n = node(value);
  n.next = head;          // point at the old front
  return n;               // the new node is the new head
}

function toArray(head) {
  const out = [];
  for (let cur = head; cur; cur = cur.next) out.push(cur.value);
  return out;
}

function get(head, index) {
  let cur = head, i = 0;
  while (cur && i < index) { cur = cur.next; i++; }
  return cur ? cur.value : undefined;   // O(n): no jump to an index
}`}

## The real implementation

The course's `SinglyLinkedList` keeps both a `head` and a `tail` so `append` is O(1), and bumps a
`length` counter on every change. Here is the faithful core:

```ts
// /dsa/05-linked-list/singly-linked-list.ts
interface SinglyListNode<T> {
    value: T;
    next: SinglyListNode<T> | null;
}

class SinglyLinkedList<T> {
    head: SinglyListNode<T> | null = null;
    tail: SinglyListNode<T> | null = null;
    length: number = 0;

    // O(1) thanks to the cached tail
    append(value: T): this {
        const newNode = { value, next: null };
        if (!this.head) { this.head = newNode; this.tail = newNode; }
        else { this.tail!.next = newNode; this.tail = newNode; }
        this.length++;
        return this;
    }

    // O(1): rewire the front, no shifting
    prepend(value: T): this {
        const newNode = { value, next: this.head };
        if (!this.head) this.tail = newNode;
        this.head = newNode;
        this.length++;
        return this;
    }

    // O(n): there is no random access — we must walk
    get(index: number): T | undefined {
        if (index < 0 || index >= this.length) return undefined;
        let current = this.head!;
        for (let i = 0; i < index; i++) current = current.next!;
        return current.value;
    }
}
```

## Complexity

The defining trade-off, in one table. Inserts and deletes at a known end are constant; anything
requiring a position is linear because you must walk there:

$$
T_{\text{prepend}} = O(1), \qquad T_{\text{append}} = O(1)\ \text{with a tail}, \qquad T_{\text{get}} = O(n).
$$

| Operation | Linked list | Dynamic array |
|-----------|-------------|---------------|
| Insert at front | $O(1)$ | $O(n)$ |
| Insert at back | $O(1)$ (with tail) | $O(1)$ amortized |
| Access by index | $O(n)$ | $O(1)$ |
| Search by value | $O(n)$ | $O(n)$ |
| Memory per element | value + pointer | value only |

Arrays win on indexing and cache locality; linked lists win when you restructure the front a lot and
never need slot numbers.

## The same idea, different worlds

</ExampleCard>

<ExampleCard domain="Rail freight" icon="🚂">
A train is a chain of cars, each coupled to the one behind it; the locomotive is the head. To add a
wagon you re-couple two links — but to reach the caboose you walk the whole train, because there is no
"car number 12" shortcut.
</ExampleCard>

<ExampleCard domain="Operating systems" icon="🧠">
A memory allocator threads its free blocks into a "free list": each free chunk stores a pointer to the
next free chunk. Freeing memory prepends a block in O(1), with no compaction of the surrounding heap.
</ExampleCard>

## Common pitfalls & interview tips

- **Losing the rest of the list.** When inserting, set `new.next = head` *before* you move `head`. Flip
  the order and the tail of the list is orphaned and garbage-collected.
- **Forgetting the tail on append.** Without a cached `tail`, every append silently becomes an O(n)
  walk — fine once, disastrous in a loop that builds a big list (that is an accidental O(n^2)).
- **Null checks while walking.** The traversal guard is `while (cur)`, not `while (cur.next)`. The
  latter skips the last node or throws on an empty list.
- **Interview cue:** if the prompt stresses "frequent insertions/deletions at the front" or "no random
  access needed," a linked list is the intended structure. Mention the head/tail/length bookkeeping out
  loud.

## Practice

1. **Easy:** Draw `head -> 1 -> 2 -> 3 -> null`. Prepend `0` and write the two pointer assignments in
   order.
2. **Easy:** Write `toArray` from scratch using a `while (cur)` loop.
3. **Medium:** Implement `get(index)` and explain why it is O(n) while an array's is O(1).
4. **Medium:** Implement `removeFirst` and return the removed value, fixing `tail` when the list becomes
   empty.
5. **Hard:** Implement `insertAt(index, value)` for any valid index, reusing `prepend` and `append` for
   the two ends and walking to `index - 1` otherwise.
