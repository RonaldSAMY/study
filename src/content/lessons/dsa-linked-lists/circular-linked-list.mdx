## The hook: a playlist that never ends

You hit "repeat all" on a playlist, the last song finishes, and instead of stopping it rolls straight
back to the first. There is no end of the list — the last node simply points back to the first. That is
a **circular linked list**: a normal chain whose `tail.next` is the `head` rather than `null`. From any
node you can keep walking and you will visit every node forever, looping around again and again. It is
the natural shape for anything that takes **turns in a cycle** — players in a board game, processes
sharing a CPU, or the seats in the gruesome counting game we will solve below.

<Callout type="intuition" title="The one-sentence idea">
A circular linked list is a singly (or doubly) linked list whose last node points back to the first, so
the chain has no <code>null</code> end — you can traverse it endlessly in a loop.
</Callout>

## Intuition: close the loop

Take an ordinary singly linked list `A → B → C` and make one change: instead of `C.next = null`, set
`C.next = A`. Now the structure is a ring. The list still keeps a `head` (a starting point) and usually a
`tail`, but the defining feature is that **walking `next` never hits `null`** — after the tail you are
back at the head.

Two consequences fall out immediately. First, **insertion at both ends is O(1)** when you keep a `tail`
pointer: to append, you splice a node between `tail` and `head` and move `tail` forward — no walk needed.
Second, **traversal needs a stop condition you choose yourself**. A plain `while (current !== null)` loop
would spin forever, so instead you loop a fixed number of times (`length`) or until you return to the
node you started from.

The price is care. Every algorithm must guard against the infinite loop, and an empty list is a special
case (there is no node to point back to). Get those right and the ring is a beautifully simple structure.

## How it works

To **append** to a circular list that keeps a `tail`, you never traverse — you splice between tail and
head in three writes:

```
before:  head → A → B → (tail)C → [back to head]
step 1:  newNode.next = head      // new node closes the ring to the front
step 2:  tail.next = newNode      // old tail points to the new node
step 3:  tail = newNode           // the new node is the tail
after:   head → A → B → C → (tail)D → [back to head]
```

To **traverse safely**, loop exactly `length` times (or stop when you come back to `head`) so you never
spin forever:

  let head = null, tail = null, length = 0;
  for (const v of arr) {
    const node = { value: v, next: null };
    if (!head) { head = node; tail = node; node.next = node; }   // ring of one
    else { node.next = head; tail.next = node; tail = node; }    // splice at end
    length++;
  }
  return { head, tail, length };
}

// Visit every node exactly once, then stop — no infinite loop.
function toArray(list) {
  const out = [];
  let cur = list.head;
  for (let i = 0; i < list.length; i++) { out.push(cur.value); cur = cur.next; }
  return out;
}

// Proof it is circular: after the tail we are back at the head.
function wrapsAround(list) {
  return list.tail.next === list.head;
}

// Josephus: every k-th person leaves until one remains (1-indexed survivor).
function josephus(n, k) {
  const list = build(Array.from({ length: n }, (_, i) => i + 1));
  let prev = list.tail, cur = list.head, remaining = n;
  while (remaining > 1) {
    for (let i = 1; i < k; i++) { prev = cur; cur = cur.next; }  // count k-1 hops
    prev.next = cur.next;                                         // splice out cur
    cur = cur.next;                                               // resume after it
    remaining--;
  }
  return cur.value;
}`}

## The real implementation

From the course's circular linked list — the O(1) end inserts that splice the ring closed, plus the
`rotate` that advances the start by moving `head` and `tail` forward together:

```ts
// /dsa/05-linked-list/circular-linked-list.ts
interface CircularNode<T> { value: T; next: CircularNode<T>; }

class CircularLinkedList<T> {
    head: CircularNode<T> | null = null;
    tail: CircularNode<T> | null = null;
    length: number = 0;

    // O(1) — splice a node between tail and head, then move tail
    insertLast(value: T): this {
        const newNode = { value } as CircularNode<T>;
        if (!this.head) {
            newNode.next = newNode;          // a ring of one points to itself
            this.head = newNode;
            this.tail = newNode;
        } else {
            newNode.next = this.head;        // close the ring to the front
            this.tail!.next = newNode;       // old tail -> new node
            this.tail = newNode;             // new node is the tail
        }
        this.length++;
        return this;
    }

    // O(k) — rotate the ring by advancing the start k positions
    rotate(k: number): this {
        if (!this.head || this.length <= 1 || k === 0) return this;
        k = ((k % this.length) + this.length) % this.length;
        for (let i = 0; i < k; i++) {
            this.tail = this.head;
            this.head = this.head!.next;
        }
        return this;
    }
}
```

Run the full file: `npx tsx 05-linked-list/circular-linked-list.ts`. The same file also ships a
`josephus(n, k)` solver and a `roundRobinScheduler` built on `rotate`.

<Callout type="why">
Because the ring has no start or end, <code>rotate</code> is just relabelling which node you call the
head — no data moves. That is why a circular list is the perfect queue for a <strong>round-robin
scheduler</strong>: each tick, <code>rotate(1)</code> hands the CPU to the next process in O(1), and
after the last one it naturally returns to the first.
</Callout>

## Complexity

You keep the linked list's cheap ends and pay only in the care your traversals need:

$$
T_{\text{insertFirst}} = T_{\text{insertLast}} = O(1), \qquad T_{\text{search}} = O(n), \qquad S = O(n).
$$

| Operation | Cost | Note |
|-----------|------|------|
| Insert first / last | $O(1)$ | with a `tail` pointer, no walk |
| Delete first | $O(1)$ | re-point `tail.next` to the new head |
| Search / access by index | $O(n)$ | must walk the ring |
| Rotate by $k$ | $O(k)$ | move `head`/`tail` forward $k$ times |
| Josephus ($n$ people, step $k$) | $O(n \cdot k)$ | $n-1$ eliminations, each $k$ hops |

The danger is never the cost — it is the **infinite loop**. A traversal that forgets to stop after
`length` nodes (or to check for `head` again) runs forever.

## Common pitfalls & interview tips

- **The infinite loop.** Never write `while (current !== null)` on a circular list — there is no
  `null`. Loop `length` times, or stop when you return to the node you started from.
- **The empty-list special case.** The first inserted node must point to **itself** (`node.next =
  node`); forgetting this leaves a dangling `next` and breaks the ring.
- **Keep `tail` in sync.** O(1) end-insertion relies on `tail`. If you delete the tail or head, re-point
  `tail.next` to the new head so the ring stays closed.
- **Interview cue:** "round-robin", "take turns in a cycle", or "every k-th element is removed" points
  straight at a circular linked list. For Josephus, mention that a circular list gives the intuitive
  $O(n \cdot k)$ simulation, while a math recurrence can do it in $O(n)$.

## Practice

1. **Easy:** Given `A → B → C` as a ring, write the single pointer change that turns the plain list
   `C.next = null` into a circular one.
2. **Easy:** Write `toArray` for a circular list that stops after exactly `length` nodes.
3. **Medium:** Implement `deleteFirst` that keeps the ring closed by updating `tail.next`.
4. **Medium:** Implement `rotate(k)` by advancing `head` and `tail` forward `k` times.
5. **Hard:** Solve the Josephus problem two ways — simulate it with a circular list in $O(n \cdot k)$,
   then with the recurrence $J(1)=0,\ J(m)=(J(m-1)+k)\bmod m$ in $O(n)$ — and check they agree.
