## The hook: turning the trail around, and catching a runaway loop

You already know a list is a treasure hunt of nodes, each pointing to the next. Two questions dominate
every linked-list interview. First: can you make the trail run *backward* — last clue first — without
copying anything into a new list? Second: what if some prankster made the last clue point back to an
earlier one, so the trail loops forever? You need to detect that loop **without** marking nodes or
keeping a set of everything you've seen. Both answers come from carefully choreographed pointers, and
both run in O(1) extra space.

<Callout type="intuition" title="The one-sentence idea">
Reversal walks the list once, flipping each node's next pointer to the previous node; cycle detection
sends a slow runner and a fast runner through the list, and if they ever meet, there is a loop.
</Callout>

## Intuition: three pointers to reverse, two speeds to detect

**Reversal.** The danger is that the moment you flip a node's `next` to point backward, you've lost the
link to the rest of the list. So you hold three references at once: `prev` (the part already reversed),
`curr` (the node you're flipping now), and a saved `next` (so you don't lose the unprocessed tail).
Each step: stash `next = curr.next`, redirect `curr.next = prev`, then slide both `prev` and `curr`
forward. When `curr` falls off the end, `prev` is the new head.

**Cycle detection (Floyd's "tortoise and hare").** Move `slow` one node per step and `fast` two. If the
list ends, `fast` reaches `null` and there is no cycle. But if there's a loop, `fast` is gaining exactly
one node on `slow` every step, so inside the loop it must eventually lap and *land on the same node* as
`slow`. A meeting is a mathematical certainty, no bookkeeping required — that's why it needs only O(1)
space, beating the obvious hash-set approach.

## How it works

Reversing `1 -> 2 -> 3 -> null`, tracking the three pointers each step:

```
prev=null  curr=1            // start
prev=1     curr=2   (1 -> null)
prev=2     curr=3   (2 -> 1 -> null)
prev=3     curr=null (3 -> 2 -> 1 -> null)
return prev = 3   // new head
```

For cycle detection on `1 -> 2 -> 3 -> 4 -> back to 2`, the runners march `slow=2,fast=3`, then
`slow=3,fast=2`, then `slow=4,fast=4` — collision, so a loop exists.

// Reverse in place: O(n) time, O(1) space
function reverse(head) {
  let prev = null, curr = head;
  while (curr) {
    const next = curr.next;   // save the rest
    curr.next = prev;         // flip the link
    prev = curr;              // advance prev
    curr = next;              // advance curr
  }
  return prev;                // new head
}

// Floyd's cycle detection: O(n) time, O(1) space
function hasCycle(head) {
  let slow = head, fast = head;
  while (fast && fast.next) {
    slow = slow.next;
    fast = fast.next.next;
    if (slow === fast) return true;
  }
  return false;
}

// build a list whose tail links back to index 1 (a cycle)

  let target = h; for (let i = 0; i < backTo; i++) target = target.next;
  tail.next = target;
  return h;
}`}

## The real implementation

Straight from the course's operations file — the iterative reverse, plus the variant that returns the
exact node where a cycle begins:

```js
// /dsa/05-linked-list/linked-list-operations.js
function reverseIterative(head) {
    let prev = null;
    let current = head;
    while (current) {
        const nextTemp = current.next;   // don't lose the rest
        current.next = prev;             // flip the pointer
        prev = current;                  // step prev forward
        current = nextTemp;              // step current forward
    }
    return prev;                         // prev is the new head
}

function detectCycleStart(head) {
    if (!head || !head.next) return null;
    let slow = head, fast = head;
    while (fast && fast.next) {          // phase 1: find a meeting point
        slow = slow.next;
        fast = fast.next.next;
        if (slow === fast) break;
    }
    if (!fast || !fast.next) return null;   // fell off the end: no cycle
    slow = head;                            // phase 2: walk to the entrance
    while (slow !== fast) {
        slow = slow.next;
        fast = fast.next;
    }
    return slow;                            // the node where the loop starts
}
```

Run the full file: `npx tsx 05-linked-list/linked-list-operations.js` (or `node`).

<Callout type="why">
Phase 2 of `detectCycleStart` looks like magic but is pure arithmetic: when the runners first meet, the
distance from the head to the loop entrance equals the distance from the meeting point to that same
entrance. So resetting `slow` to the head and stepping both one at a time lands them together exactly at
the loop's start.
</Callout>

## Complexity

Both algorithms make a constant number of passes with a fixed handful of pointers:

$$
T_{\text{reverse}} = O(n), \quad S_{\text{reverse}} = O(1), \qquad T_{\text{cycle}} = O(n), \quad S_{\text{cycle}} = O(1).
$$

| Task | Approach | Time | Space |
|------|----------|------|-------|
| Reverse | Three-pointer flip | $O(n)$ | $O(1)$ |
| Reverse | Recursion | $O(n)$ | $O(n)$ stack |
| Detect cycle | Hash set of nodes | $O(n)$ | $O(n)$ |
| Detect cycle | Floyd fast/slow | $O(n)$ | $O(1)$ |

The fast/slow trick matches the hash-set's time while crushing its space from O(n) to O(1) — the reason
it is the textbook answer.

## The same idea, different worlds

<ExampleCard domain="Memory management" icon="♻️">
Garbage collectors must spot reference loops — objects that point at each other in a ring — so they can
free them. Floyd-style pointer chasing detects such cycles without allocating extra tracking memory,
which is precious mid-collection.
</ExampleCard>

<ExampleCard domain="Athletics" icon="🏃">
On a running track, a sprinter laps a jogger precisely because they share the same closed loop and move
at different speeds. The faster runner gains a fixed amount each lap and inevitably catches up — exactly
the tortoise-and-hare guarantee.
</ExampleCard>

<ExampleCard domain="Text editing" icon="↩️">
"Reverse the word order" or undoing a sequence of edits is a list reversal: re-point each step at its
predecessor rather than copying the whole history into a fresh buffer.
</ExampleCard>

## Common pitfalls & interview tips

- **Saving `next` too late.** In reversal you must stash `curr.next` *before* overwriting it. Forget,
  and you sever the unprocessed tail and loop forever on a one-node ring.
- **Wrong fast-pointer guard.** It's `while (fast && fast.next)`. Checking only `fast.next` throws a null
  dereference the moment `fast` is already `null`.
- **Assuming a meeting point is the loop entrance.** It usually isn't — you still need phase two (reset
  `slow` to head, step both by one) to find where the cycle actually begins.
- **Interview cue:** "constant extra space" plus "linked list" almost always means a two-pointer
  technique. Say "fast and slow" or "three-pointer reverse" early and narrate the pointer moves.
