## The hook: the trig call that ate your frame

Old consoles could not afford to compute a sine. Yet their games were full of bobbing coins, rippling
water and spinning sprites — all driven by $\sin$ and $\cos$. The trick: nobody computed sine at
runtime. At startup they filled an **array** with a few thousand precomputed sine values, and every
"compute" became a single array read. They spent a sliver of memory **once** to make the math
practically free **forever after**.

That is the deal at the heart of performance engineering. You almost never get faster *and* smaller for
free — but you can very often **trade one for the other**. Knowing which way to trade is the skill.

<Callout type="intuition" title="The one-sentence idea">
A **space–time tradeoff** spends extra **memory** to save **time** (store an answer instead of
recomputing it) — or spends extra time to save memory. You pick which resource you can spare.
</Callout>

## Intuition: stop redoing work you already did

You met **Big-O** — the *shape* of how cost grows — and the **hash map** — instant lookup by key. This
lesson is what happens when they shake hands.

Recall the exploding Fibonacci tree from last lesson: the same subproblems were recomputed thousands of
times. **Memoization** fixes that with one idea — *keep a notebook*. Before computing $F(n)$, check a
cache (a hash map): if the answer is already there, return it; otherwise compute it once and write it
down. The exponential $O(2^n)$ work collapses to linear $O(n)$, at the cost of one cache entry per
distinct subproblem.

There are two flavors of the same move:

- **Memoization / caching** — fill the store **lazily**, on demand, the first time each answer is
  needed. Great when you do not know in advance which answers you will use.
- **Precomputed lookup table** — fill the store **eagerly**, up front, before the work starts. Great
  when the inputs are a known, bounded set (like every angle from $0$ to $2\pi$).

Either way you convert *repeated computation* into a *single read* — usually $O(1)$.

## The formal definition

Caching changes a function's complexity by **amortizing** repeated work. For naive recursive Fibonacci:

$$
T_{\text{naive}}(n) = O(2^n) \ \text{time}, \qquad S_{\text{naive}}(n) = O(n) \ \text{space (the stack)}.
$$

Add a cache of already-computed values and each subproblem is solved exactly once:

$$
T_{\text{memo}}(n) = O(n) \ \text{time}, \qquad S_{\text{memo}}(n) = O(n) \ \text{space (the cache)}.
$$

The pattern in code is tiny — guard the work with a lookup, and store the result before returning:

```python
cache = {}
def fib(n):
    if n < 2:
        return n
    if n in cache:          # already computed? one O(1) read
        return cache[n]
    cache[n] = fib(n-1) + fib(n-2)   # compute once, write it down
    return cache[n]
```

A precomputed table generalizes this. If a function over a bounded input is called $m$ times, naive
costs $O(m \cdot c)$ where $c$ is the per-call compute cost. A table of $N$ entries costs $O(N)$ to
build **once**, then $O(1)$ per call — total $O(N + m)$. When $m \gg N$ and $c$ is large, the table
wins overwhelmingly.

<Callout type="why">
This trade powers modern ML serving. **KV-caching** in a transformer stores the keys and values of
past tokens so each new token is generated without re-attending over the whole sequence from scratch —
the same "store it once, read it back" bargain, just with tensors instead of integers.
</Callout>

## The same idea, four different worlds

<ExampleCard domain="Retro graphics" icon="🌊">
Beyond sine, classic engines precomputed **rotation tables** and even full screen-blit offset tables.
Every per-pixel transform that would have cost a multiply became an array read. The cartridge held a
few extra kilobytes; the CPU got entire milliseconds back every frame.
</ExampleCard>

<ExampleCard domain="Rendering" icon="💡">
**Baked lighting** is a space–time trade at asset scale. Instead of computing global illumination for a
static scene every frame, the engine computes it once offline and stores the result in **lightmap**
textures. You ship bigger files so the GPU can simply sample a pixel instead of simulating light.
</ExampleCard>

<ExampleCard domain="Databases" icon="🗄️">
A database **index** is pure tradeoff: it stores an extra sorted structure (more disk) so a query
becomes an $O(\log n)$ seek instead of an $O(n)$ full-table scan. Add too many indexes and writes slow
down and storage balloons — the trade has a cost you must respect.
</ExampleCard>

<ExampleCard domain="Game AI" icon="🗺️">
When hundreds of units swarm one target, recomputing a path per unit is brutal. Engines precompute a
**flow field**: one pass stamps every tile with the direction toward the goal. Each unit then just
reads its tile — one lookup instead of a full search. Memory for the grid buys real-time movement for a
crowd.
</ExampleCard>

<Callout type="warning">
The trade is not free or always wise. Caches cost memory, can go **stale** when inputs change, and add
complexity. Cache when work is <em>repeated</em> and answers are <em>reusable</em> — not for a value
you compute once.
</Callout>
