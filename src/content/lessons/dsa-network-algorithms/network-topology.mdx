## The hook: same nodes, wildly different networks

Give a network architect eight servers and they can wire them a dozen ways. A **star** routes
everything through one hub — cheap to cable, but the hub is a fatal single point of failure. A **mesh**
connects everyone to everyone — bulletproof and lightning fast, but the cabling cost explodes. A
**ring** is tidy and cheap, yet a packet may crawl halfway around. Same eight nodes, opposite
trade-offs. **Topology** is the study of how a network's *shape* dictates its speed, cost and
resilience — and a handful of numbers let you compare any two designs at a glance.

<Callout type="intuition" title="The one-sentence idea">
A topology's character is captured by three measurements — **diameter** (worst-case hops),
**density** (links per possible link) and **clustering** (how tightly neighbourhoods knit) — and they
trade off against one another.
</Callout>

## Intuition: the three numbers that describe a shape

- **Diameter** — the *longest* shortest path between any two nodes. It is the worst-case latency the
  topology can ever impose. A mesh has diameter 1; a line of `n` nodes has diameter `n − 1`.
- **Average path length** — the typical hop count, the everyday latency rather than the worst case.
- **Density** — actual edges divided by the `n(n−1)/2` possible edges. High density buys short paths
  and redundancy but costs links (and, from the last lesson, kills bridges).
- **Clustering coefficient** — of a node's neighbours, what fraction are also connected to each other?
  High clustering means tight-knit local groups; it is the fingerprint of social and biological nets.

The famous **small-world** networks (think six degrees of separation) pull off the magic combination:
**high clustering** like a regular lattice, yet a **tiny diameter** like a random graph — a few
long-range "shortcut" edges collapse the distances.

## How it works

Diameter falls out of all-pairs shortest paths. On an unweighted graph, run a BFS from every node, note
the farthest distance each reaches (its **eccentricity**), and the maximum eccentricity is the
diameter:

```
ring of 6:  every node reaches its antipode in 3 hops  → diameter 3
star of 6:  any leaf → hub → any leaf is 2 hops         → diameter 2
line of 6:  the two ends are 5 hops apart               → diameter 5
```

## The real implementation

The course file computes all-pairs distances with Floyd-Warshall, then reads diameter, radius and
average path length straight off the matrix:

```ts
// /dsa/27-network-algorithms/network-topology.ts
diameter(): number {
  const dist = this.floydWarshall();           // O(V^3) all-pairs
  let maxDist = 0;
  for (let i = 0; i < dist.length; i++)
    for (let j = 0; j < dist[i].length; j++)
      if (dist[i][j] !== Infinity) maxDist = Math.max(maxDist, dist[i][j]);
  return maxDist;
}

averagePathLength(): number {
  const dist = this.floydWarshall();
  let total = 0, count = 0;
  for (let i = 0; i < dist.length; i++)
    for (let j = i + 1; j < dist.length; j++)
      if (dist[i][j] !== Infinity) { total += dist[i][j]; count++; }
  return count ? total / count : 0;
}
```

The same file ships `ClusteringCoefficient` (local and global), `NetworkDensity`,
`DegreeDistribution` (with a scale-free / power-law test), and a `SmallWorldDetector`.
Run the full file: `npx tsx 27-network-algorithms/network-topology.ts`.

## Complexity

Floyd-Warshall computes every pairwise distance directly; on sparse graphs, BFS-from-each-node is
cheaper:

$$
T_{\text{Floyd}}(V) = O(V^3), \qquad T_{\text{BFS-all}}(V, E) = O\big(V (V + E)\big).
$$

| Metric | What it tells you | Cost |
|--------|-------------------|------|
| Diameter | worst-case latency | $O(V^3)$ or $O(V(V+E))$ |
| Average path length | typical latency | same |
| Density | wiring cost / redundancy | $O(V + E)$ |
| Clustering coefficient | local cliquishness | $O(V \cdot d^2)$ |

<Callout type="why">
Diameter is a hard *upper bound* on how slow communication can get, no matter how clever your routing
is. Datacentre designers chase **low-diameter** topologies (fat-trees, hypercubes) precisely so that
no two machines are ever more than a few hops apart, even with tens of thousands of servers.
</Callout>

## The same idea, different worlds

<ExampleCard domain="Air travel" icon="🛫">
Airlines run a <strong>hub-and-spoke</strong> (star) topology: few routes to maintain, but every trip
detours through a hub. Low-cost carriers prefer point-to-point (denser) for shorter total travel.
</ExampleCard>

<ExampleCard domain="Blockchain" icon="⛓️">
A peer-to-peer crypto network's <strong>diameter</strong> bounds how long a transaction takes to reach
every node. Low diameter means faster consensus and fewer orphaned blocks.
</ExampleCard>

<ExampleCard domain="Neuroscience" icon="🧠">
The brain is a small-world network: dense local clusters (cortical columns) plus a few long-range
white-matter shortcuts give high clustering *and* a short diameter — efficient and robust.
</ExampleCard>

<ExampleCard domain="Electronics" icon="🔌">
Chip interconnects are chosen by topology: a mesh or torus on the die keeps the diameter low so cores
exchange cache lines in few hops, trading silicon area for latency.
</ExampleCard>

## Common pitfalls & interview tips

- **Diameter ignores disconnected pairs.** If the graph is in pieces, an unbounded distance is not a
  diameter — report the diameter per component, or say "infinite".
- **Diameter is worst-case, average path is typical.** A star has diameter 2 but a tiny average path;
  don't conflate the two when arguing about performance.
- **Density kills bridges but costs money.** Denser graphs have shorter paths and more redundancy
  (recall the last lesson) — but each extra link is real cable, ports and power.
- **Floyd-Warshall is `O(V^3)`.** Fine under ~500 nodes; beyond that, BFS/Dijkstra from each node on a
  sparse graph wins.
- **Interview cue:** "longest shortest path", "worst-case latency", or "compare two network designs"
  points at diameter and the topology metrics — and BFS layers are usually all you need.

## Practice

1. **Easy:** What is the diameter of a complete graph (mesh) on any number of nodes?
2. **Easy:** Compute the density of a tree on `n` nodes (it has `n − 1` edges).
3. **Medium:** Write `eccentricity(node)` and use it to derive both the diameter (max) and radius (min).
4. **Medium:** Compute the local clustering coefficient of a node: edges among its neighbours divided
   by the number of possible ones.
5. **Hard:** Detect a small-world network — show clustering far above, but average path length close
   to, an equivalent random graph.
