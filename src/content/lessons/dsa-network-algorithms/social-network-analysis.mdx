## The hook: the quiet person who runs the office

Every organisation has someone whose name is not on the org chart's top row but through whom every
deal somehow passes. They may know fewer people than the loud networker down the hall — yet remove
them and two halves of the company stop talking. Social network analysis makes that intuition
rigorous. The router-and-link machinery you have been building now answers a human question:
**who actually holds a network together?** The surprising answer is rarely the person with the most
connections.

<Callout type="intuition" title="The one-sentence idea">
**Betweenness centrality** ranks a node by how many shortest paths run *through* it — so a low-degree
**broker** bridging two communities can outrank a high-degree hub buried inside one.

</Callout>

## Intuition: four flavours of "important"

There is no single notion of centrality — each captures a different kind of power:

- **Degree centrality** — how many neighbours you have. Popularity. Cheap to compute, easily gamed.
- **Closeness centrality** — how *near* you are to everyone else on average. Low closeness means you
  hear news late; high closeness makes you an efficient broadcaster.
- **Betweenness centrality** — how often you sit *on the shortest path* between two other people. This
  is **broker power**: control over what flows, and a structural weak point if you leave.
- **PageRank** — importance that is *recursive*: you matter if important nodes point to you. Google's
  original web ranking and the basis of many recommendation systems.

Betweenness is computed by **Brandes' algorithm**: from each source, a BFS counts the number of
shortest paths to every node, then a reverse sweep distributes "dependency" credit back along those
paths. The nodes that accumulate the most credit are the ones traffic cannot avoid.

## How it works

Betweenness of node `w` sums, over every pair `(s, t)`, the fraction of `s`–`t` shortest paths that
pass through `w`. Brandes computes it efficiently per source `s`:

```
1. BFS from s, recording σ[v] = number of shortest paths s→v
   and the predecessors of v on those paths.
2. Pop nodes in reverse BFS order, pushing "dependency" credit
   δ[v] += (σ[v]/σ[w]) · (1 + δ[w]) to each predecessor v of w.
3. Add δ[w] (for w ≠ s) into betweenness[w].
```

A node on the only path between two big groups collects credit from *every* cross-group pair, so its
score dwarfs a node tucked inside a single clique.

## The real implementation

The course file is Brandes' algorithm verbatim — BFS to count shortest paths, then back-accumulate the
dependencies:

```ts
// /dsa/27-network-algorithms/social-networks.ts
calculate(): Map<number, number> {
  const betweenness = new Map<number, number>();
  for (const node of this.graph.keys()) betweenness.set(node, 0);

  for (const source of this.graph.keys()) {
    const stack: number[] = [], paths = new Map(), sigma = new Map(),
          distance = new Map(), delta = new Map();
    // ... init: sigma[source]=1, distance[source]=0 ...
    const queue = [source];
    while (queue.length) {                       // BFS: count shortest paths
      const v = queue.shift()!; stack.push(v);
      for (const w of this.graph.get(v) || []) {
        if (distance.get(w)! < 0) { queue.push(w); distance.set(w, distance.get(v)! + 1); }
        if (distance.get(w)! === distance.get(v)! + 1) {
          sigma.set(w, sigma.get(w)! + sigma.get(v)!);
          paths.get(w)!.push(v);
        }
      }
    }
    while (stack.length) {                        // reverse: accumulate dependency
      const w = stack.pop()!;
      for (const v of paths.get(w)!)
        delta.set(v, delta.get(v)! + (sigma.get(v)! / sigma.get(w)!) * (1 + delta.get(w)!));
      if (w !== source) betweenness.set(w, betweenness.get(w)! + delta.get(w)!);
    }
  }
  return betweenness; // normalised by (n-1)(n-2) in the full file
}
```

## Complexity

Brandes runs a BFS plus a back-sweep from every node — far cheaper than the naive all-pairs-paths
approach:

$$
T_{\text{betweenness}}(V, E) = O(V E), \qquad T_{\text{degree}}(V, E) = O(V + E).
$$

| Measure | Captures | Time |
|---------|----------|------|
| Degree | popularity | $O(V + E)$ |
| Closeness | broadcast efficiency | $O(V (V + E))$ |
| **Betweenness** (Brandes) | broker / bridge power | $O(V E)$ |
| PageRank | recursive prestige | $O(\text{iters} \cdot E)$ |

<Callout type="why">
Different questions want different centralities. "Who should we pay to advertise?" leans on degree or
PageRank (reach). "Which account, if banned, most fragments a disinformation ring?" is a betweenness
question (brokerage). Picking the wrong measure gives a confident, wrong answer.
</Callout>

## Common pitfalls & interview tips

- **Degree ≠ importance.** The whole point of betweenness is that a low-degree broker can be the single
  most critical node. Reaching for degree because it is easy is the classic mistake.
- **Normalise before comparing networks.** Raw betweenness grows with network size; divide by
  `(n−1)(n−2)` (directed) or half that (undirected) to compare across graphs.
- **Ties in shortest paths matter.** Brandes is correct *because* it tracks `σ`, the *count* of equal
  shortest paths, and splits credit proportionally. A version that keeps only one path is wrong.
- **Betweenness is expensive.** `O(VE)` is fine for thousands of nodes; for millions, use sampled or
  approximate betweenness rather than the exact algorithm.
- **Interview cue:** "most influential", "key connector", "remove to disconnect communities" → a
  centrality question; name degree, closeness, betweenness and PageRank and justify which one fits.

## Practice

1. **Easy:** In a star, what is the hub's betweenness relative to a leaf's, and why?
2. **Easy:** Compute degree centrality for a 5-node path graph by hand.
3. **Medium:** Implement closeness centrality with a BFS from each node and the reciprocal of total
   distance.
4. **Medium:** Run PageRank by power iteration on a tiny directed graph; show it converges and sums to 1.
5. **Hard:** Greedy influence maximisation — pick the `k` seeds that maximise expected spread under an
   independent-cascade simulation, and explain why greedy is near-optimal (submodularity).
