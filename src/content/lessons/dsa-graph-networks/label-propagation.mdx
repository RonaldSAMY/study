## The hook: rumours find the cliques

Start a different rumour at every person in a town. Each day, everyone switches to whichever rumour most of
their friends are currently telling. Tight friend groups quickly converge on one shared rumour — there is
no escaping the local majority — while the rumours that survive mark out exactly the communities. That is
**label propagation**: no modularity to maximise, no shortest paths to compute, no bridges to cut. Just a
trivially simple local rule, repeated until it settles, that runs in nearly linear time and powers real-time
clustering on graphs far too big for Louvain or Girvan-Newman.

<Callout type="intuition" title="The one-sentence idea">
Give every node its own label, then have each node repeatedly copy the label most common among its
neighbours — dense groups lock onto a single shared label, and those labels are the communities.
</Callout>

## Intuition: consensus with no referee

The previous algorithms all optimise something — Louvain maximises modularity, Girvan-Newman targets
betweenness. Label propagation optimises *nothing explicitly*. It just lets local agreement spread:

1. Every node starts with a unique label (its own id).
2. Sweep the nodes; each one adopts the label held by the most of its neighbours, breaking ties somehow.
3. Repeat until no node changes its label.

Inside a dense cluster, the first node to gain a small majority pulls its neighbours over, who reinforce the
majority, who pull in the rest — a fast cascade to one label. Across a sparse bridge there is no majority to
cross, so different clusters settle on different labels. Communities *emerge* from this local consensus
rather than being computed by any global objective. The catch: because there is randomness in the sweep
order and tie-breaking, you can get slightly different partitions on different runs (the demo and snippet
break ties at random but with a fixed seed, so replays are stable while still behaving like real LPA).

## How it works

The update rule for node $v$ chooses the label maximising its count among neighbours $N(v)$:

$$
\ell(v) \leftarrow \arg\max_{L} \; \bigl| \{\, u \in N(v) : \ell(u) = L \,\} \bigr|,
$$

with ties broken at random (seeded here, so the replay is stable). The loop is as short as algorithms get:

```
give every node a unique label
repeat:
  changed = false
  for each node v (some order):
    L* = most frequent label among v's neighbours (break ties at random)
    if L* != label(v): label(v) = L*; changed = true
until not changed
nodes sharing a label = one community
```

Trace two triangles, one on A-B-C and one on D-E-F, with a bridge $C-D$. Within each triangle every node sees two
neighbours; a label quickly wins the local majority and the triangle collapses to one colour. The lone bridge
edge cannot overturn a 2-vs-1 majority inside a triangle, so the two triangles settle on distinct labels.

## The real implementation

The course file shuffles the node order each pass and stops as soon as a full sweep makes no change.

```ts
// /dsa/29-graph-advanced/label-propagation.ts
detectCommunities(maxIterations = 100): Map<number, number> {
  for (let iter = 0; iter < maxIterations; iter++) {
    let changed = false;
    const shuffled = [...this.graph.nodes].sort(() => Math.random() - 0.5);   // random order
    for (const node of shuffled) {
      const newLabel = this.getMostFrequentLabel(node);                       // local majority
      if (newLabel !== this.labels.get(node)) {
        this.labels.set(node, newLabel);
        changed = true;
      }
    }
    if (!changed) break;   // converged: a full pass changed nothing
  }
  return this.labels;
}
```

## Complexity

Each pass touches every node and its edges once, and the number of passes to converge is tiny and roughly
constant in practice, so the algorithm is effectively linear:

$$
T = O(k \,(n + m)) \approx O(n + m), \qquad S = O(n + m),
$$

with $k$ the (small) pass count, $n$ nodes, $m$ edges.

| Step | Cost |
|------|------|
| One pass | $O(n + m)$ |
| Passes to converge | usually $< 10$ |
| Total (typical) | $O(n + m)$ |

<Callout type="why">
Label propagation is the cheapest community method by far — near-linear, no objective function, embarrassingly
parallel. It is the same trade ML makes when it picks a fast heuristic over an exact optimiser: you give up
the guarantee of a best partition for speed and simplicity, and on huge graphs that trade is usually worth it.
</Callout>

## Common pitfalls & interview tips

- **Non-determinism.** Random sweep order and random tie-breaking can yield different partitions across runs.
  Fix the order and tie-break rule (as here) for reproducibility, or aggregate several runs.
- **Oscillation.** With synchronous updates, two nodes can keep swapping labels forever; asynchronous
  (one-at-a-time) updates, used here, avoid that.
- **The monster community.** On graphs with no clear structure, one label can sweep everything into a single
  giant community — a sign the network simply has no strong communities, not a bug.
- **Interview cue:** "fastest community detection, no objective, just local majority voting" is label
  propagation. Mention near-linear time and the determinism/oscillation caveats.
