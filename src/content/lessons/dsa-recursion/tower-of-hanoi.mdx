## The hook: a legend about the end of the world

Monks in an ancient temple, the story goes, are moving 64 golden disks between three pegs under one
rule: never place a larger disk on a smaller one. When they finish, the world ends. Should you worry?
The minimum number of moves for $n$ disks is $2^n - 1$, so 64 disks need about $1.8 \times 10^{19}$
moves — at one move per second, **585 billion years**. The universe is safe. What makes the puzzle
beautiful is that this monstrous sequence is generated by a recursion so short you can hold all of it
in your head at once.

<Callout type="intuition" title="The one-sentence idea">
To move $n$ disks to the target peg: move the top $n-1$ out of the way, move the biggest disk across,
then move those $n-1$ back on top — each "$n-1$" handled by the very same recipe.
</Callout>

## Intuition: assume you can already do n−1

This is the leap-of-faith from the basics lesson, in its purest form. You have three pegs: **source**,
**auxiliary**, **destination**. You want all $n$ disks moved from source to destination.

Don't think about all $n$ disks. Think about just the **biggest** one at the bottom. To move it to the
destination, the other $n-1$ disks must first be sitting on the auxiliary peg, out of its way. So:

1. **Move $n-1$ disks** from source → auxiliary (trust the recursion to handle this).
2. **Move disk $n$** (the biggest) from source → destination. One single move.
3. **Move $n-1$ disks** from auxiliary → destination, on top of the big one.

The base case is $n = 0$: no disks, nothing to do. Every recursive call works on one fewer disk, so we
must reach it. Steps 1 and 3 are *the same problem on $n-1$ disks* — but with the roles of the pegs
swapped. That role-swap (auxiliary and destination trade places) is the only subtlety.

## How it works: a dry run

Three disks, source A → destination C, auxiliary B. The plan "move 2 to B, move disk 3 to C, move 2 to
C" expands to **7** moves:

1. disk 1: A → C
2. disk 2: A → B
3. disk 1: C → B  *(first two disks now parked on B)*
4. disk 3: A → C  *(the big one crosses, exactly once)*
5. disk 1: B → A
6. disk 2: B → C
7. disk 1: A → C  *(rebuild on top — done)*

## The real implementation

Faithful excerpt from the course file, including the state-tracking variant the demo is built on.

```ts
// /dsa/04-recursion/tower-of-hanoi.ts

// Print the moves — Time: O(2^n), Space: O(n) stack
function towerOfHanoi(
  n: number,
  source: Rod = 'A',
  destination: Rod = 'C',
  auxiliary: Rod = 'B'
): void {
  if (n === 0) return;
  towerOfHanoi(n - 1, source, auxiliary, destination);          // n-1 to spare
  console.log(`Move disk ${n} from ${source} to ${destination}`); // biggest across
  towerOfHanoi(n - 1, auxiliary, destination, source);          // n-1 back on top
}

// Minimum moves required is provably 2^n - 1
function minimumMoves(n: number): number {
  return Math.pow(2, n) - 1;
}
```

## Complexity

Let $M(n)$ be the number of moves. The recipe makes two calls on $n-1$ plus one move:

$$
M(n) = 2\,M(n-1) + 1, \qquad M(0) = 0.
$$

Unrolling gives $M(n) = 2^n - 1$ — exponential, and **optimal**: no sequence of legal moves can do
better.

| Quantity | Value | Why |
| --- | --- | --- |
| Moves (best = avg = worst) | $2^n - 1$ | forced by the recurrence; nothing is wasted |
| Time | $O(2^n)$ | one unit of work per move |
| Stack space | $O(n)$ | depth never exceeds the disk count |

<Callout type="why">
Like naive Fibonacci, Hanoi is $O(2^n)$ time — but for the opposite reason. Fibonacci is exponential
because it <em>wastes</em> work on repeated subproblems (memoizing fixes it). Hanoi is exponential
because the answer itself <em>contains</em> $2^n - 1$ moves — there is nothing to cache away. Same big-O,
totally different cause.
</Callout>

## The same idea, different worlds

<ExampleCard domain="Backups" icon="💾">
The "Tower of Hanoi" backup rotation schedules tape or disk reuse using the Hanoi move pattern: smaller
disks (used most often) map to media you recycle frequently, larger ones to media kept for the long
term. It naturally keeps a spread of backup ages with minimal media.
</ExampleCard>

<ExampleCard domain="Robotics" icon="🤖">
A robot arm restacking boxes faces the same constraint — you can't set a big box on a small one, and
you have limited buffer space. Planners decompose "move this stack" into "clear the top, move the base,
restore the top," which is Hanoi's recursion driving real motion.
</ExampleCard>

<ExampleCard domain="Neuropsychology" icon="🧠">
The Tower of Hanoi (and its cousin, the Tower of London) is a standard clinical test of **planning and
executive function**. Because the optimal solution requires looking several moves ahead, how a patient
plays reveals how their prefrontal-cortex planning is working.
</ExampleCard>

## Common pitfalls & interview tips

- **Mixing up the peg order.** The single most common bug. In step 1 the *auxiliary* peg is the
  temporary destination; in step 3 the original *source* becomes the temporary auxiliary. Pass the
  arguments in the right slots and the role-swap takes care of itself.
- **Wrong base case.** Use $n = 0$ (do nothing) or $n = 1$ (one move) — both work, but be consistent.
  An $n = 1$ base that forgets to actually record the move is a classic slip.
- **Expecting to beat $2^n - 1$.** You can't, with three pegs. (With *four* pegs the Frame–Stewart
  algorithm does better — a nice follow-up to mention.)
- **Trying to print all moves for large $n$.** $n = 40$ is a trillion moves. Compute the *count* with
  $2^n - 1$ instead of generating them.
- **Interview tip:** Hanoi is a favourite for testing whether you can *trust* a recursive subcall.
  Narrate "assume I can move $n-1$ disks; then the whole solution is just three steps" — that framing
  is what they're listening for.

## Practice

1. **Count moves (easy).** Return $2^n - 1$ directly and verify it equals the length of the generated
   move list.
2. **Return the move list (medium, from `recursion-practice.js`).** Produce
   `[{from: 'A', to: 'C'}, ...]` for $n$ disks — the practice file's Tower of Hanoi problem.
3. **Which peg holds disk $k$ after move $m$? (hard).** Track state through the sequence, like the
   demo's state-snapshot variant does.
4. **Four-peg Hanoi (hard).** Implement Frame–Stewart and compare its move count to $2^n - 1$ — the
   course file's `towerOfHanoi4Pegs` shows the memoized recurrence.
