## The hook: a fair draw from a river you can't hold

A firehose of log lines streams past — billions of them, far more than fit in memory, and you do not know
how many there will be. You want a *uniform* random sample of, say, 100 of them, and you get exactly one
look at each line as it flies by. You cannot store them all and pick at the end. **Reservoir sampling**
solves this beautifully: keep a small reservoir of $k$ items, and for each new arrival decide on the spot
whether it bumps out one of the current keepers — with a probability tuned so that, no matter when the
stream stops, every item seen so far has the same chance $k/n$ of being in your sample.

<Callout type="intuition" title="The one-sentence idea">
Keep the first $k$ items; then for the $i$-th item ($i > k$) replace a random reservoir slot with
probability $k/i$ — this exactly balances the books so every one of the $n$ items ends up equally likely.

</Callout>

## Intuition: the shrinking probability that keeps things fair

Start by filling the reservoir with the first $k$ items — with fewer than $k$ seen, everyone is obviously
in. Now item $i$ (1-indexed, $i > k$) arrives. Give it a chance $k/i$ of entering: draw a random slot
$j \in \{0, 1, \dots, i-1\}$; if $j < k$, item $i$ takes over reservoir slot $j$, otherwise it is dropped.

Why $k/i$? Two things must stay balanced as $i$ grows:

- **A newcomer's fair share.** When $i$ items have been seen, each *should* have probability $k/i$ of being
  held. Item $i$ enters with exactly that chance — correct by construction.
- **An old item's survival.** An item already in the reservoir survives step $i$ unless it is the unlucky
  slot chosen, i.e. with probability $1 - \tfrac{k}{i}\cdot\tfrac{1}{k} = 1 - \tfrac{1}{i} = \tfrac{i-1}{i}$.

Multiply an early item's survival across all later steps and the telescoping product collapses:

$$
P(\text{item survives to } n) = \frac{k}{i}\cdot\frac{i}{i+1}\cdot\frac{i+1}{i+2}\cdots\frac{n-1}{n} = \frac{k}{n}.
$$

Every numerator cancels the previous denominator, leaving $k/n$ — the same for every item, exactly as a
uniform sample demands. And it took one pass and only $k$ slots.

## How it works

Stream `[10, 20, 30, 40, 50]`, reservoir size $k = 2$:

```
i=1: 10 → fill      reservoir = [10, _]
i=2: 20 → fill      reservoir = [10, 20]
i=3: 30 → keep-prob 2/3: draw j in {0,1,2}; if j=1<2 → [10, 30]
i=4: 40 → keep-prob 2/4: draw j in {0..3}; if j=3≥2 → drop, [10, 30]
i=5: 50 → keep-prob 2/5: draw j in {0..4}; if j=0<2 → [50, 30]
```

Each survivor of the final reservoir had probability $2/5$ of being there — a fair sample, one pass, two slots.

function reservoirSample(stream, k) {
  const reservoir = [];
  let i = 0;
  for (const item of stream) {
    if (i < k) {
      reservoir.push(item);                 // fill phase
    } else {
      const j = Math.floor(Math.random() * (i + 1));   // j in [0, i]
      if (j < k) reservoir[j] = item;       // replace with probability k/(i+1)
    }
    i++;
  }
  return reservoir;
}

// k = 1 special case: pick ONE item uniformly (great for linked lists, LC 382).
function randomOne(stream) {
  let chosen = null, count = 0;
  for (const item of stream) {
    count++;
    if (Math.random() < 1 / count) chosen = item;   // keep the i-th with prob 1/i
  }
  return chosen;
}

  return [...hits.entries()].map(([v, c]) => [v, +(c / trials).toFixed(2)]);
}`}

## The real implementation

From the course file — the streaming sampler class (`add` items one at a time), plus **Algorithm L**, which
skips ahead over the items it can prove will be rejected for a big speedup on huge streams:

```ts
// /dsa/25-misc-algorithms/reservoir-sampling.ts
class ReservoirSampler<T> {
    private reservoir: T[] = [];
    private count = 0;
    constructor(private k: number) {}

    add(item: T): void {
        if (this.count < this.k) {
            this.reservoir.push(item);                       // fill phase
        } else {
            const j = Math.floor(Math.random() * (this.count + 1));
            if (j < this.k) this.reservoir[j] = item;        // replace with prob k/(count+1)
        }
        this.count++;
    }
    getSample(): T[] { return [...this.reservoir]; }
}

// Algorithm L: skip the doomed items instead of rolling a die for each one.
function algorithmL<T>(stream: T[], k: number): T[] {
    if (stream.length <= k) return [...stream];
    const reservoir = stream.slice(0, k);
    let W = Math.exp(Math.log(Math.random()) / k);
    let i = k;
    while (i < stream.length) {
        i += Math.floor(Math.log(Math.random()) / Math.log(1 - W)) + 1;  // jump ahead
        if (i < stream.length) {
            reservoir[Math.floor(Math.random() * k)] = stream[i];
            W *= Math.exp(Math.log(Math.random()) / k);
        }
    }
    return reservoir;
}
```

## Complexity

One pass, a fixed-size reservoir, and a constant-work decision per item:

$$
\text{time } O(n), \qquad \text{space } O(k), \qquad P(\text{item in sample}) = \frac{k}{n}.
$$

| Method | Passes | Memory | Needs $n$ up front? |
|--------|--------|--------|---------------------|
| Store all, then shuffle-pick | 1 (but stores all) | $O(n)$ | no |
| Random indices | needs random access | $O(k)$ | yes |
| **Reservoir sampling** | 1 | $O(k)$ | **no** |
| Algorithm L | 1 | $O(k)$ | no ($O(k\log\tfrac{n}{k})$ RNG calls) |

The headline is the $O(k)$ memory with no advance knowledge of $n$ — the only option when the stream is
unbounded or too large to store.

## Common pitfalls & interview tips

- **Use the right denominator.** For the $i$-th item (1-indexed) the replacement probability is $k/i$; with
  0-indexed loops that is `Math.random() * (i + 1)`. An off-by-one here quietly biases toward early or late items.
- **The reservoir isn't shuffled.** Reservoir sampling gives a uniform *set*, but the slot positions are not
  a uniform *ordering*. If you need a random permutation, shuffle the reservoir afterward.
- **Weighted sampling needs a different trick.** Plain reservoir sampling is uniform. For weights, use the
  A-Res key $u^{1/w}$ and keep the $k$ largest keys — do not just scale $k/i$.
- **Randomness quality matters.** For security-sensitive sampling, `Math.random` is not enough; use a CSPRNG.
  For huge streams, **Algorithm L** slashes the number of RNG calls.
- **Interview cue:** "one pass, unknown or unbounded length, uniform sample, limited memory" is the reservoir
  sign. For $k = 1$ (random node in a list) say "keep the $i$-th with probability $1/i$."
