## The hook: pre-sum the neighbourhood, not every house

You run a delivery network and constantly ask "what is the total demand between stop $L$ and stop $R$?"
Adding up every stop in the range is $O(n)$ per question, and there are thousands of questions. A full
segment tree gives $O(\log n)$ but is fiddly to write under time pressure. **Square-root decomposition** is
the pragmatic middle: chop the road into $\sqrt{n}$ chunks, keep a running total for each chunk, and then
any range is "a few loose stops on each edge, plus a handful of whole chunks jumped over instantly."

<Callout type="intuition" title="The one-sentence idea">
Group the array into blocks of size about $\sqrt{n}$ and precompute each block's answer, so a range query
touches at most two partial blocks element-by-element and the middle blocks in one step each — $O(\sqrt{n})$ total.
</Callout>

## Intuition: why √n is the sweet spot

Suppose you split $n$ elements into blocks of size $b$. A range $[L, R]$ then decomposes into three parts:

1. a **partial left edge** — up to $b$ loose elements you sum by hand;
2. a run of **whole blocks** in the middle — each contributes its precomputed answer in $O(1)$;
3. a **partial right edge** — again up to $b$ loose elements.

The edges cost $O(b)$; the whole blocks number at most $n/b$, so they cost $O(n/b)$. Total work is
$O(b + n/b)$. Calculus (or just trying values) says this is smallest when the two terms balance, at
$b = \sqrt{n}$, giving $O(\sqrt{n})$. That is the entire idea: **balance the cost of the edges against the
cost of the blocks.**

It is slower than a segment tree's $O(\log n)$, but it is short, general, and handles weird queries
(mode, count-less-than, frequency) that do not fit a clean tree — which is exactly why competitive
programmers keep it in their back pocket.

## How it works

For `[3,1,4,1,5,9,2,6,5,3,5,8,9,7,9,3]` ($n = 16$, block size $4$) the block sums are
`[9, 22, 21, 28]`. To answer `sum[2, 10]`:

```
blocks:  [3 1 4 1] [5 9 2 6] [5 3 5 8] [9 7 9 3]
                ↑ L=2                ↑ R=10
left edge (2..3): 4 + 1            = 5
whole block 1:    22 (in one step) = 22
right edge (8..10): 5 + 3 + 5      = 13
total = 5 + 22 + 13 = 40
```

Only two edge scans and one block lookup — not nine additions.

function makeSqrtSum(arr) {
  const bs = Math.max(1, Math.floor(Math.sqrt(arr.length)));
  const blockSum = [];
  for (let i = 0; i < arr.length; i++) {
    const b = Math.floor(i / bs);
    blockSum[b] = (blockSum[b] || 0) + arr[i];
  }
  function query(L, R) {
    let sum = 0;
    const lb = Math.floor(L / bs), rb = Math.floor(R / bs);
    if (lb === rb) { for (let i = L; i <= R; i++) sum += arr[i]; return sum; }
    for (let i = L; i < (lb + 1) * bs; i++) sum += arr[i];   // left edge
    for (let b = lb + 1; b < rb; b++) sum += blockSum[b];    // whole blocks
    for (let i = rb * bs; i <= R; i++) sum += arr[i];        // right edge
    return sum;
  }
  function update(i, val) { blockSum[Math.floor(i / bs)] += val - arr[i]; arr[i] = val; }
  return { query, update };
}

## The real implementation

From the course file — the range-sum class with an $O(1)$ point update, plus the count-with-sorted-blocks
variant that answers "how many elements $\le x$ in $[L,R]$":

```ts
// /dsa/25-misc-algorithms/sqrt-decomposition.ts
class SqrtSum {
    private arr: number[];
    private blockSize: number;
    private blockSums: number[];

    constructor(arr: number[]) {
        this.arr = [...arr];
        this.blockSize = Math.max(1, Math.floor(Math.sqrt(arr.length)));
        this.blockSums = [];
        this.build();
    }
    private build(): void {
        const numBlocks = Math.ceil(this.arr.length / this.blockSize);
        this.blockSums = Array(numBlocks).fill(0);
        for (let i = 0; i < this.arr.length; i++)
            this.blockSums[Math.floor(i / this.blockSize)] += this.arr[i];
    }
    update(index: number, value: number): void {
        const b = Math.floor(index / this.blockSize);
        this.blockSums[b] += value - this.arr[index];   // O(1): patch one block
        this.arr[index] = value;
    }
    query(left: number, right: number): number {
        let sum = 0;
        const lb = Math.floor(left / this.blockSize);
        const rb = Math.floor(right / this.blockSize);
        if (lb === rb) { for (let i = left; i <= right; i++) sum += this.arr[i]; return sum; }
        for (let i = left; i < (lb + 1) * this.blockSize; i++) sum += this.arr[i];
        for (let b = lb + 1; b < rb; b++) sum += this.blockSums[b];
        for (let i = rb * this.blockSize; i <= right; i++) sum += this.arr[i];
        return sum;
    }
}
```

## Complexity

With block size $b$, the cost of one query is edges plus blocks:

$$
T(b) = O\!\left(b + \frac{n}{b}\right), \qquad \text{minimized at } b = \sqrt{n} \;\Rightarrow\; O(\sqrt{n}).
$$

| Operation | This structure | Segment tree / BIT |
|-----------|----------------|--------------------|
| Build | $O(n)$ | $O(n)$ |
| Range query | $O(\sqrt{n})$ | $O(\log n)$ |
| Point update | $O(1)$–$O(\sqrt{n})$ | $O(\log n)$ |
| Space | $O(\sqrt{n})$ | $O(n)$ |

For $n = 10^6$, $\sqrt{n} = 1000$ versus $\log_2 n \approx 20$ — the tree is faster, but the block method is
a few lines and bends to almost any query.

## Common pitfalls & interview tips

- **Same-block queries must fall back to a scan.** When $L$ and $R$ live in one block there are no whole
  blocks to jump — loop directly. Forgetting this branch double-counts or misses elements.
- **Ragged last block.** With $n$ not a perfect square, the final block is short. Always clamp the block
  end to $n-1$; a naive `start + b - 1` can read past the array.
- **Rebuild vs. patch on update.** Sum patches a block in $O(1)$; min/max needs an $O(\sqrt{n})$ block
  rebuild because you cannot "subtract" a minimum. Know which your operation allows.
- **Range updates want lazy tags.** To add a value to a whole range, store a per-block "pending add" and
  push it down only when you touch that block's individual elements.
- **Interview cue:** "range queries, updates allowed, but the operation is unusual (mode, k-th, count-less-than)"
  — reach for blocks first; they are quicker to code correctly than a custom segment tree.

## Practice

1. **Easy:** For $n = 100$, what block size minimizes $b + n/b$, and what is that minimum?
2. **Easy:** Given block sums `[9, 22, 21, 28]` (block size 4), answer `sum[4, 7]` with one lookup.
3. **Medium:** Add a `queryMin(L, R)` method; explain why a point update now costs $O(\sqrt{n})$, not $O(1)$.
4. **Medium:** Implement a lazy `rangeAdd(L, R, v)` using a per-block pending-add array.
5. **Hard:** Support "count elements $\le x$ in $[L,R]$" by keeping each block sorted and binary-searching the whole blocks.
