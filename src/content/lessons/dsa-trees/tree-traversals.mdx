## The hook: same tree, four different to-do lists

You have a tree of folders and you want to do something to every item. Copying the structure, you must
create a folder *before* its contents — touch the parent first. Computing each folder's total size, you
need every child's size *before* the parent — touch the parent last. Listing a sorted index, you want a
left-to-right sweep. Printing the table of contents level by level, you go breadth-first. Same tree, four
visiting orders, each one exactly right for a different task.

<Callout type="intuition" title="The one-sentence idea">
A traversal visits every node exactly once; the only choice is *when you process the root relative to its
children* — before (preorder), between (inorder), after (postorder), or whole-level-at-a-time (level
order).
</Callout>

## Intuition: where does the root go?

Three of the four traversals are depth-first: dive down one branch fully before backing up. They differ
only in where the root sits in the recursion:

- **Preorder** — Root, Left, Right. Process the node, *then* its subtrees. Great for copying or
  serializing a tree (you must make a parent before its children).
- **Inorder** — Left, Root, Right. Process the left subtree, then the node, then the right. On a search
  tree this spits out values in **sorted order** — the fact the next lesson leans on.
- **Postorder** — Left, Right, Root. Process both subtrees, *then* the node. Great for deleting a tree or
  computing a folder's size, where children must finish first.

The fourth, **level order**, is breadth-first: visit depth 0, then all of depth 1, then depth 2. It uses a
**queue** instead of recursion's stack, and it is how you find the shallowest leaf or read a tree row by
row.

## How it works

The three depth-first orders are *the same three lines in a different order*. Look how only the position of
the `visit` line moves:

```
preorder(n):  visit(n); preorder(n.left);  preorder(n.right)
inorder(n):   inorder(n.left);  visit(n);  inorder(n.right)
postorder(n): postorder(n.left); postorder(n.right); visit(n)
```

Level order breaks the pattern: it pulls nodes from a queue and pushes their children, so siblings come
out together before any of their children do.

## The real implementation

From the course file — the recursive trio and the queue-based level order that returns one array per
level:

```ts
// /dsa/09-trees/tree-traversals.ts
function inorderRecursive<T>(root: TreeNode<T> | null): T[] {
    const result: T[] = [];
    function traverse(node: TreeNode<T> | null): void {
        if (!node) return;
        traverse(node.left);        // Left
        result.push(node.value);    // Root
        traverse(node.right);       // Right
    }
    traverse(root);
    return result;
}

function levelOrderLevels<T>(root: TreeNode<T> | null): T[][] {
    if (!root) return [];
    const result: T[][] = [];
    const queue: TreeNode<T>[] = [root];
    while (queue.length > 0) {
        const levelSize = queue.length;          // freeze this level's width
        const currentLevel: T[] = [];
        for (let i = 0; i < levelSize; i++) {
            const node = queue.shift()!;
            currentLevel.push(node.value);
            if (node.left) queue.push(node.left);
            if (node.right) queue.push(node.right);
        }
        result.push(currentLevel);
    }
    return result;
}
```

## Complexity

Every traversal touches each node once. Depth-first uses stack space proportional to the height; level
order uses queue space proportional to the tree's widest level:

$$
T(n) = O(n), \qquad S_{\text{DFS}}(n) = O(h), \qquad S_{\text{BFS}}(n) = O(w),
$$

where $h$ is the height and $w$ is the maximum width (which can reach $n/2$ at the bottom of a full tree).

| Traversal | Time | Extra space | Signature use |
|-----------|------|-------------|---------------|
| Preorder | $O(n)$ | $O(h)$ | copy / serialize |
| Inorder | $O(n)$ | $O(h)$ | sorted output of a BST |
| Postorder | $O(n)$ | $O(h)$ | delete / evaluate |
| Level order | $O(n)$ | $O(w)$ | shallowest node, row-by-row |

## The same idea, different worlds

</ExampleCard>

<ExampleCard domain="Publishing" icon="📚">
Printing a book's table of contents is <strong>preorder</strong>: write the chapter title, then recurse
into its sections and subsections in order — parent heading before its contents.
</ExampleCard>

<ExampleCard domain="Logistics" icon="📦">
Spreading news through a reporting hierarchy "one management layer at a time" is <strong>level order</strong>:
the queue holds everyone at the current layer before reaching anyone below — the same wavefront as a BFS.
</ExampleCard>

<ExampleCard domain="Mathematics" icon="🧮">
Evaluating an arithmetic expression tree is <strong>postorder</strong>: compute both operand subtrees
before applying the operator at the node. Reverse Polish notation is literally a postorder reading.
</ExampleCard>

## Common pitfalls & interview tips

- **Inorder = sorted only for a BST.** On an arbitrary binary tree, inorder is just Left-Root-Right with no
  sortedness promised.
- **Level order needs a queue, not a stack.** Swap in a stack and you get a depth-first order instead. To
  return one array *per level*, freeze `levelSize = queue.length` before the inner loop.
- **Recursion depth is the height.** A degenerate tree of height $n$ can blow the call stack; an iterative
  traversal with an explicit stack avoids it.
- **Interview cue:** reconstructing a tree needs **two** traversals, and one of them must be inorder —
  preorder (or postorder) alone is ambiguous because it can't tell left subtree from right.

2. **Easy:** Which traversal lists a binary search tree's values in increasing order?
3. **Medium:** Convert the recursive preorder to an iterative one using an explicit stack. Why do you push
   the right child before the left?
4. **Medium:** Modify level order to return a 2-D array, one inner array per level (freeze the level size).
5. **Hard:** Given a tree's preorder and inorder arrays, reconstruct it. Why is inorder indispensable, and
   why is preorder-plus-postorder generally not enough?
