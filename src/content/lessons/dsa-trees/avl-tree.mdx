## The hook: the BST that refuses to lean

The previous lesson left us with a gnawing weakness: feed a binary search tree sorted data and it
collapses into a leaning chain, and the dictionary trick rots back into a page-by-page scan. In 1962,
Adelson-Velsky and Landis fixed this for good. Their idea: after every insert or delete, check how lopsided
each node has become, and if any node tips too far, perform a small local **rotation** that restores
balance — instantly, in constant work. The result is a BST that *cannot* lean: its height stays near
$\log n$ no matter what order the data arrives in.

<Callout type="intuition" title="The one-sentence idea">
An AVL tree is a BST that stores a height at each node and rotates the moment any node's two subtrees
differ in height by more than one, keeping the whole tree balanced and every operation O(log n).
</Callout>

## Intuition: one number, four rotations

The watchdog is the **balance factor** of a node:

$$
\text{bf}(node) = \text{height}(node.\text{left}) - \text{height}(node.\text{right}).
$$

The AVL invariant is simply that bf stays in the set of allowed values −1, 0, +1 at every node. An insert
can push some ancestor's bf to +2 (too left-heavy) or −2 (too right-heavy). Exactly four shapes can cause
this, and each has a fixed repair:

- **Left-Left (bf +2, new node in the left child's left):** one **right rotation**.
- **Right-Right (bf −2, new node in the right child's right):** one **left rotation**.
- **Left-Right (bf +2, but new node in the left child's right):** rotate the left child **left**, then the
  node **right** — a double rotation.
- **Right-Left (bf −2, new node in the right child's left):** rotate the right child **right**, then the
  node **left**.

A rotation just re-points three links and updates two heights — constant time — and it strictly lowers the
subtree's height, so one fix per insert is always enough.

## How it works

Insertion is ordinary BST insertion, plus one extra act on the way back up the recursion: update this
node's height, read its balance factor, and if it is out of range, rotate based on which of the four shapes
occurred.

```
insert(node, value):
  do the normal BST insert, recursing left or right
  updateHeight(node)
  b = balanceFactor(node)
  if b > 1 and value < node.left.value:   return rotateRight(node)          # LL
  if b > 1 and value > node.left.value:   node.left  = rotateLeft(node.left);  return rotateRight(node)  # LR
  if b < -1 and value > node.right.value: return rotateLeft(node)           # RR
  if b < -1 and value < node.right.value: node.right = rotateRight(node.right); return rotateLeft(node)  # RL
  return node
```

## The real implementation

From the course file — the two single rotations and the rebalance dispatcher that recognizes which of the
four cases applies by inspecting the child's balance factor:

```ts
// /dsa/09-trees/avl-tree.ts
private rotateRight(y: AVLNode<T>): AVLNode<T> {
    const x = y.left!;
    const T2 = x.right;
    x.right = y;  y.left = T2;        // re-point three links
    this.updateHeight(y);             // y first (now lower), then x
    this.updateHeight(x);
    return x;                         // x is the new subtree root
}

private rebalance(node: AVLNode<T>): AVLNode<T> {
    this.updateHeight(node);
    const balance = this.getBalanceFactor(node);
    if (balance > 1) {                                   // left heavy
        if (this.getBalanceFactor(node.left) < 0)        // Left-Right
            node.left = this.rotateLeft(node.left!);
        return this.rotateRight(node);                   // Left-Left finish
    }
    if (balance < -1) {                                  // right heavy
        if (this.getBalanceFactor(node.right) > 0)       // Right-Left
            node.right = this.rotateRight(node.right!);
        return this.rotateLeft(node);                    // Right-Right finish
    }
    return node;                                         // already balanced
}
```

## Complexity

Because the height is held to $O(\log n)$, the BST walk is $O(\log n)$, and the rebalancing along the path
adds only constant work per level:

$$
T_{\text{search}} = T_{\text{insert}} = T_{\text{delete}} = O(\log n), \qquad S(n) = O(n).
$$

The AVL height bound is provably tight: the worst-case AVL tree of height $h$ holds at least
$F_{h+2} - 1$ nodes (a Fibonacci count), which forces $h \le 1.44 \log_2(n+2)$ — never more than about 44%
taller than a perfect tree.

| Operation | Plain BST (worst) | AVL (guaranteed) |
|-----------|-------------------|------------------|
| Search | $O(n)$ | $O(\log n)$ |
| Insert | $O(n)$ | $O(\log n)$ |
| Delete | $O(n)$ | $O(\log n)$ |
| Height | up to $n - 1$ | $\le 1.44 \log_2 n$ |

The price of the guarantee is bookkeeping: a height field per node and up to two rotations per update. When
writes vastly outnumber reads, a looser red-black tree rotates less; when reads dominate, AVL's tighter
balance makes lookups faster.

## Common pitfalls & interview tips

- **Update heights bottom-up, in the right order.** In a rotation the demoted node becomes lower, so update
  *its* height first, then the new subtree root's.
- **Single vs double.** Left-Left and Right-Right need one rotation; Left-Right and Right-Left need two. The
  giveaway is a zig-zag: if the new node went into the *opposite-side* grandchild, you need the double.
- **Don't forget deletion.** Deletion can trigger rotations too, and unlike insertion it may cascade —
  rebalance every node on the way back up, not just the first unbalanced one.
- **Interview cue:** if asked for "guaranteed $O(\log n)$ ordered operations," name a self-balancing BST and
  explain the balance factor and the four rotation cases — that, plus "inorder is still sorted," is the
  whole story.

## Practice

1. **Easy:** Insert `10,20,30` into an empty AVL tree. Which single rotation fires, and what becomes the root?
2. **Easy:** Compute the balance factor of a node whose left subtree has height 3 and right subtree has
   height 1. Is it within the AVL invariant?
3. **Medium:** Insert `30,10,20` and identify the case (Left-Right), then list the two rotations in order.
4. **Medium:** Implement `rotateLeft` and `rotateRight` and verify with assertions that an inorder traversal
   is unchanged by a rotation (rotations preserve BST order).
5. **Hard:** Extend insertion to deletion: after removing a node, rebalance up the path and show that a
   single deletion may require rotations at more than one ancestor.
