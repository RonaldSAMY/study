## The hook: why is seat 14F instant but boarding so slow?

You walk into a stadium and find seat 14F without scanning a single other seat — the seat number *is* the
address. Yet if one more guest squeezes into the front of your row, everyone already seated has to shuffle
down one place. An array behaves exactly like that row of seats: it is a single, unbroken strip of memory.
That one design choice explains both its superpower (instant lookup) and its weakness (shifting on insert
and delete).

<Callout type="intuition" title="The one-sentence idea">
Because an array stores elements back-to-back in memory, the position of element `i` is pure arithmetic —
so reading it is O(1) — but making room at the front means physically sliding every other element, which is
O(n).
</Callout>

## Intuition: an address, not a search

An array reserves a contiguous block of memory and lays the elements out in order. If the block starts at
address `start` and each element takes `elementSize` bytes, then element `i` lives at

$$
\text{address}(i) = \text{start} + i \times \text{elementSize}.
$$

No scanning, no comparisons — one multiply and one add. That is why `arr[i]` is O(1) no matter how big the
array is. The flip side: there are no gaps. To insert at the front you must first move every existing
element one slot to the right to free up index 0, and to delete the front you slide everyone left to close
the hole. Appending or removing at the *end* touches nothing else, so it stays O(1). Keep that asymmetry in
mind — it drives almost every array technique that follows.

## How it works

Insert `9` at the front of `[3, 1, 4, 1, 5]`. We work from the right so we never overwrite a value we still
need:

```
[3, 1, 4, 1, 5, _]      start: one free slot at the end
[3, 1, 4, 1, _, 5]      move index 4 → 5
[3, 1, 4, _, 1, 5]      move index 3 → 4
[3, 1, _, 4, 1, 5]      move index 2 → 3
[3, _, 1, 4, 1, 5]      move index 1 → 2
[_, 3, 1, 4, 1, 5]      move index 0 → 1
[9, 3, 1, 4, 1, 5]      write 9 into slot 0   → 5 shifts for ONE insert
```

Five elements, five shifts — for an array of length $n$ that is $n$ moves, i.e. O(n). Now compare a few of
the cheap, end-or-index operations that cost only O(1) or a single O(n) scan:

## The real implementation

Straight from the course file — the linear-scan and constant-time helpers, including the two-pointer
`reverseInPlace` you will lean on in the next lesson:

```ts
// /dsa/01-arrays/array-basics.ts
function sumArray(arr: number[]): number {
    return arr.reduce((sum, num) => sum + num, 0);     // O(n) scan
}

function rotateArray<T>(arr: T[], k: number): T[] {    // O(n)
    if (arr.length === 0) return [];
    k = k % arr.length;
    if (k < 0) k += arr.length;
    return [...arr.slice(-k), ...arr.slice(0, -k)];
}

function swap<T>(arr: T[], i: number, j: number): void {   // O(1)
    [arr[i], arr[j]] = [arr[j], arr[i]];
}

function reverseInPlace<T>(arr: T[]): T[] {             // O(n) time, O(1) space
    let left = 0, right = arr.length - 1;
    while (left < right) {
        swap(arr, left, right);
        left++;
        right--;
    }
    return arr;
}
```

## Complexity

The whole story is in *where* you touch the array:

$$
\text{access} = O(1), \qquad \text{insert/delete at front} = O(n), \qquad \text{append at end} = O(1)\ \text{(amortized)}.
$$

| Operation | Best | Average | Worst |
|-----------|------|---------|-------|
| Access by index | $O(1)$ | $O(1)$ | $O(1)$ |
| Search (unsorted) | $O(1)$ | $O(n)$ | $O(n)$ |
| Insert / delete at end | $O(1)$ | $O(1)$ | $O(n)$ (resize) |
| Insert / delete at front | $O(n)$ | $O(n)$ | $O(n)$ |

The "worst" for an end-append is the rare moment the underlying buffer is full and must be copied to a
bigger one — averaged over many appends, that cost vanishes, which is what *amortized O(1)* means.

## Common pitfalls & interview tips

- **`splice`/`unshift`/`shift` hide an O(n) loop.** `arr.unshift(x)` looks like one call but shifts every
  element. In a loop that quietly becomes O(n²).
- **Length is not capacity.** Dynamic arrays over-allocate; an append is usually O(1) but occasionally
  triggers a full O(n) copy to a larger buffer. Amortized, appends are O(1).
- **Prefer the end.** If you are building a result, `push` to the end and reverse once at the end rather
  than `unshift`-ing repeatedly.
- **Interview cue:** if a problem says "in place" and "O(1) extra space", it is hinting you should move data
  with index arithmetic or two pointers — not allocate a second array.

## Practice

1. **Easy (Two Sum, LeetCode 1):** return the indices of the two numbers that add to a target. Brute force
   is O(n²); a hash map gets you to O(n).
2. **Easy (Contains Duplicate, LeetCode 217):** return `true` if any value appears at least twice. A `Set`
   solves it in O(n).
3. **Medium (Rotate Array, LeetCode 189):** rotate right by `k`. Can you do it in O(1) extra space with the
   reverse-three-times trick?
4. **Medium (Product of Array Except Self, LeetCode 238):** build the prefix-and-suffix product array with
   no division and O(1) extra space (output aside).
5. **Hard (Trapping Rain Water, LeetCode 42):** compute trapped water; the O(1)-space answer uses two
   pointers walking inward — a perfect bridge to the next lesson.
