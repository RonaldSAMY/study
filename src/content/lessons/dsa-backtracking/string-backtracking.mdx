## The hook: where do the scissors go?

You have the string `"aab"` and a rule: every piece you cut must be a palindrome. Where do you place the
scissors? After the first `a`? After `aa`? After the whole thing? Each cut is a **choice**, and once you have
made it you face a smaller version of the exact same question on the leftover tail. That recursive shape —
choose a prefix, recurse on the rest — is string backtracking.

The board is gone, but the engine from N-Queens is untouched: **choose** the next piece, **explore** the
remaining suffix, and **undo** the cut when that path dead-ends. The constraints just moved from "no two
queens attack" to "this substring is a palindrome" or "this segment is a valid IP octet". The frontier is
always a single index: where does the next piece start?

<Callout type="intuition" title="The one-sentence idea">
To split a string under a rule, try every prefix as the next piece; when a prefix is <strong>valid</strong>,
recurse on the remaining suffix, then pop the piece and slide the cut one character further.
</Callout>

## Intuition: one cursor, every prefix

Fix a `start` index — everything before it is already committed to pieces. Now let `end` sweep from `start`
forward. Each `end` defines a candidate piece `s[start..end]`. If that piece satisfies the rule, push it and
recurse with the cursor moved to `end + 1`; otherwise skip it. When `start` reaches the end of the string,
every character has been consumed by valid pieces — record the partition.

This is the same forward-only cursor you saw in combinations, but the "elements" are now *substrings of
increasing length* rather than array indices. Two decisions define any string-backtracking problem:

- **The validity test.** Palindrome? A number in $[0, 255]$? A word in the dictionary? That predicate is the
  only pruning you get — a stronger test cuts more branches.
- **The stop condition.** Usually "cursor reached the end", sometimes with an extra count (an IP address needs
  exactly 4 pieces).

Everything else is the unchanging choose/recurse/undo loop.

## How it works

The palindrome-partition template — one cursor, every prefix, validity gate before the recursion:

```ts
function partition(s) {
  const result = [];
  const isPal = (l, r) => { while (l < r) { if (s[l++] !== s[r--]) return false; } return true; };
  function backtrack(start, current) {
    if (start === s.length) { result.push([...current]); return; }  // whole string consumed
    for (let end = start; end < s.length; end++) {
      if (!isPal(start, end)) continue;          // prune: not a palindrome
      current.push(s.slice(start, end + 1));      // choose the piece
      backtrack(end + 1, current);                // explore the suffix
      current.pop();                              // undo the cut
    }
  }
  backtrack(0, []);
  return result;
}
```

**Dry run** of `partition("aab")`:

```
start 0: "a" ✓ → start 1: "a" ✓ → start 2: "b" ✓ → ["a","a","b"] ✓
                  start 1: "ab" ✗ (not a palindrome) → skip
         "aa" ✓ → start 2: "b" ✓ → ["aa","b"] ✓
         "aab" ✗ (not a palindrome) → skip
```

## The real implementation

The course file uses the same template; the only difference from the demo is that the validity check is a
tight two-pointer palindrome test that avoids allocating substrings until a piece is actually taken:

```ts
// /dsa/15-backtracking/string-backtracking.ts
function partition(s: string): string[][] {
    const result: string[][] = [];

    function isPalindrome(str: string, left: number, right: number): boolean {
        while (left < right) {
            if (str[left++] !== str[right--]) return false;
        }
        return true;
    }

    function backtrack(start: number, current: string[]): void {
        if (start === s.length) {
            result.push([...current]);
            return;
        }

        for (let end = start; end < s.length; end++) {
            if (isPalindrome(s, start, end)) {
                current.push(s.substring(start, end + 1));   // choose
                backtrack(end + 1, current);                 // explore
                current.pop();                               // undo
            }
        }
    }

    backtrack(0, []);
    return result;
}
```

When the same string is partitioned repeatedly, the file also ships `partitionOptimized`, which precomputes a
`dp[i][j] = "is s[i..j] a palindrome"` table so each check becomes $O(1)$ — a first taste of the
memoization that the DP track builds on:

```ts
// /dsa/15-backtracking/string-backtracking.ts
for (let i = n - 1; i >= 0; i--) {
    for (let j = i; j < n; j++) {
        if (s[i] === s[j] && (j - i <= 2 || dp[i + 1][j - 1])) {
            dp[i][j] = true;                 // s[i..j] palindromic
        }
    }
}
```

## Complexity

A length-$n$ string has $n - 1$ gaps, each either cut or not, so there are up to $2^{\,n-1}$ ways to split it.
Palindrome partitioning may generate all of them in the worst case (think `"aaaa"`), and copying each partition
of up to $n$ pieces adds a factor of $n$:

$$
T_{\text{partition}}(n) = O\!\left(n \cdot 2^{\,n}\right), \qquad \text{gaps} = n - 1 \Rightarrow 2^{\,n-1}\ \text{splits}.
$$

| Problem | Splits explored | Validity test | Time | Space |
|---------|-----------------|---------------|------|-------|
| Palindrome partition | up to $2^{\,n-1}$ | palindrome, $O(n)$ | $O(n \cdot 2^{n})$ | $O(n)$ |
| Palindrome partition (DP table) | up to $2^{\,n-1}$ | table lookup, $O(1)$ | $O(n \cdot 2^{n})$ | $O(n^{2})$ |
| Restore IP addresses | $\le 3^{4}$ | octet in $[0,255]$ | $O(1)$ | $O(1)$ |

Restore IP is bounded: exactly 4 pieces of length 1–3 cap the tree at $3^4 = 81$ leaves, which is why its cost
is effectively constant regardless of a longer-looking input.

## Common pitfalls & interview tips

- **Off-by-one on the slice.** The piece is `s.slice(start, end + 1)` (inclusive `end`) and the recursion moves
  to `end + 1`. Drop the `+ 1` and you either lose a character or loop forever.
- **Testing validity after recursing.** Gate the piece *before* you descend — that skip is your only pruning.
  Recursing first and validating later explores dead branches needlessly.
- **Forgetting the pop.** `current.push` must be mirrored by `current.pop` after the recursive call, exactly as
  queens were removed on backtrack; otherwise pieces leak into sibling paths.
- **Leading zeros / bounds in IP.** `"010"` and `"256"` are invalid octets — a subtle validity bug that passes
  small tests and fails the tricky ones.
- **Interview cue:** "partition the string so that…", "all valid ways to split", "segment into dictionary
  words", "restore/format this string" all signal a cursor-and-prefix backtracking solution.
