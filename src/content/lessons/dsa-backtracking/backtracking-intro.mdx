## The hook: finding your way out of a maze

Imagine standing in a hedge maze. You pick a corridor, walk until it dead-ends, then back up to the last
junction and try a different turn. You never teleport; you only ever *undo your last step* and try the next
option. Keep doing that and you will eventually walk every reachable path — and find every exit.

That is **backtracking**: a disciplined way to explore all possibilities by building a solution one choice
at a time, and *un-making* a choice the moment it leads nowhere. It is the universal tool for "generate every
arrangement" and "find every valid configuration" problems, from passwords to Sudoku.

<Callout type="intuition" title="The one-sentence idea">
Backtracking explores a tree of choices depth-first: at each node you <strong>choose</strong> an option,
<strong>explore</strong> what it leads to, then <strong>un-choose</strong> it and move on — pruning any
branch that can no longer succeed.
</Callout>

## Intuition: choose, explore, un-choose

Every backtracking problem is secretly a **tree**. The root is "nothing decided yet". Each edge is one
decision, and each leaf is a finished candidate. Backtracking is just a depth-first walk of that tree with
three moves repeated forever:

1. **Choose** — commit to one option (add it to the current partial solution).
2. **Explore** — recurse to decide the next thing, given that choice.
3. **Un-choose** — remove the option so the slot is clean for the next sibling option.

That third step is the soul of the technique. Because you faithfully undo every choice on the way back up,
the same `path` variable is reused for the entire search — no copying, no leftover state.

The other superpower is **pruning**. If a partial solution already violates a constraint (the running sum
overshot the target, a queen is attacked, a piece is not a palindrome), there is no reason to explore deeper:
every descendant inherits the violation. Cutting that branch can turn an impossible search into a fast one.

## How it works

Here is the template every backtracking solution is a variation of. Memorise its shape and most "generate
all" problems become fill-in-the-blanks:

```ts
function backtrack(path, choices) {
  if (isComplete(path)) {            // base case: a finished candidate
    results.push([...path]);         // copy — path keeps mutating
    return;
  }
  for (const choice of choices) {
    if (!isValid(choice, path)) continue;  // PRUNE invalid options
    path.push(choice);               // CHOOSE
    backtrack(path, nextChoices);    // EXPLORE
    path.pop();                      // UN-CHOOSE
  }
}
```

**Dry run** of subset-sum on `[2, 3, 5]` with target `5`, starting from the empty set:

```
{}            sum 0
 ├ {2}        sum 2
 │  ├ {2,3}   sum 5  ✓ accept

 ├ {3}        sum 3
 │  └ {3,5}   sum 8  ✗ prune
 └ {5}        sum 5  ✓ accept
```

## The real implementation

The course file shows the same skeleton powering the full **power set** (every node is a valid subset) plus
the sorted, pruned subset-sum:

```ts
// /dsa/15-backtracking/backtracking-basics.ts
function subsets<T>(nums: T[]): T[][] {
    const result: T[][] = [];
    function backtrack(index: number, current: T[]): void {
        result.push([...current]);            // every path is a subset
        for (let i = index; i < nums.length; i++) {
            current.push(nums[i]);            // choose
            backtrack(i + 1, current);        // explore
            current.pop();                    // un-choose
        }
    }
    backtrack(0, []);
    return result;
}

function subsetsWithSum(nums: number[], target: number): number[][] {
    const result: number[][] = [];
    nums.sort((a, b) => a - b);               // sort enables a clean prune
    function backtrack(index: number, current: number[], sum: number): void {
        if (sum === target) { result.push([...current]); return; }
        for (let i = index; i < nums.length; i++) {
            if (sum + nums[i] > target) break;  // sorted: rest only bigger
            current.push(nums[i]);
            backtrack(i + 1, current, sum + nums[i]);
            current.pop();
        }
    }
    backtrack(0, [], 0);
    return result;
}
```

## Complexity

Backtracking is **exponential** in general, because the number of candidates is exponential. For the power
set there are two choices (in or out) for each of the n elements:

$$
\text{subsets of } n \text{ items} = 2^{n}, \qquad T(n) = O(n \cdot 2^{n}).
$$

The extra factor of n is the cost of copying each finished path. Recursion depth — and therefore the working
space beyond the output — is just the height of the tree:

$$
S(n) = O(n).
$$

| Problem | Candidates | Time | Space (stack) |
|---------|-----------|------|---------------|
| Subsets / power set | $2^{n}$ | $O(n \cdot 2^{n})$ | $O(n)$ |
| Permutations | $n!$ | $O(n \cdot n!)$ | $O(n)$ |
| Combinations $C(n,k)$ | $\binom{n}{k}$ | $O(k \cdot \binom{n}{k})$ | $O(k)$ |

Pruning does not change the worst case, but on real inputs it can chop the explored tree from astronomical
to instant — which is exactly why we always prune as early as possible.

## Common pitfalls & interview tips

- **Forgetting to un-choose.** If you `push` but never `pop`, state leaks into sibling branches and your
  results turn to garbage. Every `choose` needs a matching `un-choose` on the way back up.
- **Storing the path by reference.** `results.push(path)` saves a pointer to a still-mutating array; you end
  up with many copies of the *final* state. Always push a copy: `results.push([...path])`.
- **Pruning too late — or not at all.** Check validity *before* recursing. A constraint tested only at the
  leaf still explores the whole doomed subtree.
- **Interview cue:** the phrases "find **all**", "**every** arrangement", "all valid configurations" almost
  always mean backtracking. Say "choose, explore, un-choose" aloud and sketch the tree before coding.

## Practice

1. **Easy:** On paper, draw the choice tree for the subsets of `[a, b]`. How many leaves are there?
2. **Easy:** Modify the subset template so it collects *all* subsets (the power set) instead of only those
   hitting a target.
3. **Medium:** Generate every subset of `[1, 2, 3]` whose size is exactly 2 by adding one validity check.
4. **Medium:** Add a counter that increments once per recursive call, then compare the count with and without
   the overshoot prune on `[1, 2, 3, 4, 5]`, target `6`.
5. **Hard:** Combination Sum (LeetCode 39): the same target but each number may be reused — change a single
   index in the recursion to allow it.
