## The hook: how many ways can the band line up?

Three musicians walk on stage to take a bow. How many distinct left-to-right orders can they stand in? Pick
anyone for the leftmost spot (3 ways), anyone of the remaining two for the middle (2 ways), and the last one
is forced (1 way): `3 x 2 x 1 = 6`. That product is the **factorial**, and the six orders are the
**permutations** of the band.

Permutations are arrangements where **order matters** — `[guitar, bass, drums]` is a different photo than
`[drums, bass, guitar]`. Backtracking generates every one of them by deciding "who stands here?" position by
position.

<Callout type="intuition" title="The one-sentence idea">
To list all permutations, fill positions left to right: at each position try every element not yet used,
recurse, then mark it unused again — the un-choose step that frees it for the next branch.
</Callout>

## Intuition: a tree where each level fixes one position

Picture the choice tree for `[1, 2, 3]`. The root is empty. The first level chooses who goes in position 1
(three branches). The second level chooses position 2 from the two that remain, and the last position is
forced. Every leaf is a complete ordering, and there are exactly `3! = 6` of them.

The bookkeeping is a `used[]` flag array. "Choose element i" means set `used[i] = true` and append it;
"un-choose" means pop it and set `used[i] = false`. Because every leaf sits at depth n, no branch is ever
pruned for being incomplete — permutations explore the whole tree. The only pruning appears with
**duplicates** (next section).

This is the same machinery as the ML combinatorics track's [counting of
permutations](/lessons/combinatorics/permutations) — there you learned *how many* ($n!$); here you learn how
to *enumerate* them.

## How it works

The cleanest template tracks which elements are still available:

```ts
function permute(nums) {
  const result = [];
  const path = [];
  const used = new Array(nums.length).fill(false);
  function backtrack() {
    if (path.length === nums.length) { result.push([...path]); return; }
    for (let i = 0; i < nums.length; i++) {
      if (used[i]) continue;          // already placed → skip
      used[i] = true; path.push(nums[i]);   // choose
      backtrack();                          // explore
      path.pop(); used[i] = false;          // un-choose
    }
  }
  backtrack();
  return result;
}
```

**Dry run** of the first branch on `[1, 2, 3]`:

```
[]            → place 1 → [1]
[1]           → place 2 → [1,2]
[1,2]         → place 3 → [1,2,3]  ✓
back up, un-place 3 and 2
[1]           → place 3 → [1,3]
[1,3]         → place 2 → [1,3,2]  ✓
```

## The real implementation

The course file offers three equivalent engines; the **swap** version is the most memory-frugal because it
permutes the array in place instead of carrying a `used[]` array:

```ts
// /dsa/15-backtracking/permutations.ts
function permuteSwap<T>(nums: T[]): T[][] {
    const result: T[][] = [];
    const arr = [...nums];
    function backtrack(start: number): void {
        if (start === arr.length) { result.push([...arr]); return; }
        for (let i = start; i < arr.length; i++) {
            [arr[start], arr[i]] = [arr[i], arr[start]];  // choose: fix arr[i] here
            backtrack(start + 1);                          // explore
            [arr[start], arr[i]] = [arr[i], arr[start]];  // un-choose: swap back
        }
    }
    backtrack(0);
    return result;
}
```

Run the full file: `npx tsx 15-backtracking/permutations.ts`.

## Complexity

There are $n!$ permutations, and copying each finished arrangement costs $O(n)$:

$$
T(n) = O(n \cdot n!), \qquad S(n) = O(n).
$$

Factorial growth is brutal — it outpaces every exponential. A few values make the danger concrete:

| $n$ | $n!$ |
|-----|------|
| 5 | 120 |
| 8 | 40,320 |
| 10 | 3,628,800 |
| 13 | over 6 billion |

| Method | Idea | Extra space |
|--------|------|-------------|
| `used[]` flags | mark elements taken | $O(n)$ |
| Swap in place | fix one slot per level | $O(1)$ beyond output |
| Heap's algorithm | minimal-swap iteration | $O(1)$ |

The lesson: enumerate full permutations only for small n. Past a dozen items you almost always want a
*formula*, a *sample*, or a smarter pruned search rather than the whole list.

## Common pitfalls & interview tips

- **Reusing an element.** Without the `used[i]` guard (or the swap discipline) you generate sequences with
  repeats, not permutations. The check that an element is still free is non-negotiable.
- **Wrong duplicate prune.** For inputs with repeats you must **sort first**, then skip `nums[i]` when it
  equals `nums[i-1]` and the previous copy is unused. Skipping without sorting misses or double-counts.
- **Confusing permutations with combinations.** If `[1, 2]` and `[2, 1]` should count as the *same* answer,
  you want combinations, not permutations — different problem, smaller tree.
- **Interview cue:** "all orderings", "all arrangements", "anagrams of a string" signal permutations. State
  the $n!$ cost up front so the interviewer knows you understand the blow-up.

## Practice

1. **Easy:** By hand, list all permutations of `[a, b, c]`. Confirm there are 6.
2. **Easy:** How many distinct permutations does `[1, 1, 2]` have? Explain why it is not 6.
3. **Medium:** Generate all permutations of the string `"abc"` by reusing the swap template on a char array.
4. **Medium:** Implement k-permutations: arrangements of length k from n items by stopping recursion when the
   path reaches length k.
5. **Hard:** Next Permutation (LeetCode 31): rearrange a list into the next lexicographically greater ordering
   in $O(n)$ without generating them all — find the pivot, swap, reverse the suffix.
