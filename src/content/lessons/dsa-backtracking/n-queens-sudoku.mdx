## The hook: a puzzle that fights back

Combinations and permutations enumerate freely — every path to a leaf is a valid answer. Constraint
satisfaction is different: most branches are **illegal**, and the whole game is noticing that as early as
possible. Place a queen, and it forbids a column and two diagonals for every row below it. Write a digit into
a Sudoku cell, and it vanishes from a row, a column, and a $3\times3$ box.

These are **constraint satisfaction problems** (CSPs): assign values to variables so that a set of rules
holds. Backtracking is the natural weapon — one variable at a time, check the rules, and the moment a rule
breaks, throw the entire sub-tree away without exploring it. On an empty $9\times9$ Sudoku the naive space is
$9^{81}$; aggressive pruning is the only reason a solver returns before the heat death of the universe.

<Callout type="intuition" title="The one-sentence idea">
Assign one variable at a time and <strong>test every constraint before you recurse</strong>; an illegal
partial assignment can never become legal, so prune it now and never look inside that branch.
</Callout>

## Intuition: pruning is the whole algorithm

N-Queens places one queen **per row**, so the only decision at row `r` is *which column*. A column is legal
only if no earlier queen shares its column or either diagonal. The elegant trick is to track those three
threats in sets keyed by simple arithmetic:

- **Column:** the column index `col` itself.
- **↘ diagonal:** every cell on it has the same `row - col`.
- **↗ diagonal:** every cell on it has the same `row + col`.

So "is this square attacked?" is three $O(1)$ set lookups, not a scan of the board. When all N rows are
filled, every constraint held the entire way down — that is a solution.

Sudoku is the same skeleton with richer bookkeeping: walk cells in order, and for each empty one try digits
1–9, keeping a set per row, per column, and per box so validity is again $O(1)$. Both problems live or die on
the same move — **check, then descend; on failure, undo and try the next value.**

## How it works

One queen per row means the recursion depth is fixed at N; the branching is "which column", filtered by three
constraint sets:

```ts
function solveNQueens(n) {
  const result = [];
  const queens = [];                     // queens[row] = col
  const cols = new Set(), d1 = new Set(), d2 = new Set();
  function backtrack(row) {
    if (row === n) { result.push(queens.slice()); return; }
    for (let col = 0; col < n; col++) {
      if (cols.has(col) || d1.has(row - col) || d2.has(row + col)) continue; // prune
      queens[row] = col;                 // choose
      cols.add(col); d1.add(row - col); d2.add(row + col);
      backtrack(row + 1);                // explore
      queens.length = row;               // un-choose
      cols.delete(col); d1.delete(row - col); d2.delete(row + col);
    }
  }
  backtrack(0);
  return result;
}
```

**Dry run** of `n = 4` (columns 0–3), showing how row 0's choice cascades:

```
row 0: place col 1 → bans col 1, d1=-1, d2=1
row 1: col 0,1,2 attacked → only col 3 survives → place col 3
row 2: every column attacked → dead end → backtrack to row 1
...eventually: [1,3,0,2] and its mirror [2,0,3,1] are the 2 solutions
```

## The real implementation

The course file writes the board as strings and tracks the same three constraint sets — note the symmetric
`add` before recursing and `delete` after, the signature of correct backtracking:

```ts
// /dsa/15-backtracking/constraint-satisfaction.ts
function solveNQueens(n: number): string[][] {
    const result: string[][] = [];
    const board: string[] = Array(n).fill('.'.repeat(n));

    const cols = new Set<number>();
    const diag1 = new Set<number>();  // row - col (↘ diagonals)
    const diag2 = new Set<number>();  // row + col (↗ diagonals)

    function backtrack(row: number): void {
        if (row === n) { result.push([...board]); return; }

        for (let col = 0; col < n; col++) {
            if (cols.has(col) || diag1.has(row - col) || diag2.has(row + col)) {
                continue;                                 // pruned: under attack
            }
            board[row] = board[row].substring(0, col) + 'Q' + board[row].substring(col + 1);
            cols.add(col); diag1.add(row - col); diag2.add(row + col);

            backtrack(row + 1);

            board[row] = board[row].substring(0, col) + '.' + board[row].substring(col + 1);
            cols.delete(col); diag1.delete(row - col); diag2.delete(row + col);
        }
    }

    backtrack(0);
    return result;
}
```

The Sudoku solver in the same file is the identical pattern with one twist: it returns a **boolean** and
stops at the first full board, because a well-formed puzzle has a unique solution and there is no reason to
keep searching:

```ts
// /dsa/15-backtracking/constraint-satisfaction.ts
function backtrack(row: number, col: number): boolean {
    if (col === 9) { col = 0; row++; }
    if (row === 9) return true;                     // solved
    if (board[row][col] !== '.') return backtrack(row, col + 1);  // skip givens

    const boxIndex = Math.floor(row / 3) * 3 + Math.floor(col / 3);
    for (let num = 1; num <= 9; num++) {
        const char = String(num);
        if (isValid(row, col, char)) {              // O(1) row/col/box check
            board[row][col] = char;
            rows[row].add(char); cols[col].add(char); boxes[boxIndex].add(char);
            if (backtrack(row, col + 1)) return true;   // short-circuit on success
            board[row][col] = '.';
            rows[row].delete(char); cols[col].delete(char); boxes[boxIndex].delete(char);
        }
    }
    return false;                                   // trigger backtrack
}
```

## Complexity

Both problems are worst-case exponential; pruning slashes the *constant* and the realised tree, not the Big-O
ceiling. For N-Queens the branching collapses from $N^{N}$ to roughly $N!$ because each row must pick a fresh
column, and diagonals prune far more:

$$
T_{\text{queens}}(n) = O(n!), \qquad T_{\text{sudoku}} = O(9^{\,m}) \text{ for } m \text{ empty cells}.
$$

| Problem | Variables | Branch factor | Time (worst) | Space |
|---------|-----------|---------------|--------------|-------|
| N-Queens (all solutions) | $n$ rows | $\le n$ columns | $O(n!)$ | $O(n)$ |
| Sudoku | $m$ empty cells | $\le 9$ digits | $O(9^{m})$ | $O(1)$ board |
| Graph $m$-coloring | $V$ vertices | $m$ colors | $O(m^{V})$ | $O(V)$ |

The space is tiny — only the current partial assignment plus the constraint sets — which is exactly why
backtracking beats storing the whole search frontier.

## Common pitfalls & interview tips

- **Forgetting to undo state.** Every `add` needs its matching `delete` on the way back up. Leave one behind
  and later branches inherit a phantom constraint — the classic backtracking bug.
- **Checking constraints too late.** Test *before* you recurse, not after placing and descending. Pruning at
  the point of choice is the entire performance story.
- **Scanning the board instead of using sets.** Recomputing "is this attacked?" by looping the board turns
  $O(1)$ checks into $O(n)$ and can be the difference between milliseconds and timeouts.
- **Sudoku: not returning early.** A puzzle has one answer, so short-circuit with a boolean the instant the
  board is full; enumerating "all" solutions wastes the rest of the tree.
- **Interview cue:** "place items under rules", "no two may…", "fill the grid so that…", "color/schedule
  without conflicts" all mean CSP — reach for one-variable-at-a-time backtracking with early pruning.
