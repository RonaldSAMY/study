## The hook: let the big ones float up

Drop a fizzy tablet in water and the largest bubbles race to the surface first. Bubble sort
borrows that picture exactly: walk along the array comparing each value with its right-hand
neighbour, and whenever they are out of order, swap them. The biggest value keeps winning its
comparisons and "floats" all the way to the right end in a single pass. Do it again on what is
left, and the second-biggest settles just before it. It is the first sort almost everyone meets,
because you can run it in your head — and the last one you should ever ship, because it is slow.

<Callout type="intuition" title="The one-sentence idea">
Sweep the array swapping any adjacent pair that is out of order; after each sweep the largest
remaining value has bubbled into its final place, so one more element is locked in every pass.
</Callout>

## Intuition

Sorting means putting elements in order, and the most local thing you can check is: *are these two
neighbours in the right order?* Bubble sort does nothing but that, over and over.

In one left-to-right pass, every time the value you are carrying is bigger than the next one, they
swap, so you effectively *carry the running maximum rightward* until it can go no further — it ends
up at the end. That guarantees the last position is correct after pass one, the last two after pass
two, and so on. The unsorted region shrinks by one each pass, so after at most $n-1$ passes the
whole array is sorted.

There is one free optimisation: if a whole pass makes **no swaps at all**, nothing was out of
order, so you can stop early. That single flag is what gives bubble sort its one redeeming quality —
it is fast on data that is already (nearly) sorted.

## How it works

Trace `[5, 3, 8, 1, 2]`. Each line is one neighbour comparison:

```
Pass 1:  [5 3]8 1 2  → swap → [3 5]8 1 2
         3[5 8]1 2   → ok
         3 5[8 1]2   → swap → 3 5[1 8]2
         3 5 1[8 2]  → swap → 3 5 1[2 8]   8 is parked ✓
Pass 2:  [3 5]1 2 |8 → ok
         3[5 1]2 |8  → swap → 3[1 5]2 |8
         3 1[5 2]|8  → swap → 3 1[2 5]|8   5 parked ✓
Pass 3:  [3 1]2 |5 8 → swap → [1 3]2 |5 8
         1[3 2]|5 8  → swap → 1[2 3]|5 8   3 parked ✓
Pass 4:  [1 2]|3 5 8 → ok, no swaps → STOP
```

The runnable version uses the early-exit flag and actually returns the sorted array:

  const a = [...arr];
  const n = a.length;
  for (let i = 0; i < n - 1; i++) {
    let swapped = false;
    for (let j = 0; j < n - i - 1; j++) {

        [a[j], a[j + 1]] = [a[j + 1], a[j]];
        swapped = true;
      }
    }
    if (!swapped) break; // a clean pass means we are done
  }
  return a;
}`}

## The real implementation

From the course file — the optimised in-place version with the `swapped` flag:

```ts
// /dsa/sort/bubblesort.ts
function bubbleSortOptimized<T>(arr: T[]): T[] {
    const n = arr.length;
    const result = [...arr];

    for (let i = 0; i < n - 1; i++) {
        let swapped = false;
        for (let j = 0; j < n - i - 1; j++) {
            if (result[j] > result[j + 1]) {
                [result[j], result[j + 1]] = [result[j + 1], result[j]];
                swapped = true;
            }
        }
        if (!swapped) break; // already sorted → stop early
    }
    return result;
}
```

Run the full file: `npx tsx sort/bubblesort.ts`.

## Complexity

Best case is a single clean pass; average and worst are the full nested loop:

$$
T_\text{best}(n) = O(n), \qquad T_\text{avg}(n) = T_\text{worst}(n) = O(n^2), \qquad S(n) = O(1).
$$

| Case | Comparisons | Time |
|------|-------------|------|
| Already sorted (best) | $n-1$ | $O(n)$ |
| Random (average) | $\approx n^2/2$ | $O(n^2)$ |
| Reversed (worst) | $n(n-1)/2$ | $O(n^2)$ |

It sorts **in place** ($O(1)$ extra memory) and is **stable** — equal values never jump past each
other because we only swap on a strict `>`.

## Common pitfalls & interview tips

- **Forgetting the early-exit flag.** Without `swapped`, bubble sort is $O(n^2)$ even on sorted
  input. With it, the best case drops to $O(n)$ — interviewers love asking for exactly this tweak.
- **The inner bound is `n - i - 1`, not `n - 1`.** The last $i$ elements are already sorted; re-scanning
  them wastes time (and can read past the end).
- **Swap, do not just compare.** A common bug is detecting the inversion but forgetting to actually
  exchange the two values.
- **Interview framing:** "larger elements bubble up." Then mention it is *stable* and *in-place* but
  $O(n^2)$, so it is teaching material, not production code.

## Practice

1. **Easy:** Hand-trace one full pass of bubble sort on `[4, 2, 7, 1]`. Which value ends up last?
2. **Easy:** *Sort by parity* — rearrange `[3, 1, 2, 4]` so evens come before odds using a single
   adjacent-swap sweep. How is this just a bubble pass with a different comparison?
3. **Medium:** Modify the code to also **count** the number of swaps. How many does `[5, 4, 3, 2, 1]`
   need, and why is that the maximum possible?
4. **Medium:** *Cocktail shaker sort* — bubble left-to-right, then right-to-left, alternating. Why
   does this help arrays where one small value sits at the far right (a "turtle")?
5. **Hard:** Prove that the number of swaps bubble sort performs equals the number of **inversions**
   in the array (pairs that are out of order).
