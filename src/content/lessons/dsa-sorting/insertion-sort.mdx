## The hook: the way you sort a hand of cards

Deal yourself a hand of playing cards and you sort them without thinking: you pick up each new card
and slide it left until it sits between a smaller and a larger card. The cards already in your hand
stay sorted the whole time; you just make room for the newcomer. That is insertion sort, exactly —
and unlike the previous two sorts, it gets *dramatically faster* when the data is already close to
sorted. That single property is why it hides inside the sorting routines your language actually ships.

<Callout type="intuition" title="The one-sentence idea">
Keep a sorted region on the left; take the next value, shift the larger sorted elements one slot
right to open a gap, and drop the value into it.
</Callout>

## Intuition

Selection sort searched the *whole* unsorted tail for one element to place. Insertion sort flips
that: it takes the *next* element (no searching) and asks where it belongs in the already-sorted
left region. To insert, it slides every larger sorted value one position right — like nudging books
on a shelf — until a gap opens at the right spot.

The magic is in the stopping condition. As soon as the value to the left is **not** bigger, you
stop shifting — the element is home. On a nearly-sorted array almost nothing is out of place, so the
inner shift-loop barely runs, and the whole sort costs close to $O(n)$. On a reversed array every
element must travel all the way left, and you are back to $O(n^2)$. Insertion sort *adapts to how
sorted the input already is* — the first sort in this track that does.

## How it works

Trace `[5, 3, 8, 1, 2]`. Brackets mark the sorted region; the picked value is in caps:

```
[5] 3 8 1 2        pick 3 → 5>3 shift → [3 5] 8 1 2
[3 5] 8 1 2        pick 8 → 5<8 stop  → [3 5 8] 1 2
[3 5 8] 1 2        pick 1 → shift 8,5,3 → [1 3 5 8] 2
[1 3 5 8] 2        pick 2 → shift 8,5,3 → [1 2 3 5 8]
```

The runnable version shifts (not swaps) and returns the sorted array:

## The real implementation

From the course file — the in-place version, plus the *binary* variant that locates the gap with
binary search (fewer comparisons, same shifts):

```ts
// /dsa/sort/insertionsort.ts
function insertionSort<T>(arr: T[]): T[] {
    const result = [...arr];
    const n = result.length;
    for (let i = 1; i < n; i++) {
        const current = result[i];
        let j = i - 1;
        while (j >= 0 && result[j] > current) {
            result[j + 1] = result[j]; // shift right
            j--;
        }
        result[j + 1] = current;       // insert
    }
    return result;
}

// Binary insertion sort: search for the insertion point in O(log i).
function binaryInsertionSort<T>(arr: T[]): T[] {
    const result = [...arr];
    for (let i = 1; i < result.length; i++) {
        const current = result[i];
        let left = 0, right = i;
        while (left < right) {
            const mid = (left + right) >> 1;
            if (result[mid] <= current) left = mid + 1;
            else right = mid;
        }
        for (let j = i; j > left; j--) result[j] = result[j - 1];
        result[left] = current;
    }
    return result;
}
```

Run the full file: `npx tsx sort/insertionsort.ts`.

## Complexity

The shift-loop length depends on the data, which is exactly what makes it adaptive:

$$
T_\text{best}(n) = O(n), \qquad T_\text{avg}(n) = T_\text{worst}(n) = O(n^2), \qquad S(n) = O(1).
$$

| Input | Shifts | Time |
|-------|--------|------|
| Sorted (best) | 0 | $O(n)$ |
| Random (average) | $\approx n^2/4$ | $O(n^2)$ |
| Reversed (worst) | $n(n-1)/2$ | $O(n^2)$ |

It is **in place** ($O(1)$), **stable** (the `>` test never shifts an equal value past its twin), and
**online** — it can absorb new values one at a time as they stream in. Binary search cuts the
*comparisons* to $O(n \log n)$, but the *shifts* stay $O(n^2)$, so the overall bound is unchanged.

## Common pitfalls & interview tips

- **Shifting versus swapping.** Insertion sort *shifts* larger values right and drops the key once;
  it does not swap pairs like bubble sort. Shifting does fewer writes and is more cache-friendly.
- **Loop-guard order matters.** Write `j >= 0 && a[j] > key`: the bounds check must come first, or
  you read `a[-1]`.
- **It really is used in production.** V8, Java, and Python all fall back to insertion sort for small
  subarrays (often under ~16–64 elements) because its low overhead beats the bookkeeping of fancier sorts.
- **Interview hook:** "adaptive, stable, online, great on small/nearly-sorted data." That last clause
  is the bridge to **Timsort**, which chops the array into runs and insertion-sorts each one.
