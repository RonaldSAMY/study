## The hook: finding a word when you can't flip backwards cheaply

Imagine a dictionary printed on a long paper scroll. Flipping *forward* is easy, but reeling *back* is
slow and clumsy. Binary search would have you bouncing left and right across the whole scroll — lots of
expensive rewinds. Instead you advance in steady leaps, peeking at every few hundred words, until you
just pass your target; then you scan that short stretch carefully. That is **jump search**. Its cousin,
**exponential search**, leaps in *doubling* strides — perfect when you don't even know how long the
scroll is.

<Callout type="intuition" title="The one-sentence idea">
Skip ahead in big steps to find the small block that must contain the target, then search only that
block — fixed √n steps for jump search, or doubling steps for exponential search.
</Callout>

## Intuition

Both algorithms split the work into **locate the block, then search the block**. They differ in how the
strides grow.

**Jump search** uses a *fixed* step of $\sqrt{n}$. It checks `arr[step]`, `arr[2·step]`, `arr[3·step]`,
… until it lands on a value `>=` the target — meaning the target, if present, sits in the previous block.
Then it linear-scans that one block of length $\sqrt{n}$. Why $\sqrt{n}$? Because with step `s` you make
about `n/s` jumps plus `s` scan steps, and `n/s + s` is smallest when `s = √n`. Its appeal: it only ever
moves *forward* and backtracks at most one block, which is gentle on sequential media.

**Exponential search** uses a *doubling* step: it probes index 1, 2, 4, 8, 16, … until `arr[bound]`
overshoots the target, then runs an ordinary binary search inside `[bound/2, bound]`. If the target sits
at index `i`, it finds the range in about $\log_2 i$ probes and binary-searches a range of size `i`, for
$O(\log i)$ total. That makes it the tool for **unbounded** inputs of unknown length, and a winner when
the target is near the front.

## How it works

**Jump search** for `19` in a 16-element sorted array (step `= √16 = 4`):

```
jump: arr[4]=8   < 19 → skip block, hop to index 8
jump: arr[8]=23  ≥ 19 → overshot; target is in block [4..8]
scan: arr[4]=8  ≠ 19
scan: arr[5]=11 ≠ 19
scan: arr[6]=14 ≠ 19
scan: arr[7]=19 = 19 → FOUND at index 7
```

Two jumps plus a four-cell scan. **Exponential search** for the same target doubles `1 → 2 → 4 → 8`
until `arr[8]=23 ≥ 19`, then binary-searches the range `[4..8]`.

function jumpSearch(arr, target) {
  const n = arr.length;
  if (n === 0) return -1;
  const step = Math.floor(Math.sqrt(n));
  let prev = 0, curr = step;
  while (curr < n && arr[curr] < target) { prev = curr; curr += step; }
  for (let i = prev; i <= Math.min(curr, n - 1); i++) {
    if (arr[i] === target) return i;
  }
  return -1;
}

// Exponential search: double the bound, then binary-search the range.
function exponentialSearch(arr, target) {
  const n = arr.length;
  if (n === 0) return -1;
  if (arr[0] === target) return 0;
  let bound = 1;
  while (bound < n && arr[bound] < target) bound *= 2;
  let lo = Math.floor(bound / 2), hi = Math.min(bound, n - 1);
  while (lo <= hi) {
    const mid = lo + Math.floor((hi - lo) / 2);
    if (arr[mid] === target) return mid;
    else if (arr[mid] < target) lo = mid + 1;
    else hi = mid - 1;
  }
  return -1;
}`}

## The real implementation

From the course files — jump search with its √n step, and exponential search's doubling phase that hands
off to a bounded binary search:

```ts
// /dsa/03-search/jump-search.ts
function jumpSearch<T>(arr: T[], target: T): number {
    const n = arr.length;
    if (n === 0) return -1;
    const step = Math.floor(Math.sqrt(n));

    let prev = 0;
    let curr = step;
    while (curr < n && arr[curr] < target) {   // hop forward by √n
        prev = curr;
        curr += step;
    }
    for (let i = prev; i <= Math.min(curr, n - 1); i++) {  // scan one block
        if (arr[i] === target) return i;
    }
    return -1;
}

// /dsa/03-search/exponential-search.ts
function exponentialSearch<T>(arr: T[], target: T): number {
    const n = arr.length;
    if (n === 0) return -1;
    if (arr[0] === target) return 0;

    let bound = 1;
    while (bound < n && arr[bound] < target) {  // double until overshoot
        bound *= 2;
    }
    const left = Math.floor(bound / 2);
    const right = Math.min(bound, n - 1);       // clamp to the real end
    return binarySearch(arr, target, left, right);  // ordinary binary search
}
```

## Complexity

The block sizes are chosen to balance the two phases:

$$
T_{\text{jump}}(n) = O(\sqrt{n}), \qquad T_{\text{exp}}(n) = O(\log i),
$$

where `i` is the target's index. Both use $O(1)$ extra space.

| Algorithm | Time | Best for |
|-----------|------|----------|
| Binary search | $O(\log n)$ | Random access, known size |
| Jump search | $O(\sqrt{n})$ | Forward-only / costly backward jumps |
| Exponential search | $O(\log i)$ | Unbounded size, target near front |

Binary search is asymptotically fastest, but jump and exponential search win in the *access patterns*
they were designed for — sequential media and unbounded streams.

## Common pitfalls & interview tips

- **Wrong step size.** Any fixed step gives $O(n/s + s)$; only $s = \sqrt{n}$ minimizes it. A step of,
  say, 10 is $O(n)$ on large arrays. Compute `Math.floor(Math.sqrt(n))`.
- **Forgetting to clamp the bound.** In exponential search, `bound` can shoot past the array. The right
  end of the binary search must be `Math.min(bound, n - 1)` or you index out of range.
- **Off-by-one in the jump block.** After the jump loop, scan `[prev, min(curr, n-1)]` inclusive. Starting
  the scan at `curr` instead of `prev` skips the block that actually holds the target.
- **Still requires sorted data.** Both algorithms lean on monotonicity just like binary search; on
  unsorted input they're meaningless.
- **Interview cue:** "the array is unbounded / infinite" or "we can only read forward / backward seeks are
  expensive" is the tell. Reach for exponential search for unknown length, jump search for sequential cost.

## Practice

1. **Easy:** For `n = 100`, what step does jump search use, and how many jumps at most before the scan?
2. **Easy:** Trace exponential search for target `8` in a 16-element array — how many times does `bound`
   double before it overshoots?
3. **Medium:** Modify jump search to return the *first* occurrence when duplicates are present.
4. **Medium:** Implement `unboundedSearch(getElement, target)` where `getElement(i)` returns `undefined`
   past the end — double until you read `undefined` or overshoot, then binary-search.
5. **Hard:** Compare expected probes of jump vs exponential vs binary search when the target is always at
   index 3 of a million-element array. Which wins, and why?
