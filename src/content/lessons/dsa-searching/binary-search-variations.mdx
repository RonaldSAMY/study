## The hook: how many people scored exactly 80?

A sorted list of exam scores has lots of repeated values. "Is 80 present?" is the plain binary-search
question you already solved. But the useful questions are sharper: *where does the run of 80s begin?*
*Where does it end?* *How many 80s are there?* *If a new 80 arrives, where does it slot in?* Plain binary
search stops at *some* 80 and shrugs. The variations in this lesson stop at the exact **boundary** you
care about — and they all reuse the same halving skeleton.

<Callout type="intuition" title="The one-sentence idea">
Instead of halting at any match, keep halving past it: record a candidate and steer toward the side that
holds the boundary you want — the first match, the last match, or the spot where a condition flips.
</Callout>

## Intuition

Ordinary binary search asks a yes/no question — "is `arr[mid]` the target?" Boundary search asks a
*monotonic* question whose answer flips exactly once across the array, like "is `arr[mid] >= target`?"
On a sorted array that predicate is `false, false, …, false, true, true, …, true`, and the first `true`
is the boundary. Binary search is the perfect tool for finding where a monotone predicate changes.

Two design choices control which boundary you land on. **First**, on equality do you keep going left
(to find the *first* occurrence) or right (the *last*)? **Second**, do you treat the right end as
inclusive (`hi = n - 1`, loop `lo <= hi`) or exclusive (`hi = n`, loop `lo < hi`)? The exclusive form is
the cleanest for "bounds": `lowerBound` returns the first index with `arr[i] >= target`, and `upperBound`
the first with `arr[i] > target`. Their difference, `upperBound - lowerBound`, is the count of matches —
no separate counting pass.

## How it works

Find the **first** occurrence of `4` in `[2, 4, 4, 4, 6, 6, 8, 9]` (indices 0–7). The trick: on a match,
save the index but keep searching the left half for an earlier one.

```
lo=0 hi=7 mid=3 arr[3]=4 = 4 → record best=3, search LEFT, hi=2
lo=0 hi=2 mid=1 arr[1]=4 = 4 → record best=1, search LEFT, hi=0
lo=0 hi=0 mid=0 arr[0]=2 < 4 → search RIGHT, lo=1
lo=1 hi=0 → window empty → first occurrence is index 1
```

For the **last** occurrence you flip one line — on a match, search *right* (`lo = mid + 1`) instead of
left — and the same skeleton lands on index 3.

function lowerBound(arr, target) {
  let lo = 0, hi = arr.length;          // note: hi = n, not n - 1
  while (lo < hi) {
    const mid = lo + Math.floor((hi - lo) / 2);
    if (arr[mid] < target) lo = mid + 1;
    else hi = mid;                       // mid might be the boundary — keep it
  }
  return lo;
}

// Upper bound: first index with arr[i] > target. One comparison changes.
function upperBound(arr, target) {
  let lo = 0, hi = arr.length;
  while (lo < hi) {
    const mid = lo + Math.floor((hi - lo) / 2);
    if (arr[mid] <= target) lo = mid + 1;
    else hi = mid;
  }
  return lo;
}

function countOccurrences(arr, target) {
  return upperBound(arr, target) - lowerBound(arr, target);
}`}

## The real implementation

From the course file — the explicit first-occurrence search and the two bound helpers it builds on:

```ts
// /dsa/03-search/binary-search-variations.ts
function findFirstOccurrence<T>(arr: T[], target: T): number {
    let left = 0;
    let right = arr.length - 1;
    let result = -1;

    while (left <= right) {
        const mid = Math.floor((left + right) / 2);
        if (arr[mid] === target) {
            result = mid;       // record, then keep looking LEFT
            right = mid - 1;
        } else if (arr[mid] < target) {
            left = mid + 1;
        } else {
            right = mid - 1;
        }
    }
    return result;
}

function lowerBound<T>(arr: T[], target: T): number {
    let left = 0, right = arr.length;     // exclusive right
    while (left < right) {
        const mid = Math.floor((left + right) / 2);
        if (arr[mid] < target) left = mid + 1;
        else right = mid;
    }
    return left;                           // first index with arr[i] >= target
}
```

## Complexity

Every variation is still binary search — a constant amount of work per halving:

$$
T(n) = O(\log n), \qquad S(n) = O(1).
$$

| Question | Helper | Result when absent |
|----------|--------|--------------------|
| First index $\ge$ target | `lowerBound` | insertion point |
| First index $>$ target | `upperBound` | insertion point |
| First / last exact match | first/last occurrence | $-1$ |
| Count of matches | `upper − lower` | $0$ |

Counting occurrences the naive way is $O(n)$; two boundary searches make it $O(\log n)$ — a real win on
arrays with long runs of duplicates.

## Common pitfalls & interview tips

- **Mixing inclusive and exclusive `hi`.** The bound template uses `hi = n` with `while (lo < hi)` and
  `hi = mid` (no `-1`). Splicing in `hi = mid - 1` from the exact-match template breaks it. Pick one
  template per variation and keep it consistent.
- **The one-character difference.** `lowerBound` uses `arr[mid] < target`; `upperBound` uses `<=`. That
  single `=` is the entire distinction between "first `>=`" and "first `>`". Name them carefully.
- **Reading the not-found result.** When the target is absent, `lowerBound` returns the **insertion point**,
  not `-1`. That is a feature — it tells you where the value would go — but check `arr[idx] === target`
  if you need a true membership test.
- **Off-by-one in the count.** The count is `upper - lower`, not `upper - lower + 1`, because `upper` is
  exclusive. Verify on a single-element run: one `4` gives `lower=1, upper=2, count=1`.
- **Interview cue:** "first/last position of element", "smallest index such that…", or "how many are
  `<= x`" are all boundary searches. Reach for `lowerBound`/`upperBound`, not a raw binary search.
