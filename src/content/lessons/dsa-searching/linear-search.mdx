## The hook: finding a friend's name in an unsorted guest list

A party organizer hands you a stack of RSVP cards in the order they arrived — no alphabetical
sorting, no index. Someone asks, "did Mara reply?" There is exactly one honest way to answer:
start at the top and read card after card until you hit Mara or run out. That is **linear search** —
the most basic algorithm there is, and on jumbled data it is genuinely the best you can do.

<Callout type="intuition" title="The one-sentence idea">
Walk through the collection from one end, compare each element to the target, and stop the moment
you find it — or report failure once you have looked at everything.
</Callout>

## Intuition

Sorting lessons taught you that *order* unlocks clever shortcuts. Linear search is what you reach for
when there is **no order to exploit**. You make a single promise: every element gets at most one look.
Because you never skip or assume anything, linear search works on *any* sequence — unsorted arrays,
linked lists, streams you can only read once, even a generator that produces values on the fly.

The cost of that generality is that you have no way to rule out the elements you have not seen yet.
If the target is the last thing in the list (or absent entirely), you are forced to inspect all
`n` items. There is no free lunch: the algorithm is only as fast as "look at everything" in the worst
case. The upside is its honesty and simplicity — no preconditions, no setup, no surprises.

## How it works

Search for `6` in `[8, 3, 9, 1, 6, 4, 7]`:

```
index 0: 8 ≠ 6 → keep going
index 1: 3 ≠ 6 → keep going
index 2: 9 ≠ 6 → keep going
index 3: 1 ≠ 6 → keep going
index 4: 6 = 6 → FOUND at index 4
```

Five comparisons, then a confident answer. Had `6` been missing, we would have read all seven cards and
returned `-1`. The loop body is a single equality test, which is why linear search is the first algorithm
everyone writes.

## The real implementation

The course file is generic over any element type `T`, so it works for numbers, strings, or objects
with a custom comparator. Here is the faithful core:

```ts
// /dsa/03-search/linear-search.ts
function linearSearch<T>(arr: T[], target: T): number {
    for (let i = 0; i < arr.length; i++) {
        if (arr[i] === target) {
            return i;
        }
    }
    return -1;
}

// Find the FIRST element satisfying a predicate (not just equality).
function findIndex<T>(arr: T[], predicate: (v: T, i: number, a: T[]) => boolean): number {
    for (let i = 0; i < arr.length; i++) {
        if (predicate(arr[i], i, arr)) {
            return i;
        }
    }
    return -1;
}
```

Run the full file: `npx tsx 03-search/linear-search.ts`.

## Complexity

One pass, one comparison per element, no extra storage:

$$
T(n) = O(n), \qquad S(n) = O(1).
$$

| Case | Comparisons | When |
|------|-------------|------|
| Best | $1$ | Target is the first element |
| Average | $\tfrac{n}{2}$ | Target equally likely anywhere |
| Worst | $n$ | Target is last, or absent |

Linear search is unbeatable for a **single** lookup on unsorted data: any algorithm must, in the worst
case, examine every element at least once just to be sure the target is not hiding somewhere unseen.

## Common pitfalls & interview tips

- **Returning a boolean when the index is wanted.** "Is it present?" and "where is it?" are different
  questions. Return the index (and `-1` for absent) so the caller can choose.
- **Forgetting the not-found case.** A loop that only returns inside the `if` falls off the end with
  `undefined`. Always finish with an explicit `return -1`.
- **Reaching for linear search on sorted data.** If the array is already sorted and you will search it
  often, you are leaving an O(log n) speedup on the table — that is the next lesson.
- **Sentinel trick.** Placing the target in the last slot lets the loop drop its bounds check, shaving a
  comparison per iteration. Cute for interviews, rarely worth the mutation in practice.
- **Interview cue:** "the data is unsorted and we search once" is the signal to *not* over-engineer —
  say "linear scan, O(n), can't do better here" and move on.

## Practice

1. **Easy:** Trace `linearSearch([5, 2, 9, 2, 7], 2)` by hand. Which index is returned, and why that one?
2. **Easy:** Modify the loop to return the *last* matching index instead of the first.
3. **Medium:** Write `count(arr, target)` that returns how many times the target appears in one pass.
4. **Medium:** Find both the minimum and the maximum of an array in a single scan (track two variables).
5. **Hard:** Given a 2D grid, return the `{row, col}` of a target, scanning row by row — what is the
   worst-case number of comparisons for an `m × n` grid?
