## The hook: how a calculator really thinks

Type `2 + 3 * 4` into a calculator and it answers `14`, not `20` — it respected precedence and did the
multiply first. But computers do not actually evaluate left to right with precedence rules baked in.
Internally they rewrite the expression into a form with **no parentheses and no precedence at all**,
called postfix, and then a stack chews through it in one clean left-to-right pass. The same machinery
that matched brackets last lesson now does arithmetic.

<Callout type="intuition" title="The one-sentence idea">
In **postfix** notation an operator comes after its operands, so evaluation is just: push numbers, and
when you see an operator pop the top two, combine them, and push the result back.

</Callout>

## Intuition

Infix — the notation humans write, like `3 + 4 * 2` — is awkward for a machine because the meaning of
`+` depends on what comes *later* (`*` binds tighter, so it happens first). **Postfix** (also called
Reverse Polish Notation) removes that ambiguity by writing the operator *after* its operands:
`3 4 2 * +`. Read it aloud: "take 4 and 2, multiply, then add 3." No parentheses, no precedence table —
the order of the tokens already encodes everything.

That makes evaluation almost trivial with a stack:

- A **number** has nothing to combine yet → push it.
- An **operator** needs its two operands, and they are the two most recent numbers → pop two, apply the
  operator, push the single result.

When the tokens run out, exactly one value is left: the answer. The stack quietly holds all the
"pending operands" — the partial results waiting for an operator, which is precisely the most-recent
bookkeeping a stack is built for.

## How it works

Evaluate `5 1 2 + 4 * + 3 -`. The stack top is on the right:

```
5    push           [5]
1    push           [5, 1]
2    push           [5, 1, 2]
+    1 + 2 = 3      [5, 3]
4    push           [5, 3, 4]
*    3 * 4 = 12     [5, 12]
+    5 + 12 = 17    [17]
3    push           [17, 3]
-    17 - 3 = 14    [14]
result = 14
```

Here is the evaluator, plus the **shunting-yard** algorithm that converts ordinary infix into postfix —
notice it is *also* a stack, this time holding operators until something of lower-or-equal precedence
forces them out:

<CodeRunner
  client:visible
  code={`function evaluatePostfix(expr) {
  const stack = [];
  for (const token of expr.split(' ')) {
    if (/^-?\\d+$/.test(token)) {
      stack.push(Number(token));          // operand → push
    } else {
      const b = stack.pop();              // right operand
      const a = stack.pop();              // left operand
      if (token === '+') stack.push(a + b);
      else if (token === '-') stack.push(a - b);
      else if (token === '*') stack.push(a * b);
      else stack.push(Math.trunc(a / b));
    }
  }
  return stack.pop();
}

function infixToPostfix(expr) {
  const out = [], ops = [];
  const prec = { '+': 1, '-': 1, '*': 2, '/': 2 };
  for (const t of expr.split(' ')) {
    if (/^-?\\d+$/.test(t)) out.push(t);
    else if (t === '(') ops.push(t);
    else if (t === ')') { while (ops.at(-1) !== '(') out.push(ops.pop()); ops.pop(); }
    else { while (ops.length && ops.at(-1) !== '(' && prec[ops.at(-1)] >= prec[t]) out.push(ops.pop()); ops.push(t); }
  }
  while (ops.length) out.push(ops.pop());
  return out.join(' ');
}`}

## A second pattern: the monotonic stack

Stacks evaluate expressions, but the same "pop while the top is dominated" rhythm solves a whole family
of array problems. The classic is **next greater element**: for each number, what is the first larger
number to its right? A *monotonic* stack keeps indices whose values strictly decrease; when a bigger
value arrives, it pops everything it beats and records itself as their answer — each index pushed and
popped once, so O(n) overall:

<CodeRunner
  client:visible
  code={`function nextGreaterElement(arr) {
  const result = new Array(arr.length).fill(-1);
  const stack = [];                       // indices, values decreasing
  for (let i = 0; i < arr.length; i++) {
    while (stack.length && arr[i] > arr[stack.at(-1)]) {
      result[stack.pop()] = arr[i];       // arr[i] is the next greater
    }
    stack.push(i);
  }
  return result;                          // -1 where none exists
}`}

## The real implementation

From the course file — the postfix evaluator and the monotonic-stack scanner exactly as shipped:

```ts
// /dsa/06-stack/stack-applications.ts
function evaluatePostfix(expression: string): number {
    const stack: number[] = [];
    for (const token of expression.split(' ')) {
        if (/^\d+$/.test(token)) {
            stack.push(parseInt(token));
        } else {
            const b = stack.pop()!;          // right operand (popped first)
            const a = stack.pop()!;          // left operand
            switch (token) {
                case '+': stack.push(a + b); break;
                case '-': stack.push(a - b); break;
                case '*': stack.push(a * b); break;
                case '/': stack.push(Math.trunc(a / b)); break;
            }
        }
    }
    return stack.pop()!;
}

// Monotonic decreasing stack of indices → next greater element, O(n).
function nextGreaterElement(arr: number[]): number[] {
    const n = arr.length;
    const result = new Array(n).fill(-1);
    const stack: number[] = [];
    for (let i = 0; i < n; i++) {
        while (stack.length > 0 && arr[i] > arr[stack[stack.length - 1]]) {
            result[stack.pop()!] = arr[i];
        }
        stack.push(i);
    }
    return result;
}
```

## Complexity

Every token (or array index) is pushed once and popped at most once, so all three algorithms are linear:

$$
T(n) = O(n), \qquad S(n) = O(n).
$$

| Algorithm | Time | Space | Stack holds |
|-----------|------|-------|-------------|
| Evaluate postfix | $O(n)$ | $O(n)$ | pending operands |
| Infix → postfix (shunting-yard) | $O(n)$ | $O(n)$ | pending operators |
| Next greater element | $O(n)$ | $O(n)$ | indices, decreasing |

The monotonic-stack O(n) can surprise people: the inner `while` looks nested, but across the whole run
each index enters and leaves the stack just once, so the total pop count is at most $n$.

## Common pitfalls & interview tips

- **Operand order.** For `-` and `/`, the operand popped **first** is the right-hand side. `a = pop()`
  second, `b = pop()` first, then compute `a - b`. Swapping them flips the sign or inverts the division.
- **Integer division.** Use `Math.trunc(a / b)`, not plain `/`, when the problem wants integer results —
  and watch the sign of truncation toward zero.
- **Left- vs right-associativity.** In shunting-yard, left-associative operators pop on `>=` precedence;
  right-associative ones (like `^`) pop only on strictly `>`. Mixing these up misplaces operators.
- **Monotonic direction.** "Next greater" wants a *decreasing* stack scanned left to right; "next
  smaller" or "previous greater" flips the comparison or the scan direction. State the invariant out
  loud before coding.
