## The hook: index a string the way a book has an index

Every algorithm so far preprocesses the *pattern* and then reads the text. A suffix array flips the investment: preprocess the **text** once, and then answer question after question about it cheaply. The idea is disarmingly simple — list every suffix of the string, sort them lexicographically, and keep just their starting indices. That sorted list is the suffix array. Because occurrences of any pattern are precisely the suffixes that *start with* that pattern, and sorting places them together, every search becomes a **binary search** over the sorted suffixes. One structure then answers "does this occur?", "how many times?", "what is the longest repeated substring?", and "how many distinct substrings are there?".

<Callout type="intuition" title="The one-sentence idea">
Sort all n suffixes once; then every substring you could ever search for occupies a contiguous block of that sorted list, reachable by binary search.
</Callout>

## Intuition

A string of length $n$ has $n$ suffixes. Writing them out and sorting them feels wasteful — until you realize what sorting buys. In sorted order, all suffixes sharing a prefix are neighbors. So the occurrences of a pattern $P$ are exactly the suffixes whose first $|P|$ characters equal $P$, and those form one contiguous run. Finding that run is a binary search: at each step compare $P$ against the middle suffix's prefix and go left or right.

Its constant companion is the **LCP array** (Longest Common Prefix): $\mathrm{LCP}[i]$ is how many leading characters suffix $i$ shares with the previous suffix in sorted order. Adjacent sorted suffixes are the most similar, so the LCP array captures all the repetition in the string. A single scan of it yields the longest repeated substring (the max LCP) and the count of distinct substrings ($\tfrac{n(n+1)}{2} - \sum \mathrm{LCP}$). Kasai's algorithm computes the whole LCP array in $O(n)$.

Naively sorting suffixes is $O(n^2 \log n)$ because each comparison scans up to $n$ characters. **Prefix doubling** fixes this: sort by first character, then by first 2, then 4, 8, …, reusing the previous ranks so each round is a single $O(n)$ or $O(n \log n)$ sort — giving $O(n \log n)$ overall.

## How it works

For `"banana"` the sorted suffixes and their start indices are:

```
SA  suffix
 5  a
 3  ana
 1  anana
 0  banana
 4  na
 2  nana
→ Suffix Array = [5, 3, 1, 0, 4, 2]
   LCP Array    = [0, 1, 3, 0, 0, 2]   (max LCP = 3 → "ana" is the longest repeat)
```

Searching `"ana"` binary-searches to the block `{3, 1}`, i.e. occurrences at text positions 1 and 3. Here it is building the array and searching:

## The real implementation

The course file builds the array with **prefix doubling** — sorting by rank pairs and refreshing ranks each round — for $O(n \log^2 n)$, then Kasai's algorithm for the LCP array:

```ts
// /dsa/18-string-algorithms/suffix-array.ts
export function buildSuffixArray(s: string): number[] {
  const n = s.length;
  if (n === 0) return [];
  let rank: number[] = new Array(n);
  let tempRank: number[] = new Array(n);
  const sa: number[] = new Array(n);
  for (let i = 0; i < n; i++) { sa[i] = i; rank[i] = s.charCodeAt(i); }

  for (let k = 1; k < n; k *= 2) {                 // double the compared length
    const compare = (a: number, b: number): number => {
      if (rank[a] !== rank[b]) return rank[a] - rank[b];
      const ra = a + k < n ? rank[a + k] : -1;
      const rb = b + k < n ? rank[b + k] : -1;
      return ra - rb;                              // break ties by the next block
    };
    sa.sort(compare);
    tempRank[sa[0]] = 0;
    for (let i = 1; i < n; i++)
      tempRank[sa[i]] = tempRank[sa[i - 1]] + (compare(sa[i - 1], sa[i]) < 0 ? 1 : 0);
    [rank, tempRank] = [tempRank, rank];
    if (rank[sa[n - 1]] === n - 1) break;          // all ranks unique → done
  }
  return sa;
}
```

## Complexity

Prefix doubling does $O(\log n)$ rounds of sorting; with a comparison sort that is $O(n \log^2 n)$, with counting sort $O(n \log n)$, and specialized algorithms (DC3/SA-IS) reach $O(n)$. Kasai builds the LCP array in linear time, and a pattern query is a binary search where each comparison scans up to $m$ characters:

$$
T_{\text{build}} = O(n \log n), \qquad T_{\text{LCP}} = O(n), \qquad T_{\text{query}} = O(m \log n), \qquad S = O(n).
$$

| Task | Time | How |
|------|------|-----|
| Build suffix array | $O(n \log n)$ | prefix doubling ($O(n)$ with SA-IS) |
| Build LCP array | $O(n)$ | Kasai's algorithm |
| Search a pattern | $O(m \log n)$ | binary search over suffixes |
| Longest repeated substring | $O(n)$ | maximum of the LCP array |
| Count distinct substrings | $O(n)$ | $\tfrac{n(n+1)}{2} - \sum \mathrm{LCP}$ |

<Callout type="why">
The suffix array is the practical, memory-light cousin of the suffix tree. Sorted rotations of a string are also the core of the Burrows-Wheeler Transform — so the same idea that indexes a genome for search is what lets bzip2 compress it.
</Callout>

## The same idea, different worlds

<ExampleCard domain="Genome alignment" icon="🧬">
Read aligners such as BWA index a reference genome with a suffix array (via the Burrows-Wheeler Transform) so that millions of short DNA reads can each be located with a binary search. Build the index once; query it billions of times.
</ExampleCard>

<ExampleCard domain="Data compression" icon="🗜️">
bzip2's Burrows-Wheeler stage sorts all rotations of a block — essentially a suffix sort — to cluster similar contexts together, making the stream far more compressible. The suffix array is the engine that produces that reordering efficiently.
</ExampleCard>

<ExampleCard domain="Search engines" icon="🔍">
Full-text indexes and "search-as-you-type" over a fixed corpus can lean on suffix arrays to find every occurrence of a query substring — and, with the LCP array, to surface the longest repeated phrases or common substrings across documents.
</ExampleCard>

## Common pitfalls & interview tips

- **Append a unique sentinel.** Adding a terminator smaller than every real character (often written `$`) makes no suffix a prefix of another and simplifies both construction and search — many bugs vanish once you add it.
- **Naive sort is $O(n^2 \log n)$.** Sorting suffixes as full strings is fine for teaching and tiny inputs but quadratic in practice; know that prefix doubling or SA-IS is what makes it usable at scale.
- **Suffix arrays are static.** Any edit to the string invalidates the array — rebuild, or reach for a different structure if the text changes often.
- **Interview cue:** "build once, run many substring queries", "longest repeated substring", or "number of distinct substrings" all point to suffix array + LCP. Mention the $\tfrac{n(n+1)}{2} - \sum \mathrm{LCP}$ identity to close the distinct-substring question in one line.
