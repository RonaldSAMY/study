## The hook: every palindrome, in one pass

"Find the longest palindromic substring" is a classic that most people solve by expanding around each center — try each of the $2n-1$ centers, grow outward while characters mirror, and keep the best. That is $O(n^2)$. Manacher's algorithm gets the same answer in $O(n)$ by noticing that palindromes overlap in a structured way: if you already know a big palindrome, positions inside it have a **mirror twin** whose radius you can reuse, saving almost all the re-comparison.

It is the palindrome cousin of the Z-algorithm's Z-box and Boyer-Moore's reuse: keep the rightmost thing you know, exploit its symmetry.

<Callout type="intuition" title="The one-sentence idea">
Keep the rightmost palindrome you have found; for a new center inside it, start its radius from the mirror center's radius (clamped to the boundary) and only expand past what you already know.
</Callout>

## Intuition

Two annoyances make palindromes fiddly: **odd vs even length** (`aba` has a center character; `abba` has a center *gap*), and **re-checking**. Manacher removes the first by a preprocessing trick — insert a separator `#` between every character and at the ends:

$$
\texttt{abba} \;\longrightarrow\; \texttt{\#a\#b\#b\#a\#}
$$

Now *every* palindrome has odd length, so a single uniform routine handles both cases. Define $P[i]$ as the palindrome **radius** centered at position $i$ of the transformed string.

For the second annoyance, keep the palindrome with the rightmost boundary $R$ and its center $C$. When examining a new center $i < R$, its mirror is $i' = 2C - i$. Since the big palindrome is symmetric, $P[i]$ starts out equal to $P[i']$ — unless that would poke past $R$, in which case we clamp to $R - i$ and expand carefully from there. Every expansion step that does real work pushes $R$ rightward, and $R$ never retreats, so the whole sweep is $O(n)$.

## How it works

On `s = "abacaba"`, transform to `#a#b#a#c#a#b#a#`. The center at the middle `c` grows to cover everything; positions to its right then mirror positions to its left:

```
transform: # a # b # a # c # a # b # a #
P (radius): 0 1 0 3 0 1 0 7 0 1 0 3 0 1 0
                               ^ center of "abacaba", radius 7
```

The longest palindrome is read off the maximum radius: max $P[i] = 7$ recovers `abacaba`. The original substring length equals $P[i]$, and it starts at $\lfloor (i - P[i]) / 2 \rfloor$ in the source. Here it is computing the longest palindrome directly:

<CodeRunner

  code={`function longestPalindrome(s) {
  if (s.length < 2) return s;
  const t = "#" + s.split("").join("#") + "#";
  const n = t.length;
  const p = new Array(n).fill(0);
  let C = 0, R = 0;
  for (let i = 0; i < n; i++) {
    if (i < R) p[i] = Math.min(R - i, p[2 * C - i]);   // reuse the mirror
    while (i - 1 - p[i] >= 0 && i + 1 + p[i] < n &&
           t[i + 1 + p[i]] === t[i - 1 - p[i]]) p[i]++; // expand
    if (i + p[i] > R) { C = i; R = i + p[i]; }          // move the boundary
  }
  let best = 0, center = 0;
  for (let i = 0; i < n; i++) if (p[i] > best) { best = p[i]; center = i; }
  const start = (center - best) / 2;
  return s.slice(start, start + best);
}`}

## The real implementation

The course file separates the transform and the radius sweep, using boundary markers `^` and `$` so the expansion loop needs no explicit bounds check:

```ts
// /dsa/18-string-algorithms/manacher.ts
export function buildPArray(s: string): number[] {   // s is the transformed string
  const n = s.length;
  const p: number[] = new Array(n).fill(0);
  let C = 0, R = 0;                                   // center & right edge

  for (let i = 1; i < n - 1; i++) {
    const mirror = 2 * C - i;
    if (i < R) p[i] = Math.min(R - i, p[mirror]);     // copy from the mirror
    while (s[i + 1 + p[i]] === s[i - 1 - p[i]]) p[i]++; // expand outward
    if (i + p[i] > R) { C = i; R = i + p[i]; }         // update rightmost
  }
  return p;
}
```

Run the full file: `npx tsx 18-string-algorithms/manacher.ts`.

## Complexity

The transform is linear, and although the inner `while` looks quadratic, the same monotone-boundary argument as the Z-algorithm applies: $R$ only advances, and each advance is one comparison, so all expansions together cost $O(n)$:

$$
T(n) = O(n), \qquad S(n) = O(n).
$$

| Approach | Time | Space |
|----------|------|-------|
| Brute force (check every substring) | $O(n^3)$ | $O(1)$ |
| Expand around center | $O(n^2)$ | $O(1)$ |
| Dynamic programming | $O(n^2)$ | $O(n^2)$ |
| **Manacher** | $O(n)$ | $O(n)$ |

<Callout type="why">
The radius array does more than find the longest palindrome: summing $\lfloor (P[i]+1)/2 \rfloor$ counts *all* palindromic substrings, and the array answers "is s[l..r] a palindrome?" in $O(1)$ — the kind of preprocessing that turns palindrome-heavy problems from painful to trivial.
</Callout>

## The same idea, different worlds

<ExampleCard domain="Molecular biology" icon="🧫">
Restriction enzymes cut DNA at reverse-complement palindromes such as <code>GAATTC</code> — read the same on the opposite strand. Scanning a genome for the longest or all palindromic sites uses exactly this radius sweep, so labs can predict where an enzyme will cleave.
</ExampleCard>

<ExampleCard domain="Linguistics & wordplay" icon="🔤">
Palindrome hunters and constrained-writing tools search text for the longest symmetric run — "a man a plan a canal panama" once spaces are stripped. Linear detection lets an interactive editor flag palindromic spans as you type.
</ExampleCard>

<ExampleCard domain="Data integrity" icon="🧮">
Some error-detecting and compression schemes look for symmetric or mirrored runs in a byte stream. Locating the largest such region quickly helps a codec decide where a reflective encoding pays off, without an $O(n^2)$ scan of every candidate center.
</ExampleCard>

## Common pitfalls & interview tips

- **Forgetting the transform.** Without the `#` separators you need two passes (odd and even centers) and the mirror bookkeeping gets messy. The separators make everything odd-length and uniform.
- **Clamping to the boundary.** Inside the current palindrome you must take `min(R - i, P[mirror])`. Copying `P[mirror]` unclamped can claim a radius that pokes past what you have verified.
- **Index conversion.** Mapping the transformed center/radius back to original indices trips people up: the original length equals $P[i]$ and starts at $\lfloor (i - P[i])/2 \rfloor$. Sentinels (`^`, `$`) avoid bounds checks but shift the arithmetic — be consistent.
- **Interview cue:** if the interviewer only needs *a* longest palindrome and $n$ is small, expand-around-center is fine and far less error-prone. Reach for Manacher when they demand linear time or ask you to *count* palindromes.
