## The hook: the dropdown that reads your mind

Type *car* into any search box and a list drops down: *card*, *care*, *careful*, *cape*… wait, not
*cape* — that does not start with *car*. How does the box know, in a blink, exactly which stored words
continue your prefix and which to leave out? In the last lesson you built a trie and learned that
`startsWith("car")` is a quick walk down three letters. Autocomplete is that walk plus one more idea:
**once you reach the prefix node, every word underneath it is a valid completion.** Collect that
subtree and you have your dropdown.

<Callout type="intuition" title="The one-sentence idea">
Autocomplete is a walk then a sweep: descend to the prefix node, then gather every end-of-word in the
subtree hanging below it — those, and only those, are the words that start with your prefix.
</Callout>

## Intuition: the prefix node is the doorway

A trie already groups words by what they share at the front. So all the words beginning with "car"
do not live scattered across the structure — they all hang under a single node, the one you reach by
walking `c → a → r`. That node is a **doorway** into exactly the matching words and nothing else.

So autocomplete splits cleanly into two phases:

1. **Walk to the prefix node** (`P` steps for a prefix of length `P`). If any letter is missing, the
   prefix matches nothing and you stop early.
2. **Sweep the subtree** below that node with a depth-first traversal, collecting a word every time
   you pass an end-of-word flag, building the string up as you descend.

Word search problems are the same machinery wearing a hat: *does any stored word match this pattern?*
is a walk that branches at a wildcard; *which products match what I have typed so far?* is a walk to a
prefix followed by a sweep, capped at the top few results.

## How it works

Dictionary `["cat", "car", "card", "care", "careful", "cape", "dog"]`, prefix `"car"`:

```
Phase 1 — walk to the prefix node:
  root ─c─ a ─ r          three steps; the 'r' node is the doorway

Phase 2 — DFS sweep below 'r' (build the string as you go):
  r          not end-of-word → keep going
  r-d        "card"   ✓ end-of-word → collect "card"
  r-e        "care"   ✓ end-of-word → collect "care"
  r-e-f-u-l  "careful" ✓ end-of-word → collect "careful"

suggestions = ["card", "care", "careful"]
```

The sweep visits one node per character *in the matching words only*. "cape", "cat" and "dog" sit on
other branches and are never touched — that is why autocomplete stays fast even in a huge dictionary.

## The real implementation

The course file exposes autocomplete as `getWordsWithPrefix`, which reuses the same `findNode` walk
from the basics lesson and then recurses with `collectWords`. The `suggestedProducts` function on top
shows the real-world variant: sort, then return the top 3 after each keystroke (LeetCode 1268).

```ts
// /dsa/11-trie/trie-basics.ts
getWordsWithPrefix(prefix: string): string[] {
    const node = this.findNode(prefix);   // phase 1: walk to the prefix node
    if (!node) return [];                 // prefix not present at all
    const results: string[] = [];
    this.collectWords(node, prefix, results);   // phase 2: sweep the subtree
    return results;
}

private collectWords(node: TrieNode, prefix: string, results: string[]): void {
    if (node.isEndOfWord) results.push(prefix);
    for (const [char, child] of node.children) {
        this.collectWords(child, prefix + char, results);
    }
}

// Search-as-you-type: top 3 suggestions after every character (LeetCode 1268)
function suggestedProducts(products: string[], searchWord: string): string[][] {
    products.sort();
    const trie = new Trie();
    for (const product of products) trie.insert(product);

    const result: string[][] = [];
    let prefix = '';
    for (const char of searchWord) {
        prefix += char;
        const words = trie.getWordsWithPrefix(prefix);
        words.sort();
        result.push(words.slice(0, 3));   // cap at three suggestions
    }
    return result;
}
```

## Complexity

Let $P$ be the prefix length, $k$ the number of matching words, and $m$ their average length. Phase
one walks the prefix; phase two visits every character of every match:

$$
T_{\text{autocomplete}} = O(P + k \cdot m), \qquad S = O(k \cdot m)\ \text{for the result list}.
$$

The headline is that the cost depends on the **size of the answer**, not on $n$, the size of the
dictionary. If you only need the top few suggestions, capping the sweep makes it cheaper still.

| Step | Time | Notes |
|------|------|-------|
| Walk to prefix node | $O(P)$ | one node per prefix character |
| Sweep subtree | $O(k \cdot m)$ | touches only the matching words |
| Top-$t$ cap (sorted trie) | $O(P + t)$ | stop after $t$ completions |
| Brute-force scan (no trie) | $O(n \cdot P)$ | check every word's prefix |

For search-as-you-type over a fixed catalogue, sorting once and walking the trie per keystroke beats
re-scanning the whole list on every letter.

## Common pitfalls & interview tips

- **Re-walking the prefix on every keystroke.** For search-as-you-type, advance from the *previous*
  prefix node by one character instead of restarting at the root — you already paid for the path.
- **Unbounded sweeps.** A prefix like the empty string or a single common letter can match a huge
  subtree. If you only need the top few, stop the DFS once you have collected enough.
- **Ordering surprises.** A `Map`'s children come out in insertion order, not alphabetical. Sort the
  children (or the result) when the problem wants lexicographic suggestions, as `suggestedProducts`
  does.
- **The prefix node itself can be a word.** Completing "car" should still include "car" if it is
  stored — collect at the prefix node too, not only at deeper nodes.
- **Interview cue:** "return all words with prefix", "search suggestions", "type-ahead", or "design
  autocomplete" all map to walk-to-prefix-then-DFS. State the two phases and their $O(P + k m)$ cost.

## Practice

1. **Easy:** For the dictionary in the demo, list by hand every completion of the prefix `"ca"`.
2. **Easy:** What should `autocomplete("")` return, and why is that a useful (but potentially
   expensive) special case?
3. **Medium:** Modify `getWordsWithPrefix` to return only the first $t$ completions, stopping the DFS
   as soon as it has collected $t$ words.
4. **Medium:** Solve *Search Suggestions System* (LeetCode 1268): after each character of the search
   word, return up to three lexicographically smallest products that match.
5. **Hard:** Solve *Word Search II* (LeetCode 212): given a board of letters and a dictionary, find
   every word on the board by walking a trie alongside a DFS of the grid, pruning whenever the current
   path is not a prefix of any word.
