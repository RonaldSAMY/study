## The hook: the interviewer slides a function across the table

You know the zoo. Now comes the real test: someone shows you twenty lines of code and asks, *"What's
the time complexity?"* No stopwatch, no running it — just you, reading the structure and deriving the
answer out loud.

The good news: it comes down to **three mechanical rules**. Count the loops, multiply the nested ones,
add the sequential ones, then drop constants. Master those and you can analyze almost any function on
sight.

<Callout type="intuition" title="The one-sentence idea">
**Nested loops multiply, sequential blocks add, and constants drop** — apply those three rules and the
complexity falls out of the code's shape.
</Callout>

## Intuition: three rules and a recursion trick

**Rule 1 — sequential code adds.** One $O(n)$ loop followed by *another* $O(n)$ loop is
$O(n) + O(n) = O(2n) = O(n)$. You did twice the work, but the *shape* is still linear.

**Rule 2 — nested code multiplies.** A loop running $n$ times *inside* a loop running $n$ times is
$O(n) \times O(n) = O(n^2)$. Each outer step pays for a full inner pass.

**Rule 3 — constants and lower terms drop.** An inner loop that always runs exactly 100 times is
$O(1)$, not $O(n)$ — so two outer loops around it are $O(n^2)$, not $O(100 \, n^2)$.

**The recursion trick.** For recursion, ask two questions: *how many calls happen?* and *how much work
per call?* Two calls per level over $n$ levels is $2^n$ calls → $O(2^n)$. Halving the input each call,
with $O(n)$ work to merge, is $O(n \log n)$.

## How it works: a line-by-line dry run

Take this function and analyze it top to bottom:

```js
function analyze(arr) {
  console.log(arr[0]);                       // (a) O(1)
  for (let i = 0; i < arr.length; i++) {     // (b) O(n)
    console.log(arr[i]);
  }
  for (let i = 0; i < arr.length; i++) {     // (c) O(n)
    for (let j = 0; j < arr.length; j++) {   //     × O(n)
      console.log(arr[i], arr[j]);           //     = O(n²)
    }
  }
}
```

- **(a)** one index lookup → $O(1)$.
- **(b)** one loop over $n$ → $O(n)$.
- **(c)** a loop inside a loop, same array → $O(n) \times O(n) = O(n^2)$.

Add them (Rule 1): $O(1) + O(n) + O(n^2)$. Keep the biggest term (Rule 3): **$O(n^2)$**. Let's count
the operations for real and confirm the $n^2$ term dominates:

## The real implementation

The course file spells out the analysis method directly in code comments. Here's the heart of it:

```js
// /dsa/00-complexity/big-o-notation.js

function analyzeThis(arr) {
  console.log(arr[0]);                          // O(1)
  for (let i = 0; i < arr.length; i++) {        // O(n)
    console.log(arr[i]);
  }
  for (let i = 0; i < arr.length; i++) {        // O(n) — separate, so ADD
    console.log(arr[i] * 2);
  }
  // Total: O(1) + O(n) + O(n) = O(2n + 1) = O(n)
}

function analyzeNested(arr) {
  for (let i = 0; i < arr.length; i++) {        // O(n)
    for (let j = 0; j < arr.length; j++) {      // × O(n)
      console.log(arr[i], arr[j]);              // = O(n²)
    }
  }
}
```

## Complexity

The same code can have *different* best, average, and worst cases. Consider searching for a duplicate
pair with an early `return`:

$$
\text{best} = O(1), \qquad \text{average} = O(n^2), \qquad \text{worst} = O(n^2).
$$

| Case | When | Cost |
| --- | --- | --- |
| Best | the pair is the first two elements, return immediately | $O(1)$ |
| Average | the pair sits somewhere in the middle | $O(n^2)$ |
| Worst | no pair exists, so both loops run fully | $O(n^2)$ |

<Callout type="why">
Worst case is the honest headline here: an early `return` *can* finish in $O(1)$, but you can't promise
it. When the data refuses to cooperate, the nested loops run to completion — so we report $O(n^2)$.
</Callout>

## The same idea, different worlds

<ExampleCard domain="Image processing" icon="🖼️">
A filter that visits every pixel of a $w \times h$ photo is $O(w \cdot h)$ — linear in the pixel count.
But a naïve blur that, for each pixel, scans a patch *of size proportional to the image* becomes
$O((wh)^2)$ and grinds to a halt. Real filters keep the patch a *constant* size, so the kernel loop is
$O(1)$ and the whole pass stays linear.
</ExampleCard>

<ExampleCard domain="Finance" icon="💹">
Summing a column of $n$ trades is $O(n)$. Checking every pair of trades for a matching offset is the
$O(n^2)$ nested loop — fine for one desk's hundred trades, ruinous for a million-row ledger. Quants
swap it for a hash-map pass that's $O(n)$.
</ExampleCard>

<ExampleCard domain="Sports" icon="🏆">
A round-robin tournament where every team plays every other team is exactly the triangular loop:
$n(n-1)/2$ matches, which is $O(n^2)$. Eight teams → 28 games; double to sixteen → 120 games. The
schedule grows quadratically even though each team's own slate grows only linearly.
</ExampleCard>

<ExampleCard domain="Climate" icon="🌡️">
A weather model updates an $n \times n$ grid of cells, and for each cell looks at its handful of
neighbors — a *constant* number. So it's $O(n^2)$ in the grid width (the cell count), not $O(n^3)$.
Doubling the resolution quadruples the cells, and the runtime with it.
</ExampleCard>

## Common pitfalls &amp; interview tips

- **Two different collections give $O(n \cdot m)$, not $O(n^2)$.** Only write $n^2$ when both loops run
  over the *same* data. A 100-element array crossed with a 5-element one is 500 ops, not 10,000.
- **A constant-bound inner loop is $O(1)$.** `for (k = 0; k < 100; k++)` doesn't depend on $n$; it
  vanishes into the constant. Two outer loops around it are $O(n^2)$, full stop.
- **`j` starting at `i` is still $O(n^2)$.** The triangular $n(n+1)/2$ is half a square — a smaller
  constant, the same class.
- **Watch hidden costs in built-ins.** `arr.slice()`, `arr.indexOf()`, and `[...arr]` are each $O(n)$;
  put one inside a loop and you've quietly created an $O(n^2)$.

## Practice

From the course's practice set — derive the time complexity:

1. **(Medium)** Nested loops where the inner loop runs over a *second* array `arr2`. *(Answer:
   $O(n \cdot m)$ — different sizes, so it's a product, not $n^2$.)*
2. **(Medium)** Outer loop over $n$, inner loop with `j = i`, body $O(1)$. *(Answer: $O(n^2)$ —
   $n(n+1)/2$ iterations.)*
3. **(Hard)** Two loops over $n$ wrapping an inner loop that runs exactly 100 times. *(Answer: $O(n^2)$
   — the 100 is constant: $O(100 n^2) = O(n^2)$.)*
