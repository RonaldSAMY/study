## The hook: exploring a maze with one hand on the wall

Drop yourself into a stone maze with a single rule: pick a corridor, follow it as far as it goes, and
only when you hit a dead end do you walk back to the last junction and try the next unexplored door.
You never wander back to the entrance to reconsider — you plunge forward, committing to each choice
until the path runs out. That stubborn "go deep, then backtrack" instinct is exactly **depth-first
search**. Where BFS fanned out level by level with a queue, DFS dives down one branch with a stack,
and the order in which it finishes corridors turns out to reveal a surprising amount about a graph's
structure.

<Callout type="intuition" title="The one-sentence idea">
DFS explores a graph by **going as deep as possible** along each branch before backtracking, using a
**stack** (an explicit one, or the call stack via recursion) to remember where to resume.

</Callout>

## Intuition

In the last lesson, **BFS** used a **queue** (first-in, first-out) to visit vertices in rings of
increasing distance from the start. DFS swaps that one data structure: it uses a **stack**
(last-in, first-out). That single change flips the whole personality of the traversal.

Push the start vertex. Then repeat: **pop** a vertex, mark it visited, and **push** its unvisited
neighbours. Because the stack hands you back the *most recently* pushed vertex, you immediately dive
into the neighbour you just discovered rather than the siblings you saw earlier. You keep plunging
deeper until you reach a vertex with no unvisited neighbours — a dead end — at which point popping the
stack naturally **backtracks** you to the most recent junction that still has doors left to try.

Recursion expresses the same idea even more compactly: "visit me, then recursively visit each unvisited
neighbour." The language's **call stack** plays the role of the explicit stack for free. Both versions
visit every vertex and every edge exactly once, so both run in $O(V + E)$ time — the same budget as BFS,
just a different shape of exploration.

## How it works

Let us hand-trace recursive DFS on a tiny graph, starting at `A`. We mark a vertex the instant we enter
it, then recurse into neighbours in alphabetical order, skipping any already visited.

```
graph:  A: [B, C]   B: [A, D, E]   C: [A, F]   D: [B]   E: [B, F]   F: [C, E]

visit A            -> order: A
  visit B          -> order: A B
    visit D        -> order: A B D   (D's only neighbour B is visited -> backtrack)
    visit E        -> order: A B D E
      visit F      -> order: A B D E F   (F's neighbours C, E lead on...)
        visit C    -> order: A B D E F C   (C's neighbours all visited -> unwind)
pre-order  (entered): A B D E F C
post-order (finished): D C F E B A
```

Notice two useful orderings fall out for free. **Pre-order** is the sequence in which we *enter*
vertices; **post-order** is the sequence in which we *finish* them (after all descendants are done).
Post-order, reversed, is the backbone of topological sort in the lessons just ahead.

// so the smallest is popped first, matching the recursive order)
function dfsIterative(graph, start) {
  const visited = new Set();
  const order = [];
  const stack = [start];
  while (stack.length > 0) {
    const v = stack.pop();
    if (visited.has(v)) continue;
    visited.add(v);
    order.push(v);
    const ns = graph[v] || [];
    for (let i = ns.length - 1; i >= 0; i--) {
      if (!visited.has(ns[i])) stack.push(ns[i]);
    }
  }
  return order;
}

## The real implementation

Here is the faithful core from the course file — a generic recursive DFS plus the iterative version that
swaps the call stack for an explicit one (handy when a graph is deep enough to overflow the recursion
limit):

```ts
// /dsa/12-graphs/dfs.ts
function dfsRecursive<T>(graph: Map<T, T[]>, start: T, visited = new Set<T>()): T[] {
    const result: T[] = [];
    function dfs(vertex: T): void {
        visited.add(vertex);
        result.push(vertex);
        for (const neighbor of graph.get(vertex) || []) {
            if (!visited.has(neighbor)) dfs(neighbor);
        }
    }
    dfs(start);
    return result;
}

// Iterative DFS with an explicit stack (avoids recursion-depth limits)
function dfsIterative<T>(graph: Map<T, T[]>, start: T): T[] {
    const visited = new Set<T>();
    const result: T[] = [];
    const stack: T[] = [start];
    while (stack.length > 0) {
        const vertex = stack.pop()!;
        if (!visited.has(vertex)) {
            visited.add(vertex);
            result.push(vertex);
            const neighbors = graph.get(vertex) || [];
            for (let i = neighbors.length - 1; i >= 0; i--) {
                if (!visited.has(neighbors[i])) stack.push(neighbors[i]);
            }
        }
    }
    return result;
}
```

Run the full file: `npx tsx 12-graphs/dfs.ts`.

As DFS runs it implicitly classifies every edge. A **tree edge** leads to a brand-new vertex (these form
the emerald DFS tree in the demo). A **back edge** points to an ancestor still on the stack — the telltale
sign of a cycle, which is exactly what the next lesson exploits. **Forward** and **cross edges** (in
directed graphs) point to already-finished vertices. And remember the two visit orders: **pre-order**
records when a vertex is entered, **post-order** when it is finished — reversing post-order gives a valid
topological ordering, the foundation of the topological-sort lesson coming up.

## Complexity

Let $V$ be the number of vertices and $E$ the number of edges. DFS touches every vertex once and scans
every edge once, so:

$$
\text{time } O(V + E), \qquad \text{space } O(V).
$$

The space is the depth of the recursion (or the explicit stack) plus the visited set — in the worst case
a single long path forces all $V$ vertices onto the stack at once. DFS and BFS share the same time bound
but differ sharply in feel:

| Aspect | BFS | DFS |
|--------|-----|-----|
| Frontier structure | Queue (FIFO) | Stack (LIFO) |
| Extra memory | $O(\text{width})$ — a whole level | $O(\text{height})$ — one path deep |
| Finds shortest path (unweighted) | Yes | No |
| Enumerating all paths / deep solutions | Harder | Easier |
| Natural for | Levels, nearest things | Backtracking, ordering, cycles |

The memory contrast is the practical headline: BFS can blow up on a bushy graph with enormous levels,
while DFS stays cheap on those but can run deep on a long, skinny graph.

## The same idea, different worlds

<ExampleCard domain="Maze solving" icon="🧩">
A robot solving a labyrinth follows one corridor to its end, and only on hitting a wall does it retreat
to the last junction and try an untried door. That "deep then backtrack" wall-following is DFS, with the
stack remembering every junction left half-explored behind you.

</ExampleCard>

<ExampleCard domain="File systems" icon="🗂️">
When `du` or a recursive `find` walks a directory tree, it descends fully into the first subfolder,
totals everything inside it, and only then climbs back to visit the next sibling folder. That is DFS
over the directory graph — the call stack mirrors the folder path you are currently inside.

</ExampleCard>

<ExampleCard domain="Memory management" icon="♻️">
A garbage collector's mark-and-sweep phase starts from the root references and follows pointers as deep
as they go, marking every object it can reach. Anything DFS never touches is unreachable and safe to
free — reachability is just a depth-first traversal of the object graph.

</ExampleCard>

<ExampleCard domain="Puzzle solving" icon="👑">
A Sudoku or N-Queens backtracking solver is DFS over a tree of partial states: place a value, recurse to
the next cell, and if a constraint breaks, backtrack and try the next value. Each "undo" is the stack
popping back to the last decision that still has options.

</ExampleCard>

## Common pitfalls & interview tips

- **Forgetting the visited set.** Without it, any cycle sends DFS into infinite recursion. Mark a vertex
  the moment you enter it, before recursing into neighbours.
- **Marking at the wrong time in the iterative version.** Check and set "visited" when you **pop**, not
  when you push — a vertex can be pushed several times by different neighbours before it is ever popped.
- **Recursion-depth overflow.** A graph that is one long chain of, say, 100,000 vertices can blow the
  call stack. Switch to the explicit-stack iterative version when depth could be large.
- **Reversed-order surprise.** Pushing neighbours in forward order makes the iterative DFS visit them in
  reverse. Push in reverse if you want it to match the recursive, alphabetical order (as in the demo).
- **Interview cue:** if a problem asks for *any* path, connected components, cycle detection, or an
  ordering of dependencies, reach for DFS. If it asks for the *shortest* unweighted path, reach for BFS.

## Practice

1. **Easy:** Hand-trace recursive DFS on `A-B, A-C, B-D` from `A`, pushing neighbours alphabetically.
   Write the visit order.
2. **Easy:** For the same graph, give the **post-order** (the order vertices *finish*).
3. **Medium:** Modify `dfsIterative` to also return each vertex's **depth** (distance from the start
   along the DFS tree).
4. **Medium:** Write `countComponents(graph)` that runs DFS from every unvisited vertex and counts how
   many separate connected components a graph has.
5. **Hard:** Detect whether an **undirected** graph has a cycle using DFS, by spotting a visited
   neighbour that is not the vertex you arrived from.
