## The hook: wiring up a new town for the least cash

A property developer has staked out seven houses and needs to run electrical cable so every house has
power. Cable costs money by the metre, and the survey lists the price of trenching between each pair of
houses. You do not need a direct line from every house to every other house — you only need the whole
estate to be **connected**, so current can reach all of it. Lay too much cable and you burn the budget;
lay too little and a house goes dark. The cheapest set of trenches that still links everyone is a
**minimum spanning tree**, and finding it is one of the most satisfying greedy algorithms in computing.

<Callout type="intuition" title="The one-sentence idea">
A minimum spanning tree connects all $V$ vertices of a weighted undirected graph using exactly $V-1$
edges, no cycles, and the smallest possible total weight.
</Callout>

## Intuition

Take a connected, weighted, **undirected** graph. A **spanning tree** is any subset of edges that touches
every vertex while staying a tree: connected and acyclic. A tree on $V$ vertices always has exactly $V-1$
edges, so a spanning tree is the leanest possible way to keep the graph in one piece. A **minimum**
spanning tree is the spanning tree whose edge weights add up to the least total.

Why is greed safe here? Because of the **cut property**: for any way of splitting the vertices into two
groups, the cheapest edge crossing the split must belong to some minimum spanning tree. That single fact
is what licenses us to keep grabbing cheap edges without fear of regret.

**Kruskal's algorithm** turns that idea into a recipe driven by edges:

1. Sort every edge by weight, cheapest first.
2. Walk the sorted list. For each edge, ask: do its two endpoints already sit in the same connected
   component?
3. If they sit in **different** components, the edge joins two pieces that were not yet linked — keep it,
   and merge the two components.
4. If they sit in the **same** component, they are already connected, so this edge would close a cycle —
   skip it.
5. Stop once you have collected $V-1$ edges. The tree is complete.

The "same component?" question is exactly what **Union-Find** answers in near-constant time, which is why
the previous lesson set it up. `find` tells you which component a vertex is in; `union` merges two
components and reports whether they really were distinct.

There is a second classic, **Prim's algorithm**, which is *vertex*-based: start from one vertex and
repeatedly pull in the cheapest edge that reaches a new vertex, using a priority queue to find it. Both
find a minimum spanning tree; they just grow it differently. Kruskal shines on **sparse** graphs (it only
sorts the edge list), while Prim tends to win on **dense** graphs.

Here is the cheapest-first walk over the demo's default graph. Each line asks whether the edge's two ends


function kruskal(n, edges) {
  const parent = Array.from({ length: n }, (_, i) => i);
  const find = (x) => { while (parent[x] !== x) { parent[x] = parent[parent[x]]; x = parent[x]; } return x; };
  const union = (a, b) => {
    const ra = find(a), rb = find(b);
    if (ra === rb) return false;        // same component, would form a cycle
    parent[ra] = rb;                    // merge the two components
    return true;
  };

  const sorted = [...edges].sort((a, b) => a[2] - b[2]);  // cheapest first
  const chosen = [];
  let total = 0;
  for (const [u, v, w] of sorted) {
    if (union(u, v)) {                  // joins two components, keep it
      chosen.push([u, v, w]);
      total += w;
      if (chosen.length === n - 1) break;
    }
  }
  return { total, chosen };
}

## The real implementation

The course file uses the path-compression, union-by-rank `UnionFind` from the previous lesson. The key
move is that `union` returns `false` when the two endpoints already share a component — so the cycle test
and the merge happen in a single call.

```ts
// /dsa/12-graphs/mst.ts  — Kruskal's algorithm
interface Edge { from: number; to: number; weight: number; }

// Union-Find from the previous lesson (path compression + union by rank).
// union(a, b) returns false when a and b are already in the same set.
class UnionFind {
    private parent: number[];
    private rank: number[];
    constructor(n: number) {
        this.parent = Array.from({ length: n }, (_, i) => i);
        this.rank = new Array(n).fill(0);
    }
    find(x: number): number {
        if (this.parent[x] !== x) this.parent[x] = this.find(this.parent[x]);  // path compression
        return this.parent[x];
    }
    union(a: number, b: number): boolean {
        const ra = this.find(a), rb = this.find(b);
        if (ra === rb) return false;                       // same component → cycle
        if (this.rank[ra] < this.rank[rb]) this.parent[ra] = rb;
        else if (this.rank[ra] > this.rank[rb]) this.parent[rb] = ra;
        else { this.parent[rb] = ra; this.rank[ra]++; }    // union by rank
        return true;
    }
}

function kruskalMST(numVertices: number, edges: Edge[]): { edges: Edge[]; totalWeight: number } {
    const sorted = [...edges].sort((a, b) => a.weight - b.weight);  // cheapest first
    const uf = new UnionFind(numVertices);
    const mstEdges: Edge[] = [];
    let totalWeight = 0;
    for (const e of sorted) {
        if (uf.union(e.from, e.to)) {           // joins two components → keep it
            mstEdges.push(e);
            totalWeight += e.weight;
            if (mstEdges.length === numVertices - 1) break;   // tree complete
        }
        // else: same component → skip (would create a cycle)
    }
    return { edges: mstEdges, totalWeight };
}
```

## Complexity

Let $V$ be the number of vertices and $E$ the number of edges. Kruskal's cost is dominated by the sort;
the Union-Find work that follows is near-linear because each `find` and `union` runs in almost constant
amortised time:

$$
\text{Kruskal: } O(E \log E), \qquad \text{Prim (binary heap): } O(E \log V), \qquad \text{space } O(V).
$$

Since $E \le V^2$, we have $\log E \le 2 \log V$, so $O(E \log E)$ and $O(E \log V)$ are the same order —
the practical choice comes down to the graph's density and whether the edges arrive already as a list.

| Aspect | Kruskal | Prim |
|--------|---------|------|
| Driven by | edges (sorted) | vertices (grown from a seed) |
| Core data structure | Union-Find | priority queue / min-heap |
| Time | $O(E \log E)$ | $O(E \log V)$ |
| Best when | graph is **sparse** | graph is **dense** |
| Needs the whole edge list up front | yes (to sort) | no (grows incrementally) |

## Common pitfalls & interview tips

- **Using it on a directed graph.** Minimum spanning trees are defined for **undirected** graphs. The
  directed analogue (a minimum spanning arborescence) needs a different algorithm entirely.
- **Forgetting the cycle check.** If you add edges purely by weight without asking whether the endpoints
  are already connected, you get cycles and overspend. The Union-Find `union`-returns-false test is the
  whole trick.
- **Not stopping at $V-1$ edges.** Once the tree has $V-1$ edges it spans everything; continuing only
  wastes time considering edges that must form cycles.
- **Assuming the MST is unique.** When several edges share a weight, more than one minimum spanning tree
  can exist — but they all have the **same total weight**.
- **Interview cue:** if the prompt says "connect all the nodes for the minimum total cost" on a weighted
  undirected graph, reach for Kruskal plus Union-Find first; mention Prim if the graph is dense.
