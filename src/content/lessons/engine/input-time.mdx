## The hook: "I pressed jump RIGHT on time!"

You are one pixel from the ledge. You stab the jump button the instant your feet touch the
ground — and your character walks calmly off the edge into the void. You *know* you pressed it.
The replay agrees you pressed it. So why did the game ignore you?

It didn't ignore you. The game simply wasn't *looking* at the exact microsecond your thumb moved.
A game reads the controller at fixed moments — once per frame — and your press happened in the gap
between two of those moments. The fix that every good game ships is tiny and beautiful: remember the
press for a few frames so "almost on time" becomes "on time."

<Callout type="intuition" title="The one-sentence idea">
A game samples your input **once per frame**, so a press lands on the *next* frame after you make
it — and an **input buffer** remembers that press for a few frames so it still counts when the moment
to use it finally arrives.
</Callout>

## Intuition: input is just one more step in the loop

In **the game loop** you saw the heartbeat: every frame the engine does *update*, then *render*,
over and over. Reading the player is not some separate magic — it is the very first thing *update*
does. Each turn of the loop the engine asks the hardware, "what is held down right now?", acts on the
answer, then draws the result.

That has a consequence people miss: between two frames, the game is effectively blind. Your keypress
might physically happen at 8.3 ms into a 16.7 ms frame, but the engine won't *see* it until it
reaches the next "read input" step. So input is always quantized to frame boundaries — it snaps to
the grid of frames, just like the fixed timestep snapped your physics in the previous lesson.

A buffer is the gentle fix. Instead of throwing a press away the instant the current frame can't use
it, the engine keeps it on a sticky note for a few frames. When a usable moment shows up — you land,
the attack recovers, the menu opens — the press is still there, waiting.

## The formal definition

There are two ways to get input from the player:

- **Event-driven**: the operating system hands you a message the moment something happens
  ("key A went down at time $t$"). Events are precise but arrive *whenever* — possibly several, or
  none, between two frames.
- **Polling**: once per frame the game asks "what is the state right now?" and reads it. Simple and
  predictable, but it only looks at discrete instants.

Most engines do both: events are collected as they arrive, then the game *polls* the accumulated
state at the start of each frame. If frames run at a rate of $\text{fps}$ frames per second, the time
between samples is

$$
\Delta t = \frac{1}{\text{fps}}.
$$

At 60 fps that is about $16.7$ ms. A press can sit unseen for up to one whole $\Delta t$ before the
next sample notices it.

An **input buffer** is a tiny queue that remembers a press for a window of $N$ frames. In seconds
that window is

$$
w = \frac{N}{\text{fps}}.
$$

A press made at frame $f_{\text{press}}$ is still valid at an action frame $f$ as long as it falls
inside the window:

$$
f_{\text{press}} \in [\, f - N,\ f \,].
$$

The mirror image is **coyote time**: for a few frames *after* you leave a ledge, the game still lets
you jump as if you were grounded — forgiveness applied to the *late* side instead of the early side.

A per-frame poll plus buffer check is just a few lines:

```js
// runs once every frame, inside update()
if (input.pressed("jump")) buffer = N;   // remember the press for N frames
if (canAct && buffer > 0) {              // a usable moment arrived
  doJump();
  buffer = 0;                            // consume it
}
buffer = Math.max(0, buffer - 1);        // age the buffer each frame
```

## The same idea, four different worlds

Sampling a continuous world at discrete moments — and adding a grace window so near-misses still
count — shows up far beyond a jump button.

<ExampleCard domain="Fighting games" icon="🥊">
A combo only links if your next attack arrives during a short **cancel window**. Press too early and
nothing happens — except buffered inputs *do* fire on the first valid frame. Pros literally pre-press
the next move so the engine executes it the instant the window opens. That is the input buffer doing
competitive work.
</ExampleCard>

<ExampleCard domain="Photography" icon="📷">
A camera shutter samples a moving scene at a single instant. A propeller spinning between exposures
gets frozen mid-blur, and fast subjects can fall *between* frames of a video entirely. Same lesson:
a continuous world read at discrete moments. Burst mode is the buffer — capture a spread of frames so
the perfect instant is somewhere in the set.
</ExampleCard>

<ExampleCard domain="Civic deadlines" icon="🗳️">
A mail-in ballot must arrive by a cutoff, but many places accept ballots **postmarked** by election
day even if they land a few days later. The postmark is your press; the grace period is the buffer
window $N$. Arrive inside the window and your input still counts — exactly the jump-buffer rule in
civic clothing.
</ExampleCard>

<ExampleCard domain="Rhythm & mobile games" icon="🎵">
A rhythm tap is judged against a **hit window** of a few milliseconds around the beat. Touchscreen
games widen that window on purpose, because a finger on glass is laggier and less precise than a
button. Tighten the window for "Perfect", widen it for "Good" — you are literally tuning $w = N/\text{fps}$.
</ExampleCard>

<Callout type="note">
Notice the spread: a 60 fps platformer, a fighting-game cancel, a postal deadline measured in days.
The units change wildly, but the structure is identical — discrete samples plus a tolerance window.
</Callout>
