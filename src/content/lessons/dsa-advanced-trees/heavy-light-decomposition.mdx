## The hook: a segment tree is great on an array — so flatten the tree

A segment tree answers "sum of this contiguous range" in $O(\log n)$ and updates a value just as fast —
but only on a flat array. Trees are not flat: the path between two nodes zig-zags up and down and is *not*
a contiguous slice of any linearization. Heavy-light decomposition (HLD) is the bridge. It carves the tree
into a small set of **straight chains**, lays each chain down as a contiguous block of one big array, and
proves that the path between any two nodes touches only $O(\log n)$ of those chains. A path query then
becomes a handful of ordinary segment-tree range queries — so "maximum toll on the road from $u$ to $v$,
with live updates" runs in $O(\log^2 n)$ instead of $O(n)$.

<Callout type="intuition" title="The one-sentence idea">
From each node draw a "heavy" edge to its largest-subtree child; the heavy edges link up into long chains,
each chain becomes a contiguous array segment, and because a path can enter at most $O(\log n)$ chains, a
tree-path query reduces to $O(\log n)$ array-range queries.
</Callout>

## Intuition: why "heavy" edges make chains short in number

At every node, look at its children and follow the edge into the child with the **largest subtree** — call
that the heavy edge; all its siblings are light edges. Chaining heavy edges together partitions the tree
into disjoint vertical paths.

The magic is a counting argument about light edges. When you step *down* a light edge, you move into a
subtree of at most **half** the size of where you were — because if a sibling's subtree were more than
half, *it* would have been the heavy child. Halving can happen at most $\log_2 n$ times before the subtree
shrinks to one node, so **any root-to-leaf path crosses at most $O(\log n)$ light edges**, hence enters at
most $O(\log n)$ chains. A general $u$–$v$ path is just two root-ward walks meeting at their LCA, so it too
spans only $O(\log n)$ chains. Because the second DFS visits the heavy child first, every chain occupies a
*contiguous* run of positions in the flattened array — exactly what a segment tree needs.

## How it works

Two depth-first passes build everything:

```
1. dfs1: compute parent, depth, subtreeSize, and pick heavy[v] = child with the largest subtree.
2. dfs2: assign positions; recurse into the HEAVY child first (same chainHead),
         then into each light child (which opens a NEW chain headed by itself).
3. Load node values into a segment tree indexed by chainPos.
4. queryPath(u, v): while u and v are in different chains, lift the one whose chain head is deeper —
   add the segment-tree range from its chain head up to it, then jump to the parent of that head.
   When they share a chain, add the single range between them. That closing range straddles the LCA.
```

const adj = Array.from({ length: N }, () => []);
for (const [a, b] of EDGES) { adj[a].push(b); adj[b].push(a); }

const parent = Array(N).fill(-1), depth = Array(N).fill(0);
const size = Array(N).fill(1), heavy = Array(N).fill(-1);
function dfs1(v, p, d) {
  parent[v] = p; depth[v] = d; size[v] = 1;
  let best = 0;
  for (const u of adj[v]) if (u !== p) {
    dfs1(u, v, d + 1); size[v] += size[u];
    if (size[u] > best) { best = size[u]; heavy[v] = u; } // largest-subtree child
  }
}
dfs1(0, -1, 0);

const head = Array(N).fill(0), pos = Array(N).fill(0);
let cur = 0;
function dfs2(v, h) {
  head[v] = h; pos[v] = cur++;
  if (heavy[v] !== -1) dfs2(heavy[v], h);                 // heavy child continues the chain
  for (const u of adj[v]) if (u !== parent[v] && u !== heavy[v]) dfs2(u, u); // light child: new chain
}
dfs2(0, 0);

const arr = Array(N).fill(0);
for (let v = 0; v < N; v++) arr[pos[v]] = values[v];      // contiguous per chain
const rangeSum = (l, r) => { let s = 0; for (let i = l; i <= r; i++) s += arr[i]; return s; };

  if (depth[u] > depth[v]) { const t = u; u = v; v = t; }
  res += rangeSum(pos[u], pos[v]);                        // final chunk across the LCA
  return res;
}`}

## The real implementation

The course file builds a lazy segment tree, runs the two DFS passes, and exposes path/subtree queries plus
updates. Here is the path-query loop that walks chain by chain — the heart of HLD:

```ts
// /dsa/20-advanced-trees/heavy-light-decomposition.ts
queryPath(u: number, v: number): number {
  let result = 0;

  while (this.chainHead[u] !== this.chainHead[v]) {
    // Move the one with deeper chain head up
    if (this.depth[this.chainHead[u]] < this.depth[this.chainHead[v]]) {
      [u, v] = [v, u];
    }

    // Query from u to head of its chain
    result += this.segTree.query(
      this.chainPos[this.chainHead[u]],
      this.chainPos[u]
    );

    // Move to parent of chain head
    u = this.parent[this.chainHead[u]];
  }

  // Both in same chain now
  if (this.depth[u] > this.depth[v]) {
    [u, v] = [v, u];
  }

  result += this.segTree.query(this.chainPos[u], this.chainPos[v]);
  return result;
}
```

## Complexity

A path visits $O(\log n)$ chains, and each chain is handled by one segment-tree operation costing
$O(\log n)$, so path work multiplies to $O(\log^2 n)$. Subtree queries need only a single range (a subtree
is itself contiguous under this ordering), so they cost $O(\log n)$:

$$
\text{build} = O(n), \qquad \text{path query/update} = O(\log^2 n), \qquad \text{subtree query} = O(\log n).
$$

| Operation | Cost | Why |
|-----------|------|-----|
| Build (two DFS + seg tree) | $O(n)$ | linear passes over the tree |
| Path query / update | $O(\log^2 n)$ | $O(\log n)$ chains $\times$ $O(\log n)$ per range |
| Subtree query / update | $O(\log n)$ | one contiguous range |
| LCA (as a by-product) | $O(\log n)$ | walk up until chains match |

The extra $\log$ over plain binary-lifting LCA is the price of also supporting **updates** along paths —
that is HLD's whole reason to exist.

## Common pitfalls & interview tips

- **Recursing light-first.** The heavy child *must* be visited first in `dfs2`, otherwise a chain's nodes
  are no longer contiguous in the array and the segment tree ranges break.
- **Node values vs. edge values.** If weights live on edges, push each edge's value onto its *child* node
  and exclude the LCA from the final range — a classic silent off-by-one.
- **Forgetting to swap by chain-head depth.** In the loop you lift whichever endpoint has the *deeper chain
  head*, not the deeper node; swapping on the wrong key can loop forever.
- **Confusing HLD with plain LCA.** If you only need LCA or distance on a static tree, binary lifting is
  simpler and one $\log$ faster. Reach for HLD when the tree carries **updatable path data**.
- **Interview cue:** "Path queries with updates on a tree?" → "Heavy-light decomposition: chains plus a
  segment tree, $O(\log^2 n)$ per operation."
