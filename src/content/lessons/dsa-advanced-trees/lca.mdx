## The hook: the meeting point of two family lines

Point at any two people in a family tree and ask "who is their closest shared ancestor?" You could crawl
up from each of them one generation at a time and wait for the paths to collide — but that is $O(n)$ per
question, and on a tree with millions of nodes answering thousands of questions that way is hopeless. The
**lowest common ancestor** (LCA) is the deepest node that is an ancestor of both $u$ and $v$, and it is
the hinge of nearly every tree-path computation: the distance between two nodes, the sum along a path, the
node where two lineages merge. The trick that makes it fast is to precompute, for every node, its
ancestor $2^0, 2^1, 2^2, \dots$ levels up — so you can leap up the tree in giant power-of-two strides
instead of crawling one step at a time.

<Callout type="intuition" title="The one-sentence idea">
Store each node's $2^j$-th ancestor in a table, then answer any LCA query by first lifting the deeper node
to the same depth and then jumping both nodes upward by the largest strides that keep them apart — where
they finally meet is the LCA.
</Callout>

## Intuition: climb in powers of two, not one step at a time

Every positive integer is a sum of distinct powers of two — that is just its binary representation. Binary
lifting turns that fact into a movement strategy on a tree. If a node needs to rise 13 levels, and
$13 = 8 + 4 + 1$, then three precomputed jumps ($2^3$, then $2^2$, then $2^0$) get it there instead of
thirteen single steps.

The query has two phases. First, **level the two nodes**: whichever is deeper is lifted by exactly
$\text{depth}(u) - \text{depth}(v)$ levels, using the binary decomposition above, so both sit at the same
depth. If they landed on the same node, one was an ancestor of the other and we are done. Otherwise,
**rise together**: for $j$ from large to small, if the $2^j$-th ancestors of $u$ and $v$ *differ*, jump
both up by $2^j$. Each such jump is safe because differing ancestors are still strictly below the LCA.
When no jump is safe anymore, $u$ and $v$ are the two children just below the meeting point, so their
common parent $\text{up}[u][0]$ is the LCA.

## How it works

Preprocessing runs one DFS to record each node's depth and immediate parent, then fills the jump table by
the recurrence "your $2^j$-th ancestor is the $2^{j-1}$-th ancestor of your $2^{j-1}$-th ancestor":

```
1. DFS from the root: record depth[v] and up[v][0] = parent(v).
2. For j = 1..LOG:  up[v][j] = up[ up[v][j-1] ][j-1]   (a jump is two half-jumps).
3. Query(u, v): lift the deeper node to the shallower one's depth using the binary digits of the gap.
4. If they coincide, that's the LCA. Otherwise jump both up by every 2^j whose ancestors still differ.
5. The LCA is up[u][0] — the parent just above where they stopped.
```

function dfs(v, p, d) {
  depth[v] = d; up[v][0] = p;
  for (const u of adj[v]) if (u !== p) dfs(u, v, d + 1);
}
dfs(0, -1, 0);
// Build the jump table: a 2^j hop is two 2^(j-1) hops stitched together.
for (let j = 1; j < LOG; j++)
  for (let i = 0; i < N; i++)

function kthAncestor(v, k) {
  for (let j = 0; j < LOG && v !== -1; j++) if ((k >> j) & 1) v = up[v][j];
  return v;
}
function lca(u, v) {
  if (depth[u] < depth[v]) { const t = u; u = v; v = t; } // u is deeper

  return up[u][0];
}

## The real implementation

The course file offers four LCA strategies — naive, binary lifting, Euler tour + RMQ, and Tarjan's offline
union-find. Here is the binary-lifting query, the workhorse most people reach for because it also answers
"$k$-th ancestor" and "distance" for free:

```ts
// /dsa/20-advanced-trees/lca-algorithms.ts
query(u: number, v: number): number {
  // Ensure u is deeper
  if (this.depth[u] < this.depth[v]) {
    [u, v] = [v, u];
  }

  // Bring u to same depth as v
  const diff = this.depth[u] - this.depth[v];
  u = this.getKthAncestor(u, diff);

  if (u === v) return u;

  // Binary search for LCA
  for (let j = this.LOG - 1; j >= 0; j--) {
    if (this.parent[u][j] !== this.parent[v][j]) {
      u = this.parent[u][j];
      v = this.parent[v][j];
    }
  }

  return this.parent[u][0];
}
```

## Complexity

Let $n$ be the number of nodes. The jump table has $n$ rows and $\lceil\log_2 n\rceil + 1$ columns, and
each query performs a constant amount of work per column:

$$
\text{preprocess} = O(n \log n), \qquad \text{query} = O(\log n), \qquad \text{space} = O(n \log n).
$$

| Approach | Preprocessing | Query | Space |
|----------|---------------|-------|-------|
| Naive (crawl to root) | $O(1)$ | $O(n)$ | $O(1)$ |
| Binary lifting | $O(n \log n)$ | $O(\log n)$ | $O(n \log n)$ |
| Euler tour + sparse-table RMQ | $O(n \log n)$ | $O(1)$ | $O(n \log n)$ |
| Tarjan's offline (union-find) | $O(n + q)$ | amortized $O(1)$ | $O(n)$ |

Binary lifting is the sweet spot for online queries: simple to code, and its jump table doubles as a
$k$-th-ancestor oracle. The Euler-tour method wins when you need true $O(1)$ queries and never update the
tree.

## The same idea, different worlds

<ExampleCard domain="Version control" icon="🔀">
When Git performs a three-way merge it computes the <code>merge-base</code> of two branches — the lowest
common ancestor in the commit DAG. The diff each side made <em>since that shared commit</em> is exactly
what gets reconciled.
</ExampleCard>

<ExampleCard domain="Computational biology" icon="🧬">
Phylogenetic software maps species onto a tree of life; the LCA of two organisms is their most recent
common ancestor, and the distance formula $\text{depth}(u) + \text{depth}(v) - 2\,\text{depth}(\text{LCA})$
estimates evolutionary divergence.
</ExampleCard>

<ExampleCard domain="Networking" icon="🌐">
IP multicast and routing hierarchies deliver a packet up to the LCA of the sender and receiver in the
distribution tree, then back down — the shortest path in a tree always passes through their LCA.
</ExampleCard>

## Common pitfalls & interview tips

- **Forgetting to level first.** Jumping both nodes together only works once they share a depth. Lift the
  deeper node by exactly the depth gap before the second phase, or the "differ?" test is meaningless.
- **Overshooting the LCA.** During the rise-together phase you jump only when the $2^j$-th ancestors
  *differ*. If you jumped when they were equal you would sail past the LCA into a common ancestor above it.
- **Off-by-one on `LOG`.** The table needs $\lceil\log_2 n\rceil + 1$ columns; sizing it to
  $\lfloor\log_2 n\rfloor$ silently drops the top jump on the deepest trees.
- **Rebuilding on every change.** Binary lifting assumes a static tree. If parents change often you must
  rebuild the table — that is the cue to reach for a link-cut tree or Euler-tour-on-a-BIT instead.
- **Interview cue:** "Distance between two tree nodes?" → "LCA, then
  $\text{depth}(u) + \text{depth}(v) - 2\,\text{depth}(\text{LCA})$."
