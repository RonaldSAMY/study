## The hook: when the bottleneck is the disk head, not the CPU

A binary search tree over a billion rows is about 30 levels deep. In RAM that's nothing. But if each node
lives on a spinning disk or an SSD page, *each level is a separate read* — and a disk read is roughly a
million times slower than a comparison. The fix is to stop thinking in binary. Pack hundreds of keys into
one fat node sized to a disk block, and the same billion rows become a tree only **three or four levels
deep**. That is a B-tree, and it is the data structure inside essentially every database index and file
system on Earth.

<Callout type="intuition" title="The one-sentence idea">
Let each node hold many keys (a whole disk block's worth) so the tree is wide and shallow, then keep it
balanced by splitting any node that overflows and pushing its median key up to the parent.
</Callout>

## Intuition: trade depth for width

A red-black tree minimizes *comparisons*. A B-tree minimizes *block transfers*. The trick is the
branching factor. In a binary tree each node sends you down one of two ways, so depth is $\log_2 n$. In a
B-tree of order $m$, each node sends you down one of up to $m$ ways, so depth is $\log_m n$ — and with
$m$ in the hundreds, that logarithm is tiny.

Balance is maintained from the **bottom up by splitting**. Insertion always lands in a leaf. If that leaf
is already full, it splits in two and its middle key floats up into the parent. If the parent is now full,
*it* splits too, and the split can ripple all the way to the root — which is the only way the tree ever
grows a new level. Because growth happens at the root, **all leaves always stay at the same depth**. That
is the B-tree's balance invariant, and it is gloriously simple compared with red-black recoloring.

## How it works

Insertion is "search down, split on the way, place in a leaf":

```
1. If the root is full, split it first (this is the only step that adds a level).
2. Walk down toward the correct leaf.
3. Before descending into any FULL child, split that child and lift its median into the current node.
4. Insert the key into the (now guaranteed non-full) leaf, in sorted order.
```

Splitting a node with `maxKeys = m - 1` keys takes its middle key out, leaves the lower half behind,
makes a new sibling from the upper half, and inserts the middle key into the parent. Because we split
*proactively* on the way down, the parent is always guaranteed to have room — no need to back up.

  let i = node.keys.length - 1;

    while (i >= 0 && key < node.keys[i]) i--;

## The real implementation

The course file is a full B-tree with search, insert (with split), delete (with borrow and merge), and
range queries — plus a sketch of the B+ tree variant that databases actually use. Here is the splitting
core that keeps it balanced:

```ts
// /dsa/20-advanced-trees/b-tree.ts
private splitChild(parent: BTreeNode<K, V>, index: number): void {
  const child = parent.children[index];
  const midIndex = Math.floor(this.maxKeys / 2);

  const sibling = new BTreeNode<K, V>(child.isLeaf);
  sibling.keys = child.keys.splice(midIndex + 1);        // upper half -> new sibling
  sibling.values = child.values.splice(midIndex + 1);
  if (!child.isLeaf) sibling.children = child.children.splice(midIndex + 1);

  const midKey = child.keys.pop()!;                       // median floats up
  const midValue = child.values.pop()!;
  parent.keys.splice(index, 0, midKey);
  parent.values.splice(index, 0, midValue);
  parent.children.splice(index + 1, 0, sibling);
}
```

## Complexity

With branching factor $m$ the height is $\log_m n$, and each node visited costs one block read, so the
number of disk I/Os — the metric that actually matters — is logarithmic in base $m$:

$$
h = O(\log_m n), \qquad \text{disk I/Os per operation} = O\!\left(\log_m n\right).
$$

| Operation | Comparisons | Disk reads |
|-----------|-------------|------------|
| Search | $O(\log_2 n)$ | $O(\log_m n)$ |
| Insert | $O(\log_2 n)$ | $O(\log_m n)$ |
| Range of $k$ results | — | $O(\log_m n + k/m)$ |

For $n = 10^9$ and $m = 256$, $\log_m n \approx 4$ — four reads to find any of a billion keys. That is
why your database does not slow to a crawl.

## The same idea, different worlds

<ExampleCard domain="Databases" icon="🗄️">
Every relational database — Postgres, MySQL, SQLite — stores its indexes as B-trees (usually the B+ tree
variant with all data in linked leaves). A <code>WHERE id = ?</code> lookup is a handful of B-tree page
reads, and a <code>BETWEEN</code> range scan walks the linked leaves.
</ExampleCard>

<ExampleCard domain="File systems" icon="💽">
NTFS, ext4, HFS+, and APFS organize directories and file metadata in B-trees (or B+ trees). Opening a
file in a folder with a million entries is fast because the directory is a shallow, block-sized tree.
</ExampleCard>

<ExampleCard domain="Geographic search" icon="🗺️">
Map and GIS systems index huge coordinate sets in B-tree-derived structures so a "find everything in this
bounding box" query touches only a few disk pages instead of scanning the planet.
</ExampleCard>

## Common pitfalls & interview tips

- **Splitting too late.** This implementation splits a full child *before* descending into it, which
  guarantees the parent always has room. Splitting only after you arrive forces awkward back-propagation.
- **Off-by-one on the median.** With an even maximum number of keys you must be consistent about which
  middle element goes up; mixing conventions corrupts the ordering.
- **Confusing B-tree with B+ tree.** In a plain B-tree, values live in internal nodes too. In a B+ tree
  (what databases use) all values live in the leaves, which are linked for fast range scans.
- **Interview cue:** "Why do databases use B-trees instead of binary search trees?" → "High fan-out makes
  the tree shallow, so the number of slow disk reads, not comparisons, is minimized."
