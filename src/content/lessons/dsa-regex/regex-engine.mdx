## The hook: the machine behind the pattern

You have been driving the engine; now open the hood. A regular expression is not run character-by-character
by ad-hoc rules — it is first **compiled into a graph of states**, a *nondeterministic finite automaton*
(NFA), and then that automaton is *simulated* over the input. Understanding this graph explains two things
at once: why a clean engine can match in guaranteed linear-ish time, and why the everyday backtracking
engine in most languages can grind to a halt for seconds on a 30-character string. The difference between
"fast forever" and "denial-of-service" is a choice about how you walk this graph.

<Callout type="intuition" title="The one-sentence idea">
A pattern compiles to an NFA — a graph of states joined by character edges and free epsilon edges — and
matching is just tracking the *set* of states you could be in as you read the input one character at a time.
</Callout>

## Intuition

**Thompson's construction** builds the NFA piece by piece. Each fragment has one entry and one exit state:

- a literal `a` is two states joined by an edge labelled `a`;
- **concatenation** `ab` wires the exit of `a` to the entry of `b` with a free **epsilon** (ε) edge;
- **alternation** `a|b` adds a fork state with ε-edges into both branches and a join state after;
- **star** `a*` wraps a fragment with ε-edges that let you skip it or loop back through it.

"Nondeterministic" means that from one state, an input character (or an ε-edge) might lead to *several*
states. Rather than guess and backtrack, the linear-time simulation keeps a **set** of all states you could
currently be in. Read a character: from every active state, follow matching character edges, then take the
**ε-closure** (everything reachable through free edges). The pattern matches if an *accept* state is in the
set once the input is consumed. Because the set never has more than $m$ states and you take one step per
input character, the whole thing is $O(n \cdot m)$ — no backtracking, ever.

Here is the crux: the NFA *is a graph*, and this simulation is breadth-first exploration of it. Everything
you learned about graph traversal applies directly — states are nodes, transitions are edges, ε-closure is
reachability.

## How it works

Simulating `(a|b)*c` over `"ac"` as a set of states:

```
start:      ε-closure of the start  → { s0, a-branch, b-branch, exit, c-in }
read 'a':   follow 'a' edges, ε-close → back to the loop states + c-in
read 'c':   follow 'c' edge          → { accept }
end:        accept ∈ active set       → MATCH
```

A **backtracking** engine instead tries one path at a time and rewinds on failure. On `(a+)+` with a string
of `a`s then a non-`a`, it explores every way to split the `a`s between the inner and outer `+`, which is
exponentially many. The NFA simulation cannot do this — it has no notion of "trying a path," only "the set
of reachable states." Compare the two engines directly:

<CodeRunner
  client:visible
  code={`// A minimal NFA simulator (Thompson): match a*b* style patterns
// as a SET of states, one pass, no backtracking.
function simulate(edges, start, accept, text) {
  // edges[state] = { eps:[...states], on:{char:state} }
  const closure = (set) => {
    const seen = new Set(set), stack = [...set];
    while (stack.length) {
      const s = stack.pop();
      for (const e of edges[s].eps) if (!seen.has(e)) { seen.add(e); stack.push(e); }
    }
    return seen;
  };
  let cur = closure([start]);
  for (const ch of text) {
    const next = new Set();
    for (const s of cur) { const to = edges[s].on[ch]; if (to !== undefined) next.add(to); }
    cur = closure([...next]);
  }
  return cur.has(accept);
}

// NFA for /a*b/ :  0 --a--> 0 (loop), 0 --b--> 1 (accept)
const edges = { 0: { eps: [], on: { a: 0, b: 1 } }, 1: { eps: [], on: {} } };`}

## The real implementation

The performance file spells out exactly which patterns detonate and how to defuse them — the practical
face of "backtracking versus automaton":

```ts
// /dsa/26-regular-expressions/regex-performance.ts
// ⚠️ Catastrophic backtracking: nested quantifiers over the same class.
//    /(a+)+$/, /(a*)*$/, /(a|a)*$/  → O(2^n) on non-matching input.

// A crude static check for the dangerous shape:
function isPotentiallyDangerous(pattern: string): boolean {
    const nestedQuantifiers = /(\+|\*|\{\d+,\}){2,}/;
    if (nestedQuantifiers.test(pattern)) return true;
    const groupQuantifier = /\([^)]*(\+|\*)\)(\+|\*)/;
    return groupQuantifier.test(pattern);
}

// The fix is almost always a NEGATED CLASS (no overlap = no backtracking):
const slowHTML = /<.*?>/g;      // backtracks
const fastHTML = /<[^>]*>/g;    // linear, atomic in effect

// ...and bounding quantifiers, anchoring, and preferring string methods.
```

## Complexity

The whole lesson is a complexity story. The NFA simulation carries at most $m$ states and advances once per
character, so it is linear in both; a backtracking engine can revisit the same positions exponentially often:

$$
T_\text{NFA}(n, m) = O(n \cdot m), \qquad T_\text{backtracking}^\text{worst}(n) = O(2^{\,n}).
$$

| Engine | Method | Worst case | Backreferences? |
|--------|--------|-----------|-----------------|
| NFA simulation (Thompson) | set of states, ε-closure | $O(n \cdot m)$ | no |
| DFA (subset construction) | one state per step | $O(n)$ match, $O(2^m)$ build | no |
| Backtracking (PCRE, JS) | try-and-rewind | $O(2^{\,n})$ | yes |

DFA-based tools like **RE2** and Go's `regexp` guarantee linear time by refusing backreferences and
lookarounds — a deliberate trade of expressive power for safety.

## The same idea, different worlds

<ExampleCard domain="Compilers" icon="⚙️">
Every lexer is an NFA/DFA over regular expressions: keywords, numbers, and identifiers are token patterns,
and tools like `lex`/`flex` compile them into a single state machine that tokenises source in one linear
pass — the same construction you just watched animate.
</ExampleCard>

<ExampleCard domain="Network security" icon="🛰️">
Intrusion-detection systems (Snort, Suricata) scan every packet against thousands of signatures. They lean
on automaton engines precisely because a backtracking pattern in a rule could be weaponised — a crafted
payload triggering catastrophic backtracking would take the sensor offline.
</ExampleCard>

<ExampleCard domain="Web platform safety" icon="🐌">
"ReDoS" outages are real: a single user-supplied string against a vulnerable `(a+)+`-style validator has
frozen production services for seconds per request. Cloudflare's 2019 global outage was, at its root, one
catastrophic-backtracking regex.
</ExampleCard>

## Common pitfalls & interview tips

- **NFA vs DFA vs backtracking are different machines.** An NFA tracks a set of states (linear to
  simulate); a DFA precomputes one state per input (linear to run, possibly exponential to build);
  backtracking explores paths and can blow up. Know which one your language ships.
- **Nested quantifiers are the classic ReDoS.** `(a+)+`, `(.*)*`, `(a|a)*` — any repetition of something
  itself repeatable, matched against input that *almost* matches. The demo makes the explosion visible.
- **You cannot build an NFA for backreferences.** `\1` is not a regular language, so linear-time engines
  drop it. If you need backreferences *and* safety, validate length/shape first, or switch algorithms.
- **The cure is structural, not cosmetic.** Replace overlapping alternation with a negated class, anchor
  the pattern, bound the quantifiers, and reach for a linear engine (RE2) on untrusted input.
- **Interview cue:** "make this regex safe / why is it slow?" wants the words *catastrophic backtracking*,
  *NFA*, and *negated class*. Sketch the state graph and you have demonstrated real depth.
