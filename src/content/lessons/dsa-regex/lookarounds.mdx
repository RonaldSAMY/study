## The hook: check the neighbours, take nothing

You want the number *before* `px` but not the `px` itself. You want the digits *after* a dollar sign but
not the sign. You want a password that *contains* an uppercase letter, a digit, and a symbol — without
saying anything about their order. All three need the same trick: a way to *assert a condition about the
surrounding text without consuming it*. That is a **lookaround** — a zero-width assertion that peeks ahead
or behind, returns yes or no, and leaves the cursor untouched.

<Callout type="intuition" title="The one-sentence idea">
A lookaround is a sub-pattern in `(?= )`, `(?! )`, `(?<= )` or `(?<! )` that must (or must not) match at
the current position but consumes no characters — it tests the neighbourhood and moves on.
</Callout>

## Intuition

There are four, split by direction and polarity:

- `(?=…)` **positive lookahead** — succeeds if the sub-pattern matches *starting here*.
- `(?!…)` **negative lookahead** — succeeds if it does *not*.
- `(?<=…)` **positive lookbehind** — succeeds if the sub-pattern matches *ending here* (just behind).
- `(?<!…)` **negative lookbehind** — succeeds if it does not.

The defining property is **zero width**. After a lookaround the cursor has not moved, so whatever it peeked
at is *not* part of the match. `\d+(?=px)` matches the digits in `10px` and returns `10`, not `10px` — the
`px` was inspected but left on the table. Lookbehind is the mirror image: `(?<=\$)\d+` grabs `100` from
`$100` without swallowing the sign.

Because they consume nothing, lookarounds compose beautifully. Password rules stack several independent
lookaheads at the very start of the pattern: `(?=.*[a-z])(?=.*\d)` each rewinds to the same spot and checks
one requirement, so together they mean "all of these are true, in any order."

## How it works

Dry-run `\d+(?=px)` on `"10px"`:

```
\d+     greedily matches "10"  (cursor now at index 2)
(?=px)  peek: do "px" match starting at 2? yes ✓
        cursor stays at 2 — "px" is NOT consumed
result  match = "10"  at [0, 2)
```

For a lookbehind the engine checks whether the sub-pattern matches text *ending* exactly at the current
position. Run a few, including a stacked-lookahead password check:

## The real implementation

From the course file — extraction with lookaround and the classic stacked-lookahead password validator:

```ts
// /dsa/26-regular-expressions/lookarounds.ts
function extractNumberWithUnit(text: string, unit: string): string | null {
    const m = text.match(new RegExp(`\\d+(?=${unit})`)); // number, unit excluded
    return m ? m[0] : null;
}
function extractPriceInDollars(text: string): string | null {
    const m = text.match(/(?<=\$)\d+(?:\.\d{2})?/);      // digits after $
    return m ? m[0] : null;
}

// Each lookahead checks one rule; none consume, so order is free.
const strongPasswordRegex =
    /^(?=.*[a-z])(?=.*[A-Z])(?=.*\d)(?=.*[@$!%*?&])[A-Za-z\d@$!%*?&]{8,}$/;

// Negative lookahead rejects reserved names.
function isValidUsername(username: string): boolean {
    return /^(?!admin|root|system)[a-z][a-z0-9_]{2,15}$/i.test(username);
}
```

## Complexity

A lookaround runs its own little match at the current position, so its cost is the cost of that
sub-pattern — but since it consumes nothing, a careless lookaround inside a loop can repeat work. For
simple assertions the overall scan stays linear:

$$
T(n) = O(n \cdot m), \qquad S = O(1).
$$

Stacked lookaheads that each start with `.*` re-scan the string once per rule, so $k$ rules cost about
$O(k \cdot n)$ — usually fine, but worth knowing when a validator feels slow.

| Assertion | Succeeds when | Consumes |
|-----------|---------------|----------|
| `(?=…)` | pattern matches ahead | nothing |
| `(?!…)` | pattern does *not* match ahead | nothing |
| `(?<=…)` | pattern matches just behind | nothing |
| `(?<!…)` | pattern does *not* match behind | nothing |

## The same idea, different worlds

<ExampleCard domain="Finance" icon="💵">
Pulling amounts from statements uses a lookbehind for the currency marker:
`(?<=\$)\d+(?:\.\d{2})?` returns `29.99` from a "Total" line while leaving the currency symbol behind, so
the captured value parses straight to a number.
</ExampleCard>

<ExampleCard domain="Account security" icon="🛡️">
Password policies are pure stacked lookahead:
`(?=.*[a-z])(?=.*[A-Z])(?=.*\d)(?=.*[@$!%*?&])`. Each zero-width check enforces one requirement
independently of the others, which is why the rules can appear in any order in the input.
</ExampleCard>

<ExampleCard domain="Number formatting" icon="🔢">
Inserting thousands separators uses a lookahead that counts groups of three digits:
`\d(?=(\d{3})+$)` matches each position followed by a multiple of three digits, so a single
`replace` turns `1234567` into `1,234,567`.
</ExampleCard>

<ExampleCard domain="Data privacy" icon="🔒">
Redaction keeps a hint while hiding the rest: a negative lookahead like `\d(?!\d{0,3}$)` masks
every digit of a card *except* the last four, so logs can show a suffix for support without leaking the
number.
</ExampleCard>

## Common pitfalls & interview tips

- **Lookarounds match a position, not text.** They never appear in the captured result. If you find
  yourself wanting the peeked text *in* the match, you wanted a normal group, not a lookaround.
- **Lookbehind support was historically spotty.** Modern JavaScript (ES2018+) supports it, but older
  runtimes and some other tools do not. Check your target before relying on `(?<=…)`.
- **Negative lookahead is "unless."** `q(?!u)` matches a `q` *not* followed by `u`. Reading it as "q then
  not-u" is wrong — nothing after the `q` is consumed; it is a veto, not a token.
- **Stacked lookaheads with `.*` re-scan.** Fine for short inputs, but for very long strings prefer a
  single pass or bounded lookaheads to avoid $O(k \cdot n)$ work.
- **Interview cue:** "match X but only when preceded/followed by Y — and don't include Y" is the textbook
  lookaround prompt. Emphasise the words "zero width" and you have signalled you understand the mechanism.
