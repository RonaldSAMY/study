## The hook: "one or more, but how many?"

A phone number is *some* digits. A tag is `<` then *stuff* then `>`. "Some" and "stuff" are repetition,
and repetition is where regex gets both powerful and slippery. A **quantifier** attaches to the token
before it and says how many times it may repeat. The twist: when a quantifier *could* match different
amounts, it must decide — grab as much as possible, or as little? That single choice, **greedy** versus
**lazy**, decides whether `<div>a</div><div>b</div>` comes back as one giant match or two neat ones.

<Callout type="intuition" title="The one-sentence idea">
A quantifier repeats the preceding token; by default it is greedy (takes the most it can, then hands
characters back only when forced), and adding `?` makes it lazy (takes the fewest, then grabs more only
when forced).

</Callout>

## Intuition

The four core quantifiers:

- `*` — zero or more (`ab*` matches `a`, `ab`, `abbb`).
- `+` — one or more (`ab+` needs at least one `b`).
- `?` — zero or one, i.e. optional (`colou?r` matches both spellings).
- `{n}`, `{n,}`, `{n,m}` — exact, at-least, and ranged counts (`\d{3}`, `\d{2,}`, `\d{3,5}`).

By default every one of these is **greedy**: it first consumes as many repetitions as it can, then, if the
rest of the pattern cannot match, it **backtracks** — releasing one character at a time until the rest fits
or it drops below its minimum. Add a `?` after the quantifier to make it **lazy**: now it takes the
*fewest* repetitions and only grabs more when the rest of the pattern demands it. Same characters accepted;
completely different amount consumed.

This is your first real taste of **backtracking**, the engine's undo-and-retry mechanism. Watching a
greedy `.*` grab the whole line and then reluctantly give characters back is the single most clarifying
thing you can see in regex.

## Play with it

Compare `a.*c` (greedy) with `a.*?c` (lazy) on the same text. Watch the greedy version swallow everything
after `a` (sky), hit the end, and then *release* characters (amber) until a `c` lines up — that releasing
is backtracking. The lazy version takes the opposite tack: match `c` as soon as possible.

## How it works

Dry-run greedy `a.*c` on `"abcayc"`:

```
a      matches 'a' at 0
.*     greedily grabs "bcayc" to the end (pos 6)
c      needs a 'c' at pos 6 — end of string ✗
.*     backtracks: give back 'c' (pos 5)
c      'c'? char at 5 is 'c' ✓  → match "abcayc" [0,6)
```

The lazy `a.*?c` instead expands from empty: match `a`, try `c` immediately (fails on `b`), grab one, try
`c` again, … stopping at the first `c` for the match `abc`. Run both and a few counted quantifiers:

## The real implementation

From the course file — counted quantifiers for exact formats, and the greedy/lazy pair that separates
"one big match" from "many small ones":

```ts
// /dsa/26-regular-expressions/quantifiers-anchors.ts
function isValidPhone(phone: string): boolean {
    return /^\d{10}$/.test(phone);               // exactly ten digits
}
function isValidUsernameLength(u: string): boolean {
    return /^[a-zA-Z0-9_]{3,16}$/.test(u);       // between 3 and 16
}
function isValidZipCode(zip: string): boolean {
    return /^\d{5}(-\d{4})?$/.test(zip);         // optional +4 suffix
}

// Greedy vs lazy — the classic <div> example:
const html = "<div>Hello</div><div>World</div>";
html.match(/<div>.*<\/div>/);    // ONE match: the whole string (greedy)
html.match(/<div>.*?<\/div>/g);  // TWO matches: each <div>…</div> (lazy)

// Lazy extraction of quoted text:
function extractQuoted(text: string): string[] {
    return (text.match(/".*?"/g) || []).map(m => m.slice(1, -1));
}
```

## Complexity

A single quantifier over a simple token is still linear — the engine may backtrack, but only within that
one repetition. The danger is *nested* quantifiers, where backtracking multiplies:

$$
T_\text{simple}(n) = O(n), \qquad T_\text{nested-worst}(n) = O(2^{\,n}).
$$

That exponential is catastrophic backtracking; the automaton you meet in the final lesson matches in
$O(n \cdot m)$ with no backtracking at all.

| Pattern | Behaviour | Consumes |
|---------|-----------|----------|
| `.*` | greedy | as much as possible, then backtracks |
| `.*?` | lazy | as little as possible, then grows |
| `\d{3}` | exact | precisely 3 |
| `\d{2,}` | at least | 2 or more (greedy) |
| `(a+)+` | nested | ⚠️ exponential — avoid |

## Common pitfalls & interview tips

- **Greedy by default bites HTML.** `<.*>` on `<a><b>` matches the whole thing. Use `<.*?>` or, better,
  the negated class `<[^>]*>` which cannot overshoot in the first place.
- **Lazy is not "correct," just different.** Both greedy and lazy find *a* match if one exists; they only
  differ in *which*. Choose by what you want to capture, not by superstition.
- **`*` and `?` can match nothing.** `a*` succeeds on `"xyz"` by matching zero `a`s. If you need at least
  one, use `+`; if you forgot anchors, an empty match can hide bugs.
- **Nested quantifiers are a ReDoS landmine.** `(a+)+`, `(a*)*`, `(a|a)*` make the engine try exponentially
  many splits on non-matching input. Prefer a single quantifier and a negated class; the last lesson shows
  the blow-up live.
- **Interview cue:** if asked to "extract each tag / each quote / each field," reach for a lazy quantifier
  or a negated class, and be ready to explain greedy backtracking on a whiteboard.

## Practice

1. **Easy:** What does `ab*` match that `ab+` does not?
2. **Easy:** Write a pattern for a 4-to-6 digit PIN using a counted quantifier.
3. **Medium:** On the text `"'a' and 'b'"`, why does `'.*'` capture the whole thing while `'.*?'` captures
   just `'a'`? Trace the backtracking.
4. **Medium:** Rewrite the greedy `<.*>` as a negated-class pattern that matches a single tag and cannot
   overshoot.
5. **Hard:** Explain, step by step, why matching `(a+)+$` against `"aaaaaaaaaa!"` is slow, and how many
   ways the engine can split ten `a`s among the repetitions.
