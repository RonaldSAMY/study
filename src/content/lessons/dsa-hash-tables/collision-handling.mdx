## The hook: two coats, one hook

The cloakroom ticket for "Dolores" and the one for "Bernard" happen to compute to the *same* hook. Now
what? You cannot refuse the coat. You have exactly two sane options: hang both on that hook (one behind
the other) and check each when someone returns, or walk down the rack to the next empty hook and use
that instead. Those two instincts are the only two collision strategies that matter:
**separate chaining** and **open addressing**.

<Callout type="intuition" title="The one-sentence idea">
A collision is two keys hashing to the same index; you either store a list of entries inside that bucket
(chaining) or probe forward to the next empty slot in the array (open addressing).
</Callout>

## Intuition: collisions are guaranteed, so plan for them

Collisions are not a bug — they are arithmetic. By the **pigeonhole principle**, once you have more keys
than buckets, two must share. And it happens far sooner than you would guess: by the **birthday paradox**,
just 23 keys give a coin-flip chance of a clash among 365 buckets. The collision probability is roughly
$n^2 / 2m$. So the question is never *if* but *how you cope*.

**Separate chaining.** Each bucket holds a linked list. On a collision you append to the list; on lookup
you scan the (short) list. Simple, tolerates load factors above 1, and deletion is trivial — but it
costs pointer memory and scatters data across the heap.

**Open addressing.** Every entry lives in the array itself. On a collision you **probe** for another
slot. With *linear probing* you try $h, h+1, h+2, \dots$ (wrapping around) until you find an empty one:

$$
\text{slot}_i = (h(k) + i) \bmod m, \qquad i = 0, 1, 2, \dots
$$

Cache-friendly and pointer-free, but it suffers **clustering** (filled runs that grow and merge) and
deletion needs a *tombstone* marker so probe chains do not break.

## How it works

Insert keys that collide at bucket 3, into a table of size 7.

```
Chaining (bucket 3 holds a list):
  bucket 3:  dog → cat → owl        (each new collision appended)
  get("cat"): hash→3, scan the list, found.

Linear probing (scan forward):
  hash("dog")=3 → slot 3 free      → place at 3
  hash("cat")=3 → slot 3 taken     → probe 4 (free) → place at 4
  hash("owl")=3 → slot 3,4 taken   → probe 5 (free) → place at 5
  get("owl"): hash→3, probe 3,4,5 until match.
```

Both find every key; they differ in *where* the extra entries live — inside the bucket, or spilled into
neighbouring slots.

function makeProbeTable(capacity = 7) {
  const slots = new Array(capacity).fill(null);
  const hash = (key) => {
    let h = 0;
    for (let i = 0; i < key.length; i++) h = (h * 31 + key.charCodeAt(i)) % capacity;
    return h;

  function set(key) {
    let i = hash(key), steps = 0;
    while (slots[i] !== null && steps < capacity) { i = (i + 1) % capacity; steps++; }
    slots[i] = key; probes[key] = steps;  // steps = how far we probed
  }
  return { set, slots, probes, hash };
}

## The real implementation

From the course file — chaining inserts at the head of a linked list; linear probing scans forward and
deletes with a tombstone so it never severs a probe chain:

```ts
// /dsa/08-hash-table/collision-handling.ts

// Separate chaining: prepend to the bucket's linked list
set(key: K, value: V): void {
    const index = this.getIndex(key);
    let current = this.buckets[index];
    while (current) {                       // update if key already present
        if (current.key === key) { current.value = value; return; }
        current = current.next;
    }
    this.buckets[index] = { key, value, next: this.buckets[index] }; // new head
    this._size++;
}

// Linear probing: scan h, h+1, h+2 … and tombstone on delete
delete(key: string): boolean {
    let index = this.hash(key);
    while (this.keys[index] !== null) {
        if (this.keys[index] === key) {
            this.keys[index] = this.DELETED;  // tombstone, not null
            this.values[index] = undefined;
            this._size--;
            return true;
        }
        index = (index + 1) % this.capacity;
    }
    return false;
}
```

## Complexity

With load factor $\lambda$ and a good hash, the expected probe/scan length stays small:

$$
\text{chaining: } O(1 + \lambda), \qquad \text{linear probing: } O\!\left(\frac{1}{1 - \lambda}\right) \text{ (expected)}.
$$

Chaining degrades gently as $\lambda$ grows; open addressing degrades sharply as $\lambda \to 1$ — which is
why probing tables resize at a *lower* threshold (often 0.5–0.7).

| Strategy | Avg lookup | Worst | Memory | Deletion |
|----------|-----------|-------|--------|----------|
| Separate chaining | $O(1 + \lambda)$ | $O(n)$ | extra pointers | easy |
| Linear probing | $O\!\left(\tfrac{1}{1-\lambda}\right)$ | $O(n)$ | cache-friendly | tombstones |
| Quadratic / double hashing | better spread | $O(n)$ | cache-friendly | tombstones |

Both share the same $O(n)$ worst case: an adversary (or an awful hash) that forces every key into one
chain or one cluster.

## Common pitfalls & interview tips

- **Deleting by setting a slot to null breaks open addressing.** A `null` in the middle of a probe chain
  makes later keys unreachable. Use a `DELETED` tombstone and skip it while probing.
- **Primary clustering bites linear probing.** Long filled runs merge and slow everything; quadratic probing
  or double hashing spreads keys out to fix it.
- **Open addressing needs headroom.** Performance falls off a cliff near $\lambda = 1$, so keep these tables
  under ~0.7 full and resize early.
- **Chaining can degrade too.** A bad hash funnels everything into one bucket, turning the table into a single
  linked list — $O(n)$ again. Good hash + resize still matter.
- **Interview cue:** "chaining vs open addressing?" → chaining for high/variable load and easy deletes; open
  addressing for cache performance and tight memory. Mention tombstones to score points.
