## The hook: the cloakroom that grows itself

Back at the coat-check. The hash function tells the attendant *which hook* to use — but what happens
when the rack fills up and three coats keep landing on the same hook? Service slows to a crawl. A smart
cloakroom watches how full it is and, once it passes a threshold, wheels in a bigger rack and
re-distributes every coat. A hash table does exactly this, and the threshold has a name: the
**load factor**.

<Callout type="intuition" title="The one-sentence idea">
A hash table is an array of buckets indexed by a hash function; it stays fast by keeping the load factor
(items ÷ buckets) low, doubling its size and rehashing everything when it gets too crowded.
</Callout>

## Intuition: function + array + a growth rule

You already have the hash function. A hash table adds two things around it:

- **An array of buckets.** `set(key, value)` computes `index = hash(key) % capacity` and stores the pair there.
  `get(key)` recomputes the same index and reads it back. Same key, same index — that is the whole trick.
- **A growth rule.** Define the **load factor**

$$
\lambda = \frac{n}{m},
$$

the number of entries $n$ over the number of buckets $m$. As $\lambda$ rises, buckets hold more than one
item and operations slow down. So when $\lambda$ crosses a threshold (commonly $0.75$), the table
**doubles its capacity and rehashes** every key into the larger array. Bigger $m$ means smaller
$\lambda$, and speed is restored.

Resizing is the price you pay for staying fast on average — and because it only happens occasionally, that
price is tiny when spread out (we will quantify this under Complexity).

## How it works

Insert `apple, banana, cherry, date` into a table that starts at capacity 4 with threshold 0.75:

```
cap=4  set apple   → bucket 1   size=1  λ=0.25
cap=4  set banana  → bucket 3   size=2  λ=0.50
cap=4  set cherry  → bucket 3   size=3  λ=0.75
cap=4  set date    → bucket 0   size=4  λ=1.00 > 0.75
       RESIZE → cap=8, rehash all 4 keys into the bigger table
```

The fourth insert pushes $\lambda$ to 1.0, tripping the threshold, so the table grows to 8 and every key
is re-placed using `% 8` instead of `% 4`.

function makeTable(capacity = 4) {
  let buckets = Array.from({ length: capacity }, () => []);
  let size = 0;
  const hash = (key) => {
    let h = 0;
    for (let i = 0; i < key.length; i++) h = (h * 31 + key.charCodeAt(i)) % buckets.length;
    return h;
  };
  const resize = () => {
    const old = buckets;
    buckets = Array.from({ length: old.length * 2 }, () => []);
    size = 0;
    for (const b of old) for (const [k, v] of b) set(k, v);
  };
  function set(key, value) {
    if (size / buckets.length > 0.75) resize();
    const b = buckets[hash(key)];
    const found = b.find((e) => e[0] === key);
    if (found) { found[1] = value; return; }
    b.push([key, value]); size++;
  }
  const get = (key) => {
    const e = buckets[hash(key)].find((e) => e[0] === key);
    return e ? e[1] : undefined;
  };
  return { set, get, stats: () => ({ size, capacity: buckets.length }) };
}

## The real implementation

From the course file — the `set` path with its load-factor check and the `resize` that rehashes
everything. Prime capacities (53, then the next prime after doubling) improve distribution:

```ts
// /dsa/08-hash-table/hash-table.ts
set(key: string, value: V): void {
    if (this._size / this.capacity >= this.loadFactorThreshold) { // 0.75
        this.resize();
    }
    const index = this.hash(key);
    if (!this.buckets[index]) this.buckets[index] = [];
    const bucket = this.buckets[index]!;
    for (let i = 0; i < bucket.length; i++) {
        if (bucket[i][0] === key) { bucket[i][1] = value; return; } // update
    }
    bucket.push([key, value]);  // insert
    this._size++;
}

private resize(): void {
    const oldBuckets = this.buckets;
    this.capacity = this.nextPrime(this.capacity * 2);
    this.buckets = new Array(this.capacity);
    this._size = 0;
    for (const bucket of oldBuckets) {
        if (bucket) for (const [key, value] of bucket) this.set(key, value); // rehash
    }
}
```

## Complexity

With a good hash and a bounded load factor, each bucket holds a constant number of items on average:

$$
T_{\text{avg}}(\text{get, set, delete}) = O(1), \qquad T_{\text{worst}} = O(n).
$$

A single resize copies all $n$ entries, costing $O(n)$ — but it only happens after the table has doubled,
so those costs spread out to $O(1)$ **amortized** per insert.

| Operation | Average | Worst case |
|-----------|---------|------------|
| get / has | $O(1)$ | $O(n)$ |
| set (insert) | $O(1)$ amortized | $O(n)$ (on a resize) |
| delete | $O(1)$ | $O(n)$ |
| space | — | $O(n)$ |

The worst case ($O(n)$) happens when every key collides into one bucket — which is exactly the problem
the next lesson tackles.

## The same idea, different worlds

<ExampleCard domain="Web infrastructure" icon="🗄️">
A cache like Redis or Memcached is a giant hash table: it hashes a request key to a slot and returns the
stored response in one hop, turning a slow database query into a microsecond lookup.
</ExampleCard>

<ExampleCard domain="Compilers" icon="⚙️">
A compiler's symbol table hashes each variable name to its type and scope. Every time the parser meets
an identifier, one hash lookup answers "have I seen this name, and what is it?"
</ExampleCard>

<ExampleCard domain="Biology" icon="🧬">
Genome search tools (BLAST and friends) index DNA by hashing short k-mers — fixed-length substrings — so
a query sequence finds its candidate matches by bucket lookup instead of scanning billions of bases.
</ExampleCard>

## Common pitfalls & interview tips

- **Resizing is mandatory, not cosmetic.** Skip it and the load factor climbs, buckets grow, and your
  "O(1)" table quietly becomes a linked-list scan.
- **You must rehash on resize.** Indices are computed `% capacity`; once capacity changes, old indices are
  wrong. Copying buckets verbatim corrupts the table.
- **Insert must check for an existing key.** Otherwise `set(k, 1)` then `set(k, 2)` leaves two entries and
  `get(k)` is ambiguous.
- **"O(1)" is average and amortized, not guaranteed.** Say this in interviews. A single insert can trigger an
  $O(n)$ resize, and adversarial keys can force $O(n)$ lookups.
- **Interview cue:** when asked "how does a hash table stay fast as it grows?", answer "load factor + resize +
  rehash" — that triplet is the whole story.
