## The hook: "find me more like this"

Spotify suggests a song that *feels* like the one you're playing. A support bot finds the help article
that *means* the same thing as your question, even with zero words in common. Google Photos groups every
picture of your dog without you tagging a thing. None of these match keywords — they compare **meaning**.
The trick is to turn every item into a point in a high-dimensional space so that similar items land near
each other. Then "find me more like this" becomes a geometry problem: **find the nearest points**.

<Callout type="intuition" title="The one-sentence idea">
An embedding maps each item to a vector so that semantic similarity becomes geometric closeness, and
search becomes a nearest-neighbor lookup — usually by cosine similarity.
</Callout>

## Intuition: direction is meaning

An embedding model (word2vec, a sentence transformer, CLIP) is trained so that related items point in
similar directions. The magic property is that **directions carry meaning**: the classic
$\text{king} - \text{man} + \text{woman} \approx \text{queen}$ works because gender and royalty are
consistent directions in the space.

To compare two vectors we usually use **cosine similarity** — the cosine of the angle between them:

$$
\cos(\mathbf{a}, \mathbf{b}) = \frac{\mathbf{a} \cdot \mathbf{b}}{\lVert \mathbf{a}\rVert\, \lVert \mathbf{b}\rVert}.
$$

Cosine ignores length and looks only at *direction*, so a long document and a short query can still match
if they're about the same thing. A score near 1 means "very similar"; near 0 means "unrelated." Searching
is then: embed the query, score it against every item, and return the top-$k$.

## How it works: the dry run

Score the query against each item, then sort:

```
query q, items A B C:
  cos(q, A) = 0.98   ← tiny angle, most similar
  cos(q, B) = 0.10   ← nearly perpendicular
  cos(q, C) = -0.85  ← opposite direction

  sort descending → [A, B, C];  top-1 = A
```

  let dot = 0, na = 0, nb = 0;

## The real implementation

The course file indexes documents by their embedding, then scores a query against all of them —
brute-force nearest neighbor:

```ts
// /dsa/27-ml-algorithms/09-embeddings/sentence-embeddings.ts
search(query: string, topK = 5): Array<{ text: string; score: number }> {
  const q = this.encoder.encode(query);
  return this.documents
    .map((doc) => ({ text: doc.text, score: this.cosineSimilarity(q, doc.embedding) }))
    .sort((a, b) => b.score - a.score)   // rank by similarity
    .slice(0, topK);                     // return the nearest
}
```

Run it with `npx tsx 27-ml-algorithms/09-embeddings/sentence-embeddings.ts`. The sibling
`word2vec.ts` trains the embeddings themselves, and `contrastive-learning.ts` shows the InfoNCE loss that
pulls matching pairs together and pushes mismatches apart (the recipe behind CLIP).

## Complexity

A brute-force scan compares the query to every item:

$$
T_{\text{search}} = O(N d), \qquad T_{\text{one comparison}} = O(d),
$$

for $N$ items of dimension $d$.

| Approach | Query time | Trade-off |
|----------|-----------|-----------|
| Brute force | $O(N d)$ | Exact, but slow at scale |
| LSH | ~$O(d)$ expected | Approximate, hashes similar items together |
| HNSW graph | ~$O(d \log N)$ | Approximate, state-of-the-art recall/speed |

<Callout type="why">
At billions of vectors, $O(Nd)$ per query is far too slow, so production systems use an
<strong>approximate nearest-neighbor index</strong>. Locality-sensitive hashing buckets similar vectors
together so a query only scans a handful of candidates instead of the whole database.
</Callout>

## Common pitfalls & interview tips

- **Forgetting to normalize.** If you use cosine similarity, either normalize vectors up front or divide
  by the norms — raw dot products let long vectors dominate.
- **Wrong metric.** Cosine (angle) vs Euclidean (distance) vs dot product can give different rankings;
  match the metric to how the embeddings were trained.
- **Brute force at scale.** Fine for thousands of items, hopeless for billions — reach for an ANN index
  (LSH, HNSW, IVF).
- **Interview cue:** "How would you build semantic search?" Embed items and query with the same model,
  store vectors in an ANN index, retrieve top-$k$ by cosine similarity.
