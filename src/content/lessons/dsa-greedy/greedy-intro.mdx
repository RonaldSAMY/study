## The hook: paying with the biggest coins first

A cashier handing back change does not solve an optimization problem in their head. They reach for
the largest coin that still fits, then the next largest, and so on. For real-world coins this simple
habit gives the fewest coins every time. That instinct — **take the biggest bite you can right now,
then repeat** — is the whole of greedy algorithm design. The hard part is not the habit; it is knowing
when the habit is actually right.

<Callout type="intuition" title="The one-sentence idea">
A greedy algorithm builds a solution one step at a time, always taking the choice that looks best at
this instant, and never reconsidering — which is optimal exactly when each locally best choice is safe.
</Callout>

## Intuition: when is "best right now" also "best overall"?

In the Longest Increasing Subsequence lesson you saw dynamic programming weigh *every* option before
committing. Greedy is the opposite temperament: it commits immediately and moves on. That makes it
faster and simpler, but it only finds the true optimum when two properties hold.

1. **Greedy-choice property** — there is always an optimal solution that *contains* the greedy choice.
   Picking it first never paints you into a corner.
2. **Optimal substructure** — after making that choice, what remains is a smaller version of the same
   problem, and solving it optimally finishes the job.

The standard way to *prove* the greedy-choice property is the **exchange argument**. Take any optimal
solution that does not start with the greedy choice. Swap its first choice for the greedy one and show
the solution is no worse. Since you can always exchange your way toward the greedy choice without losing
value, some optimal solution must begin with it. When that swap fails — when exchanging makes things
worse — greedy is the wrong tool and you fall back to dynamic programming.

## How it works

Making change for `63` with `25, 10, 5, 1`, greedy peels off the biggest coin each round:

```
remaining 63 → take 25 → 38
remaining 38 → take 25 → 13
remaining 13 → take 10 → 3
remaining  3 → take  1 →  2
remaining  2 → take  1 →  1
remaining  1 → take  1 →  0
result: 25, 25, 10, 1, 1, 1  → 6 coins (optimal)
```

Each step shrinks the problem to a smaller identical one (make change for the remainder), and for this
coin system the big-coin choice is always safe. Compare greedy against a dynamic-programming minimum to
see where the two agree and where they part ways:

function greedyCoins(amount, coins) {
  const sorted = [...coins].sort((a, b) => b - a);
  let count = 0;
  for (const c of sorted) { count += Math.floor(amount / c); amount %= c; }
  return amount === 0 ? count : -1;
}

// DP: the true minimum number of coins (always correct).
function dpCoins(amount, coins) {
  const dp = new Array(amount + 1).fill(Infinity);
  dp[0] = 0;
  for (let i = 1; i <= amount; i++)
    for (const c of coins) if (c <= i) dp[i] = Math.min(dp[i], dp[i - c] + 1);

}`}

## The real implementation

From the course file — the greedy coin maker beside the DP that proves greed can be wrong:

```ts
// /dsa/14-greedy/greedy-basics.ts
function greedyMinCoins(amount: number, coins: number[]): number {
    const sorted = [...coins].sort((a, b) => b - a);
    let count = 0;
    for (const coin of sorted) {
        count += Math.floor(amount / coin);   // take as many of the biggest as fit
        amount %= coin;                        // then move to the next size down
    }
    return amount === 0 ? count : -1;
}

function dpMinCoins(amount: number, coins: number[]): number {
    const dp = new Array(amount + 1).fill(Infinity);
    dp[0] = 0;
    for (let i = 1; i <= amount; i++) {
        for (const coin of coins) {
            if (coin <= i && dp[i - coin] + 1 < dp[i]) {
                dp[i] = dp[i - coin] + 1;      // consider EVERY coin, not just the biggest
            }
        }
    }
    return dp[amount] === Infinity ? -1 : dp[amount];
}
```

Run the full file: `npx tsx 14-greedy/greedy-basics.ts`.

## Complexity

A greedy pass usually costs a sort plus a single sweep, which is why it is so attractive:

$$
T(n) = O(n \log n), \qquad S(n) = O(1).
$$

| Approach | Time | Space | Always optimal? |
|----------|------|-------|-----------------|
| Greedy | $O(n \log n)$ | $O(1)$ | Only with the greedy-choice property |
| Dynamic programming | $O(n \cdot W)$ | $O(W)$ | Yes |
| Brute force | $O(2^n)$ | $O(n)$ | Yes |

The trade is stark: greedy is dramatically cheaper, but you must *earn* its correctness with a proof.

## The same idea, different worlds

<ExampleCard domain="Networking" icon="📶">
Dijkstra's shortest-path algorithm is greedy: it permanently settles the closest unvisited node at each
step. With non-negative edge weights that local choice is provably safe, so the first time you reach a
node you already have its shortest distance.
</ExampleCard>

<ExampleCard domain="Finance" icon="💸">
A cash machine dispensing the fewest notes uses greedy coin change on banknote denominations. Because
note systems are designed to be "canonical," grabbing the largest note that fits is always optimal — a
property the bank relies on without re-deriving it each time.
</ExampleCard>

<ExampleCard domain="Climate" icon="🌦️">
Allocating a fixed carbon budget by funding the cheapest tonnes of abatement first is a greedy sweep
over a "marginal abatement cost curve." It maximizes emissions cut per dollar — the same value-density
logic behind fractional knapsack, which is coming up next.
</ExampleCard>

## Common pitfalls & interview tips

- **Greedy is a claim that needs proof.** The most common interview mistake is asserting a greedy rule
  works without an exchange argument. If you cannot argue it, look for a counterexample.
- **Non-canonical coins break greedy.** Coins `1, 3, 4` and amount `6` give greedy three coins (`4+1+1`)
  but two suffice (`3+3`). The big-coin choice is no longer safe, so dynamic programming is required.
- **"Maximize value, not count" is a red flag.** Activity selection by count is greedy; the moment items
  carry different weights or profits, greedy often fails and you need DP (you will see this contrast in
  fractional vs 0/1 knapsack).
- **Interview cue:** if the problem says "minimum coins / maximum non-overlapping / cheapest first," try
  greedy first — it is fast to sketch and you can fall back to DP if a counterexample appears.

## Practice

1. **Easy:** Make change for `30` with coins `25, 10, 5, 1` by hand using the greedy rule. How many coins?
2. **Easy:** Find an amount under `10` where greedy on coins `1, 5, 6, 9` uses more coins than necessary.
3. **Medium:** Argue with an exchange argument why greedy is optimal for the coin set `1, 5, 10, 25`.
4. **Medium:** Given costs and an unlimited budget, sketch a greedy rule for "buy as many distinct items
   as possible" and prove it sorts by price ascending.
5. **Hard:** Decide whether greedy "always send the closest courier" minimizes total delivery time, or
   construct a counterexample where it does not.
