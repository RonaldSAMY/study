## The hook: cramming the most talks into one room

A conference has one room and a pile of talks, each with a fixed start and end time. You want to host as
*many* talks as possible — every talk counts equally, you just want the biggest headcount of sessions.
Which do you accept? The surprising answer is to ignore start times entirely and obsess over **finish
times**: always accept the talk that ends soonest among those that still fit. Finishing early frees the
room sooner, leaving the most runway for everything after.

<Callout type="intuition" title="The one-sentence idea">
To maximize the number of non-overlapping activities, repeatedly pick the one that finishes earliest and
is still compatible — finishing soonest leaves the most room for the rest.
</Callout>

## Intuition: why finish time, not start time or length?

From the greedy intro you know a greedy rule is only as good as its safety proof. Here the greedy choice
is "the compatible activity with the earliest finish time," and the **exchange argument** makes it
airtight.

Suppose some optimal schedule does not start with the earliest-finishing activity $g$. Its first
activity $x$ must finish no earlier than $g$ does. Swap $x$ out and $g$ in. Because $g$ ends at or before
$x$ ended, $g$ cannot overlap anything that came after $x$ — so the schedule stays valid and keeps the
same count. We just converted an optimal schedule into one that begins with the greedy choice without
losing anything. Therefore picking the earliest finisher first is always safe, and by optimal
substructure the same logic repeats on what remains.

Tempting alternatives fail: picking the **shortest** activity can wedge two long ones apart, and picking
the **earliest start** can grab a single marathon talk that blocks the whole day.

## How it works

With activities sorted by finish time, sweep once and keep an activity whenever its start is at or after
the last kept finish:

```
sorted by end:  B(1-4) C(3-5) D(5-7) A(0-6) F(5-9) E(3-9) G(6-10) H(8-11)
keep B → lastEnd = 4
C starts 3 < 4 → skip
D starts 5 ≥ 4 → keep, lastEnd = 7
A,F,E start < 7 → skip
G starts 6 < 7 → skip
H starts 8 ≥ 7 → keep, lastEnd = 11
kept: B, D, H  → 3 activities
```

## The real implementation

From the course file — sort by end time, then a single linear sweep:

```ts
// /dsa/14-greedy/activity-selection.ts
function activitySelection<T extends Activity>(activities: T[]): T[] {
    if (activities.length === 0) return [];

    // Sort by end time - THE KEY INSIGHT!
    const sorted = [...activities].sort((a, b) => a.end - b.end);

    const selected: T[] = [sorted[0]];
    let lastEnd = sorted[0].end;

    for (let i = 1; i < sorted.length; i++) {
        if (sorted[i].start >= lastEnd) {   // compatible with the last pick?
            selected.push(sorted[i]);
            lastEnd = sorted[i].end;        // commit to its finish time
        }
    }
    return selected;
}
```

Run the full file: `npx tsx 14-greedy/activity-selection.ts`.

## Complexity

The sort dominates; the sweep is linear and uses a single variable:

$$
T(n) = O(n \log n), \qquad S(n) = O(1).
$$

| Step | Cost |
|------|------|
| Sort by finish time | $O(n \log n)$ |
| Single greedy sweep | $O(n)$ |
| Extra space (just `lastEnd`) | $O(1)$ |

If the activities arrive already sorted by finish time, the whole thing collapses to a single $O(n)$ pass.

## Common pitfalls & interview tips

- **Sorting by the wrong key.** Sorting by start time, by duration, or by "least conflicting" all fail.
  Only **finish time** carries the exchange-argument guarantee.
- **Equal counts, unequal value.** This greedy maximizes *count*. If activities carry different profits,
  it can be badly suboptimal — that is weighted interval scheduling and it needs DP.
- **Half-open vs closed intervals.** Decide whether an activity ending exactly when the next begins is a
  conflict. Using `start >= lastEnd` treats touching intervals as compatible; be explicit in interviews.
- **Same problem, different phrasing.** "Minimum intervals to remove to make the rest non-overlapping" is
  just `n` minus the number you can keep — recognize the disguise.

## Practice

1. **Easy:** From `(1-3), (2-5), (4-7), (6-8)` select the most non-overlapping activities by hand.
2. **Easy:** Show that sorting by shortest duration first fails on `(0-5), (4-6), (5-10)`.
3. **Medium:** Modify the sweep to also return how many activities were *rejected*.
4. **Medium:** LeetCode 452 — given balloon intervals, find the minimum number of arrows to burst them all
   (hint: sort by end, shoot at each end).
5. **Hard:** Extend the algorithm to `k` rooms so you can run `k` activities at once, maximizing total count.
