## The hook: a cheat sheet for "what's the smallest in this stretch?"

You have a fixed array of elevations and thousands of questions of the form "what is the lowest point
between mile 12 and mile 47?" Scanning each range is $O(n)$ per query — death by a thousand loops. A
segment tree gets you to $O(\log n)$, but you can do better when the array never changes. The trick:
precompute the minimum of every block whose length is a **power of two**. Then *any* range, however
awkward, is covered by just **two** of those blocks that may overlap in the middle — and because taking
a minimum twice over the same element changes nothing, the overlap is harmless. Two lookups, one
`min`, done in $O(1)$.

<Callout type="intuition" title="The one-sentence idea">
Precompute the answer for every length-$2^j$ block by doubling, then cover any query range with two
overlapping power-of-two blocks — overlap is free because $\min$ is idempotent.

</Callout>

## Intuition: build by doubling, query by overlapping

The table `table[j][i]` holds the minimum of the block starting at $i$ with length $2^j$. Level 0 is the
array itself (blocks of length 1). To build level $j$, glue two level-$(j-1)$ blocks side by side:
`table[j][i] = min(table[j-1][i], table[j-1][i + 2^{j-1}])`. Each level doubles the reach, so after
$\log n$ levels you have every power-of-two block precomputed — $O(n \log n)$ work and space.

Querying a range $[l, r]$ of length $L$ uses the largest power of two that fits, $k = \lfloor \log_2 L
\rfloor$. One block of length $2^k$ starts at $l$; another ends at $r$ (it starts at $r - 2^k + 1$).
Since $2 \cdot 2^k \ge L$, those two blocks **together cover the whole range**, overlapping in the
middle. For a min (or max, gcd, AND, OR) the overlap double-counts some elements — and that is fine,
because $\min(x, x) = x$. This is the crucial difference from a **segment tree**: a segment tree handles
updates and any associative operation, but a sparse table needs a *static* array and an **idempotent**
operation to get its $O(1)$ punchline.

## How it works

For `[5, 2, 7, 4, 8, 1, 6, 3]`, building the min table by doubling:

```
j=0 (len 1): [5, 2, 7, 4, 8, 1, 6, 3]
j=1 (len 2): [2, 2, 4, 4, 1, 1, 3]          min of each adjacent pair
j=2 (len 4): [2, 2, 1, 1, 1]                 min of two len-2 blocks
j=3 (len 8): [1]                             whole array

query min(1, 6):  L = 6, k = floor(log2 6) = 2  -> blocks of length 4
  block A = table[2][1]            = 2   (covers 1..4)
  block B = table[2][6 - 4 + 1=3]  = 1   (covers 3..6)
  answer  = min(2, 1) = 1          (blocks overlap on 3..4 — harmless)
```

The whole thing, runnable:

## The real implementation

The course file precomputes a `log` lookup table so the query never calls `Math.log`, and the doubling
build is exactly the recurrence above:

```ts
// /dsa/21-advanced-data-structures/sparse-table.ts
private buildTable(arr: number[]): number[][] {
  const k = this.log[this.n] + 1;
  const table = Array.from({ length: k }, () => Array(this.n).fill(0));
  for (let i = 0; i < this.n; i++) table[0][i] = arr[i];          // length-1 blocks
  for (let j = 1; j < k; j++)
    for (let i = 0; i + (1 << j) <= this.n; i++)
      table[j][i] = Math.min(table[j-1][i], table[j-1][i + (1 << (j-1))]); // double
  return table;
}

query(l: number, r: number): number {
  const j = this.log[r - l + 1];                                  // largest power of two ≤ len
  return Math.min(this.table[j][l], this.table[j][r - (1 << j) + 1]); // two blocks, O(1)
}
```

Run the full file: `npx tsx 21-advanced-data-structures/sparse-table.ts`.

## Complexity

Build fills $O(n \log n)$ cells; each query is two array reads and one combine:

$$
T_\text{build} = O(n \log n), \qquad T_\text{query} = O(1), \qquad S = O(n \log n).
$$

| Structure | Preprocess | Query | Updates? | Operation needs |
|-----------|-----------|-------|----------|-----------------|
| Prefix sums | $O(n)$ | $O(1)$ | no | invertible (sum) |
| **Sparse table** | $O(n \log n)$ | $O(1)$ | no | **idempotent** (min/max/gcd) |
| Segment tree | $O(n)$ | $O(\log n)$ | **yes** $O(\log n)$ | associative |

If the array is static and the operation is idempotent, the sparse table's $O(1)$ query is unbeatable;
the moment you need updates, reach for a [segment tree](/lessons/dsa-advanced-algorithms/segment-tree)
instead.

## Common pitfalls & interview tips

- **Idempotency is mandatory for $O(1)$.** Min, max, gcd, lcm, bitwise AND/OR survive overlap. **Sum and
  XOR do not** — overlap double-counts, so for those a sparse table needs *disjoint* blocks and becomes
  $O(\log n)$ per query (a "disjoint sparse table" restores $O(1)$ for those).
- **Static only.** One changed element invalidates up to $O(n \log n)$ cells. If the data updates, this
  is the wrong tool.
- **Precompute the log table.** Calling `Math.log2` per query is slow and float-imprecise; the
  `log[i] = log[i>>1] + 1` recurrence gives exact integer logs in $O(n)$.
- **Watch the second index.** The right block starts at `r - (1 << j) + 1`, not `r - (1 << j)`; an
  off-by-one here is the most common bug.

## Practice

1. **Easy:** For `[5, 2, 7, 4, 8, 1, 6, 3]`, which $k$ and which two blocks answer `min(2, 7)`?
2. **Easy:** Why is `min` safe under overlapping blocks but `sum` is not? Give a tiny counterexample.
3. **Medium:** Build a `SparseTableMax` and a `SparseTableGCD` from the same doubling recurrence.
4. **Medium:** Return the *index* of the minimum in a range (store positions, compare by value) — useful
   for LCA.
5. **Hard:** Implement a 2D sparse table answering submatrix-minimum queries in $O(1)$ after
   $O(nm \log n \log m)$ preprocessing.
