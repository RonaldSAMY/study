## The hook: an undo button that costs almost nothing

You want unlimited undo, time-travel queries, or a lock-free snapshot readers can trust — in short, you
want every past version of a structure to stay accessible forever. The brute-force answer is to copy the
whole thing on every edit: $O(n)$ time and space per update, which collapses under a million edits. But
look at what one update actually *changes* in a balanced tree: a single leaf, and the handful of nodes
on the path from the root down to it. Everything hanging off to the side is **identical** to the
previous version. So don't copy it — **share** it. Copy only the $O(\log n)$ nodes on the path, point
their off-path children at the *old* subtrees, and you have a brand-new root for the new version while
the old root still describes the old version, intact.

<Callout type="intuition" title="The one-sentence idea">
To edit a tree without destroying its past, copy only the root-to-leaf path you touch and let the new
nodes point back into the unchanged old subtrees — $O(\log n)$ new nodes buy a whole new version.

</Callout>

## Intuition: path copying, structural sharing

Call this **path copying**. An update at some leaf proceeds top-down: to make a new version's root you
must make a new root node, because its child on the path changed. That new root keeps one pointer aimed
at the *old, shared* subtree (the side you did not descend into) and one pointer aimed at a freshly
copied child on the path. Recurse: each node on the path is copied, keeps its off-path child shared, and
re-copies its on-path child — until you reach the leaf, which you copy and overwrite with the new value.

The result is two roots that coexist. The old root reaches the old data; the new root reaches the new
value through freshly minted nodes but reuses every untouched subtree by pointer. Because a balanced tree
is $O(\log n)$ deep, each version costs $O(\log n)$ new nodes — so $m$ updates use $O(n + m \log n)$ total
space, not $O(nm)$. Crucially this demands **immutability**: shared nodes must never be mutated in place,
or you would silently corrupt every version that points at them.

## How it works

Updating index 2 of `[1, 2, 3, 4, 5]` in a persistent segment tree copies the root, one child, and the
target leaf; the sibling subtree is shared by pointer:

```
version 0 (sums):            update index 2 -> 10, build version 1
        [15]                         [22]  <- new root
       /    \                       /    \
    [6]      [9]                 [13]     [9]      <- [9] SHARED with v0
   /   \    /   \               /   \      |
 [3]   [3] [4] [5]            [3]  [10]  (old [9] subtree, untouched)
                              |     |
                          [3](shared) [10] new leaf

only 3 new nodes; v0's root [15] still answers the old sums
```

  if (l === r) return { sum: val, left: null, right: null };  // new leaf
  const m = (l + r) >> 1;

    const nl = update(node.left, l, m, i, val);               // copy left path
    return { sum: nl.sum + node.right.sum, left: nl, right: node.right }; // SHARE right

}

## The real implementation

The course file's persistent segment tree never touches an existing node: `updateNode` returns a **new**
node on the path and hands the off-path child straight through, so old roots stay valid forever:

```ts
// /dsa/21-advanced-data-structures/persistent-data-structures.ts
private updateNode(
  node: PersistentSegmentNode,
  l: number, r: number,
  index: number, value: number
): PersistentSegmentNode {
  if (l === r) {
    return new PersistentSegmentNode(value);                 // fresh leaf
  }
  const mid = Math.floor((l + r) / 2);
  if (index <= mid) {
    const newLeft = this.updateNode(node.left!, l, mid, index, value);
    return new PersistentSegmentNode(newLeft.sum + node.right!.sum, newLeft, node.right); // share right
  } else {
    const newRight = this.updateNode(node.right!, mid + 1, r, index, value);
    return new PersistentSegmentNode(node.left!.sum + newRight.sum, node.left, newRight); // share left
  }
}
```

Each call to `update` pushes a new root onto `roots`, so `query(version, ...)` can read any past snapshot.
Run the full file: `npx tsx 21-advanced-data-structures/persistent-data-structures.ts`.

## Complexity

Let $n$ be the number of elements and $m$ the number of updates. One update rebuilds one root-to-leaf
path and shares the rest:

$$
T_\text{update} = O(\log n), \qquad S_\text{update} = O(\log n), \qquad S_\text{total} = O(n + m \log n).
$$

| Strategy | Update time | Extra space / update | Old versions? |
|----------|-------------|----------------------|---------------|
| Copy-everything | $O(n)$ | $O(n)$ | yes (wasteful) |
| Mutate in place | $O(\log n)$ | $O(1)$ | **no** |
| **Path copying** | $O(\log n)$ | $O(\log n)$ | **yes** |

Path copying is the sweet spot: full persistence at only a logarithmic overhead per edit, because every
new version reuses almost the entire previous one.

## The same idea, different worlds

<ExampleCard domain="Functional programming" icon="λ">
Immutable maps and vectors in Clojure, Scala, and Haskell are persistent HAMTs / RRB-trees under the
hood. `assoc` returns a new collection that shares all untouched branches with the old one — which is why
"copying" an immutable map is cheap and old references stay valid.
</ExampleCard>

<ExampleCard domain="Version control" icon="🌱">
Git's object store is persistence made literal: a commit is a new root tree, but unchanged directories
point at the *same* tree objects as the parent commit. That structural sharing is why a commit stores a
diff-sized amount of data yet snapshots the whole repository.
</ExampleCard>

<ExampleCard domain="Databases" icon="🗄️">
MVCC engines and copy-on-write B-trees (LMDB, CouchDB) write a new root page per transaction while
sharing untouched pages, giving readers a consistent snapshot with no locks — long-running reads simply
hold an old root.
</ExampleCard>

## Common pitfalls & interview tips

- **Immutability is non-negotiable.** The instant you mutate a shared node in place, every version
  pointing at it changes underneath you. Persistent updates must *only ever* allocate new nodes.
- **Copy the path, share the rest.** The common bug is copying too much (whole subtrees) or too little
  (mutating an on-path node's shared sibling). Copy exactly the nodes on the root-to-leaf route.
- **Keep the roots.** Persistence is only useful if you retain a handle to each version's root — store
  them in an array or map, keyed by version number or timestamp.
- **Partial vs full persistence.** Partial persistence lets you *read* any version but *update* only the
  latest; full persistence lets you branch off any version. Path copying gives full persistence for free.
- **Garbage collection matters.** A version becomes collectable only when no root can still reach its
  nodes; in reference-counted or manual settings, dropping a root must not free shared nodes.

## Practice

1. **Easy:** In an 8-leaf balanced tree, how many nodes does a single leaf update copy? Why is it
   $\log_2 n + 1$?
2. **Easy:** Explain in one sentence why mutating a shared node in place breaks every earlier version.
3. **Medium:** Extend the persistent segment tree so each `update` also returns the number of newly
   allocated nodes; verify it is $\lceil \log_2 n \rceil + 1$.
4. **Medium:** Use a persistent segment tree of prefix versions to answer *k*-th smallest in a range
   $[l, r]$ (the classic "persistent segment tree on values" trick).
5. **Hard:** Implement a **fully** persistent balanced BST (persistent treap or persistent red-black
   tree) supporting insert and delete on any version.
