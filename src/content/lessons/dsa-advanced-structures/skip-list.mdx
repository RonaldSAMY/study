## The hook: express trains over a sorted list

A sorted linked list is honest but slow: to find a value you walk every node, one local stop at a
time, $O(n)$. Subway systems solved this physically — they run **express trains** that skip most
stations, and you only drop to the local line near your destination. A skip list builds those express
lines *on top of* an ordinary linked list. The top layer skips far; each layer below skips less; the
bottom layer is the full list. Search rides the expresses down to the right neighborhood in
$O(\log n)$ — and the whole thing is built with nothing fancier than a coin flip.

<Callout type="intuition" title="The one-sentence idea">
Stack extra forward pointers on a sorted linked list so search can take long hops on high layers and
short hops on low ones, dropping down a level whenever the next node would overshoot.
</Callout>

## Intuition: levels you build by flipping coins

Every node lives on level 0 (the full list). When you insert a node, you flip a coin: heads, it also
joins level 1; flip again, maybe level 2; and so on. Because each extra level halves how many nodes
reach it, roughly half the nodes are on level 1, a quarter on level 2, an eighth on level 3 — the same
geometric thinning that gives a balanced tree its $\log n$ height, but with **no rotations and no
rebalancing**. The randomness does the balancing for you, *in expectation*.

Searching is then a staircase: start at the head on the highest level, walk right while the next key is
still smaller than your target, and the moment it would overshoot, **drop down one level** and keep
going. Each drop lands you in a shorter stretch of the list, so you converge fast. This is the same
divide-and-narrow instinct behind binary search and the tree decompositions you have already met — only
here the structure is a humble linked list wearing express lanes.

## How it works

Search keeps a single rule at every level: *advance while the next key is strictly less than the
target; otherwise drop.* Here is the search for `60` over a perfect skip list, written so the express
hops are visible:

```
L3: HEAD ─────────────────────────────▶ 50 ──────────▶ NIL
L2: HEAD ─────────▶ 25 ───────────────▶ 50 ──────────▶ NIL
L1: HEAD ─▶ 12 ──▶ 25 ──▶ 38 ───────▶ 50 ─▶ 60 ─────▶ NIL
L0: 5 ▶ 12 ▶ 19 ▶ 25 ▶ 31 ▶ 38 ▶ 44 ▶ 50 ▶ 60 ▶ 72 ...

find(60): L3 head→50 (50<60, hop)         then 50's next is NIL, drop
          L2 from 50: next NIL, drop
          L1 from 50: 50→60  ✓ found
```

The express hop to `50` skipped seven local stops in one move. Below is the exact level-walk in plain
JavaScript — a deterministic "perfect" skip list so you can re-run it and trace the same path:

<CodeRunner

  code={`// A sorted skip list with deterministic levels (node i on level = trailing zeros of i+1).
function makeLevels(keys) {
  return keys.map((_, i) => { let p = i + 1, l = 0; while ((p & 1) === 0) { p >>= 1; l++; } return l; });
}
// search: walk right while next < target, else drop a level. Returns nodes visited.
function search(keys, target) {
  const level = makeLevels(keys);
  const maxL = Math.max(...level);
  let visited = 0, rank = -1; // -1 = head
  for (let L = maxL; L >= 0; L--) {
    let next = keys.findIndex((_, i) => i > rank && level[i] >= L);
    while (next !== -1 && keys[next] < target) {
      rank = next; visited++;
      next = keys.findIndex((_, i) => i > rank && level[i] >= L);
    }
    if (next !== -1 && keys[next] === target) return { found: true, visited: visited + 1 };
  }
  return { found: false, visited };
}`}

## The real implementation

The course file uses true randomized levels — a coin flip per node via `randomLevel` — and an `update[]`
array to remember where the search dropped on each level, so insert and delete can splice the node into
exactly those layers:

```ts
// /dsa/21-advanced-data-structures/skip-list.ts
private randomLevel(): number {
  let level = 0;
  while (Math.random() < this.probability && level < this.maxLevel) level++;
  return level;                                  // ~half stop at 0, a quarter at 1, ...
}

get(key: K): V | undefined {
  let current = this.header;
  for (let i = this.level; i >= 0; i--) {        // top lane down to level 0
    while (current.forward[i] !== null &&
           this.compare(current.forward[i]!.key, key) < 0) {
      current = current.forward[i]!;             // hop right while still smaller
    }                                            // overshoot? fall through to next i (drop)
  }
  current = current.forward[0]!;                 // step onto the candidate at level 0
  return current !== null && this.compare(current.key, key) === 0
    ? current.value : undefined;
}
```

## Complexity

With probability $p$ per level, a node's height is geometric, the expected number of levels is
$O(\log_{1/p} n)$, and the expected work per level is $O(1/(1-p))$, so:

$$
\mathbb{E}[T_\text{search}] = O\!\left(\frac{\log n}{p\,(1-p)^{-1}}\right) = O(\log n), \qquad
\mathbb{E}[S] = O(n).
$$

| Operation | Skip list (expected) | Skip list (worst) | Balanced BST |
|-----------|----------------------|-------------------|--------------|
| Search    | $O(\log n)$          | $O(n)$            | $O(\log n)$  |
| Insert / delete | $O(\log n)$    | $O(n)$            | $O(\log n)$  |
| Min / next | $O(1)$              | $O(1)$            | $O(1)$–$O(\log n)$ |

The worst case is a degenerate run of unlucky coin flips, but its probability is astronomically small;
in practice a skip list matches a balanced tree while being far simpler to code and to make concurrent.

## The same idea, different worlds

<ExampleCard domain="Databases" icon="🗄️">
Redis stores its sorted sets (`ZSET`) as a skip list, so range queries like "give me ranks 10–20 by
score" and rank lookups run in $O(\log n)$ without the rotation bookkeeping a balanced tree demands.
</ExampleCard>

<ExampleCard domain="Transit" icon="🚆">
Express and local subway lines are a physical skip list: ride the express (high level) across town,
then transfer down to the local (level 0) for the last few stops — exactly drop-when-you-overshoot.
</ExampleCard>

<ExampleCard domain="Concurrency" icon="🧵">
Java's `ConcurrentSkipListMap` is the go-to lock-free ordered map. Skip lists are easy to make
concurrent because an insert only re-points a few forward pointers — there is no whole-subtree rotation
to coordinate across threads.
</ExampleCard>

<ExampleCard domain="Memory tables" icon="📒">
LSM-tree storage engines keep their in-memory write buffer (the memtable) as a skip list, so writes and
ordered flushes to disk stay $O(\log n)$ while many threads append at once.
</ExampleCard>

## Common pitfalls & interview tips

- **The `update[]` array is the whole trick for insert/delete.** As search drops on each level, record
  the node you dropped *from*; splicing in (or out) only needs those remembered predecessors.
- **Levels are capped.** A `maxLevel` (often $\lceil \log_{1/p} n \rceil$) stops a freak run of heads
  from building a useless skyscraper of pointers above a tiny list.
- **It is probabilistic, not adversarial-proof.** With user-controlled keys an attacker cannot force
  bad behavior (levels come from your RNG, not the keys), but the worst case is still $O(n)$ — say
  "expected $O(\log n)$" in interviews.
- **Don't forget $O(1)$ min.** The level-0 head's first forward pointer is always the minimum; range
  scans just walk level 0 from there.

## Practice

1. **Easy:** On the perfect skip list above, list every node the search for `38` touches.
2. **Easy:** If $p = 0.5$, what fraction of nodes do you expect on level 2? On level 3?
3. **Medium:** Implement `insert` using an `update[]` array: search, record predecessors, flip for a
   random level, then splice the new node into levels 0..L.
4. **Medium:** Add a `range(low, high)` that returns all keys in $[low, high]$ in $O(\log n + k)$.
5. **Hard:** Augment each forward pointer with a `span` (how many level-0 nodes it skips) to support
   `getByRank(i)` and `getRank(key)` in $O(\log n)$ — an *indexed* skip list.
