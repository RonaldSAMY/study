## The hook: editing a megabyte without shifting a megabyte

Insert one character into the middle of a flat string and, under the hood, every character after it
slides down one slot — $O(n)$ work for a one-key press. A text editor holding a large file and doing this
on every keystroke would grind to a halt. The rope's answer is to stop storing the text as one
contiguous array and instead break it into small **chunks** hung on the leaves of a **balanced binary
tree**. Concatenation becomes "make a new node with these two subtrees as children" — $O(1)$-ish instead
of copying everything. Insert and delete reduce to **split** and **concat**, each of which walks a single
root-to-leaf path. Editing anywhere costs $O(\log n)$, and no bulk character-shifting ever happens.

<Callout type="intuition" title="The one-sentence idea">
Store a string as a balanced tree of small chunks with each node caching its left subtree's length, so
edits walk one $O(\log n)$ path instead of shifting an $O(n)$ array of characters.

</Callout>

## Intuition: weights turn a tree into an addressable string

Each internal node stores a **weight**: the total number of characters in its **left** subtree. That
single number is enough to navigate by position. To find character $i$, compare $i$ to the weight: if
$i <$ weight the character lives in the left subtree, so recurse left; otherwise it is in the right
subtree at offset $i -$ weight, so recurse right subtracting the weight as you go. Leaves hold the actual
substring chunks, so you fall out of the tree at the right one in $O(\log n)$.

Every operation is a variation on that walk. **Concat** just builds a new root whose left is one rope and
right is the other, with weight set to the left rope's length. **Split at $i$** walks the path to
position $i$, snipping the tree into everything-before and everything-after and re-joining the loose ends
with concats along the way — $O(\log n)$ nodes touched, the bulk of the text shared, not copied.
**Insert** is split-then-concat-concat; **delete** is split-split-concat. And because subtrees are
shared rather than mutated, ropes marry beautifully with the persistence idea from the last lesson:
undo/redo is just keeping old roots around.

## How it works

Indexing `"Hello_World"` stored as a rope: descend, comparing the position to each node's weight and
subtracting when you go right, until you land in a leaf chunk:

```
                [6]                 weight 6 = length of "Hello_"
              /     \
          [3]        [2]
         /   \       /  \
    "Hel" "lo_"  "Wor" "ld"

charAt(8):  root weight 6, 8 >= 6 -> go RIGHT, k = 8 - 6 = 2
            node weight 3, 2 <  3 -> go LEFT
            leaf "Wor", offset 2  -> 'r'
```

A rope with weight-guided index and an $O(\log n)$ split, runnable:

## The real implementation

The course file's `splitNode` is the heart of the rope: it compares the index to the node's weight,
recurses into exactly one child, and re-joins the leftover with a concat — an $O(\log n)$ path walk that
leaves the rest of the tree shared:

```ts
// /dsa/21-advanced-data-structures/rope.ts
private splitNode(
  node: RopeNode | null,
  index: number
): [RopeNode | null, RopeNode | null] {
  if (!node) return [null, null];
  if (node.isLeaf()) {                                   // cut inside a chunk
    if (index <= 0) return [null, node];
    if (index >= node.data!.length) return [node, null];
    const leftLeaf = new RopeNode(node.data!.substring(0, index));
    const rightLeaf = new RopeNode(node.data!.substring(index));
    return [leftLeaf, rightLeaf];
  }
  if (index < node.weight) {                             // cut is on the left
    const [leftL, leftR] = this.splitNode(node.left, index);
    const newRight = this.concatNodes(leftR, node.right); // rejoin loose end
    return [leftL, newRight];
  } else if (index > node.weight) {                      // cut is on the right
    const [rightL, rightR] = this.splitNode(node.right, index - node.weight);
    const newLeft = this.concatNodes(node.left, rightL);
    return [newLeft, rightR];
  } else {                                               // clean split at the seam
    return [node.left, node.right];
  }
}
```

Insert is then just `split` then two `concat`s. Run the full file: `npx tsx 21-advanced-data-structures/rope.ts`.

## Complexity

Let $n$ be the total string length. Every core operation walks a path of a balanced tree:

$$
T_\text{index} = T_\text{split} = T_\text{concat} = T_\text{insert} = T_\text{delete} = O(\log n),
$$

versus $O(n)$ for the middle-of-string operations on a flat array.

| Operation | Rope | Flat string |
|-----------|------|-------------|
| Index (charAt) | $O(\log n)$ | $O(1)$ |
| Concat | $O(\log n)$ | $O(n)$ |
| Split | $O(\log n)$ | $O(n)$ |
| Insert / Delete in middle | $O(\log n)$ | $O(n)$ |

The trade is deliberate: a rope gives up $O(1)$ random indexing to make every *edit* logarithmic — a
great deal when edits dominate, a bad one when you mostly scan characters. Balance matters: an unbalanced
rope degrades to $O(n)$, so implementations rebalance (often toward Fibonacci-height bounds).

## The same idea, different worlds

<ExampleCard domain="Text editors" icon="📝">
Editors and editor cores (early Emacs, Xi, many IDE buffers) store the document as a rope or the closely
related piece-table / gap-buffer family, so inserting a character in a huge file is $O(\log n)$ rather
than shifting megabytes on every keystroke.
</ExampleCard>

<ExampleCard domain="Collaborative editing" icon="👥">
Because rope subtrees are shared and cheap to snapshot, ropes underpin undo/redo and pair naturally with
CRDT/OT engines in real-time collaborative editors, where many versions of a document coexist.
</ExampleCard>

<ExampleCard domain="Immutable strings" icon="🧵">
JVM ropes and similar "cord" libraries represent very large or frequently concatenated strings as trees
so that building a big document from many pieces avoids repeated $O(n)$ copies.
</ExampleCard>

## Common pitfalls & interview tips

- **Weight is the *left* subtree's length only** — not the whole node's length. Getting this wrong sends
  the index walk down the wrong branch every time.
- **Balance is everything.** Naive concats grow a spindly right-leaning tree that degrades to $O(n)$.
  Real ropes rebalance (e.g. rebuild when depth exceeds a Fibonacci bound) to keep height $O(\log n)$.
- **Leaf size is a tuning knob.** Tiny leaves waste memory on pointers; huge leaves reintroduce $O(n)$
  copies inside a chunk. Editors pick a leaf chunk in the hundreds-to-thousands of bytes.
- **Index vs edit is the real trade-off.** If the workload is mostly random `charAt`, a flat array's
  $O(1)$ indexing wins. Reach for a rope only when inserts/deletes/concats dominate.
- **Insert and delete are just split + concat.** Deriving them live in an interview from `split` and
  `concat` is cleaner and less bug-prone than writing bespoke traversals.

## Practice

1. **Easy:** For the rope over `"Hello_World"` above, trace `charAt(3)` and `charAt(6)` step by step.
2. **Easy:** Why is `insert(i, s)` equal to `split(i)` followed by two concats? Write the three-line
   recipe.
3. **Medium:** Implement `delete(start, end)` using two splits and one concat, and show it is
   $O(\log n)$.
4. **Medium:** Add a `report(i, j)` that returns the substring in $[i, j)$ by walking only the overlapping
   leaves — analyze its cost in terms of the number of leaves spanned.
5. **Hard:** Make the rope **persistent** (share subtrees, never mutate) so every edit yields a new
   version, giving $O(\log n)$ undo/redo — tying back to the previous lesson.
