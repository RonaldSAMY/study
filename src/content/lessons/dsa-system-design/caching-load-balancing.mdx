## The hook: the same viral photo, ten million times

A photo goes viral. Without help, every one of ten million views becomes a database read for the *same*
bytes — a self-inflicted denial-of-service. Two ideas prevent it. First, **cache** the photo so the second
through ten-millionth reads never touch the database: a cache read is under a millisecond, a database read
is 10–100 ms. Second, **load-balance** the requests across a fleet of servers so no single machine melts.
The subtle part is the glue: when you spread cached data across many cache servers and the fleet grows or
shrinks, you do *not* want every key to suddenly land on a different server. That is the problem consistent
hashing was invented to solve.

<Callout type="intuition" title="The one-sentence idea">
A cache keeps hot data one hop away instead of on disk; a load balancer fans requests across a fleet; and
consistent hashing lets you add or remove servers while remapping only about 1/n of the keys.
</Callout>

## Intuition: absorb reads, then spread them

Start with the **80/20 rule** you know from everywhere else: a small slice of data serves most of the
traffic. A cache exploits exactly that — store the hot 20% in fast memory and you can absorb 80–99% of
reads before they reach the database. Good caches routinely hit 95%+ hit rates; an e-commerce catalog can
reach 99.9%.

But a cache raises three questions that recur through the whole field:

- **What do you evict when it fills?** The classic answer is **LRU** (least-recently-used): drop whatever
  has gone longest untouched. It runs in $O(1)$ using a hash map plus a doubly linked list — the same
  structure you met in [hash tables](/lessons/dsa-hash-tables/collision-handling).
- **How do writes stay consistent?** *Write-through* updates cache and database together (safe, slower);
  *write-back* updates the cache now and the database later (fast, risky); *cache-aside* lets the app load
  on a miss and populate the cache itself.
- **How stale is too stale?** A **TTL** trades freshness for hit rate — seconds for prices, an hour for
  descriptions.

Once one cache server is not enough, you shard the cache across many — and now you need a rule mapping each
key to a server. Naive `hash(key) % n` looks fine until you add a server: **n** changes, and nearly every
key remaps at once, stampeding the database. **Consistent hashing** fixes this by placing servers *and*
keys on a ring; a key belongs to the first server clockwise, so adding a server only steals one arc.

Meanwhile, at the front door, a **load balancer** picks which server handles each request: **round-robin**
cycles through them, **least-connections** favors the least-busy, **weighted** variants respect bigger
boxes, and **IP-hash** or consistent hashing pin a client to a server for stickiness.

## How it works: routing keys on the ring

A key's home is found by hashing it to a position, then binary-searching for the first server position at
or after it (wrapping around if you run off the end). Adding a server inserts one new position; only the
keys between it and the previous server change hands.

      for (const n of ring) if (n.pos >= p) return n.id;

## The real implementation

The course ships a production-flavored ring with virtual nodes, binary search, and replication:

```ts
// /dsa/system-design/algorithms/03-load-balancing/consistent-hashing.ts
class ConsistentHashRing {
  private ring: VirtualNode[] = []; // sorted by hash

  addNode(node: PhysicalNode): void {
    // give each physical node many positions so keys spread evenly
    const vnodeCount = Math.floor(this.virtualNodesCount * (node.weight || 1));
    for (let i = 0; i < vnodeCount; i++) {
      const hash = this.hashFunction(`${node.id}:vnode:${i}`);
      this.insertSorted({ hash, physicalNodeId: node.id, virtualIndex: i });
    }
  }

  getNode(key: string): PhysicalNode | null {
    const hash = this.hashFunction(key);
    const index = this.findNodeIndex(hash);      // binary search: first vnode >= hash
    return this.nodes.get(this.ring[index].physicalNodeId) || null;
  }
}
```

And the two everyday load-balancing algorithms, verbatim in spirit:

```ts
// /dsa/system-design/algorithms/03-load-balancing/round-robin.ts — cycle with wraparound
getNextServer() {
  const server = this.servers[this.currentIndex];
  this.currentIndex = (this.currentIndex + 1) % this.servers.length;
  return server.healthy ? server : /* skip to next healthy */ null;
}

// /dsa/system-design/algorithms/03-load-balancing/least-connections.ts — pick the least busy
getNextServer() {
  let min = null, best = Infinity;
  for (const s of this.servers) if (s.healthy && s.activeConnections < best) { best = s.activeConnections; min = s; }
  if (min) min.activeConnections++;
  return min;
}
```

## Numbers that decide the design

$$
\text{keys remapped when adding a server} \approx \frac{1}{n}, \qquad \text{hit rate} = \frac{\text{hits}}{\text{hits} + \text{misses}}.
$$

| Concern | Cheap tier | Costly tier | Ratio |
|---------|------------|-------------|-------|
| Read latency | cache < 1 ms | database 10–100 ms | ~10–100× |
| DB load with 95% hit rate | 5% of reads reach DB | — | 20× fewer |
| Modulo rehash on scale | — | ~100% keys move | worst case |
| Consistent-hash rehash | ~1/n keys move | — | best case |
| Load-balancer choice | round-robin $O(1)$ | least-conn $O(n)$ | per pick |

A cache with a 95% hit rate cuts database read traffic by 20×; push it to 99% and it is 100×. That single
lever is often the difference between one database and a painful sharding project.

## The same idea, different worlds

<ExampleCard domain="Music streaming" icon="🎧">
Spotify caches the audio for popular tracks on edge servers near listeners. The hot 1% of songs drive most
plays, so a small cache serves the vast majority of streams — and consistent hashing routes each track ID
to a stable cache node even as the edge fleet autoscales through the day.
</ExampleCard>

<ExampleCard domain="Multiplayer gaming" icon="🎮">
A game backend uses IP-hash or consistent hashing to pin every player in a match to the *same* game server,
so their state stays in one place. Least-connections then spreads brand-new matches toward the emptiest
boxes, keeping any one server from tipping over at peak.
</ExampleCard>

<ExampleCard domain="News at breaking-story scale" icon="📰">
When a story breaks, one article gets hammered. A CDN plus an application cache serve it from memory, so the
origin sees a handful of requests instead of millions — but editors set a short TTL so a correction
propagates in seconds, the classic freshness-versus-hit-rate trade-off.
</ExampleCard>

## Common pitfalls & interview tips

- **The thundering herd.** When a hot key's TTL expires, thousands of requests miss at once and stampede the
  database. Fix with request coalescing (single-flight): let one request refill while the rest wait.
- **Naive `hash % n`.** It is fine for a fixed pool, catastrophic for an elastic one — every scaling event
  reshuffles the whole cache. Reach for the ring the moment servers come and go.
- **Skipping virtual nodes.** A few single-position servers distribute unevenly (one may own 80% of the
  ring). Give each server 100–200 virtual nodes.
- **Least-connections without health checks.** Sending traffic to the "least busy" server is a trap if that
  server is least busy because it is *dead*. Always health-check first.
- **Caching writes indiscriminately.** Write-heavy or rarely-read data pollutes the cache. Cache what reads
  often and updates rarely; consider write-around for the rest.
- **Interview cue:** name L4 vs L7 load balancing (L4 routes on IP/port, fast; L7 reads URLs/headers, smart)
  and say *why* consistent hashing beats modulo — "only 1/n keys move."

## Practice

1. **Easy:** A cache serves a 90% hit rate on 1,000,000 reads. How many reach the database?
2. **Easy:** With round-robin over 4 healthy servers, which server handles the 9th request if the 1st went to server 0?
3. **Medium:** On a ring with servers at 30°, 150°, 270°, which server owns a key hashing to 200°? To 300°?
4. **Medium:** Explain why adding one server to a 4-server modulo scheme (`% 4` → `% 5`) can move ~80% of keys.
5. **Hard:** Design the cache + load-balancing tier for a product page hit 200,000 times/second, 99% cacheable.
   Pick eviction policy, write policy, TTL, and the routing algorithm — and justify each with a number.
