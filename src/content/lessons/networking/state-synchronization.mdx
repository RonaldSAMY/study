## The hook: how do 100 players share one world?

Every smoothing trick so far — prediction, interpolation, rewind — runs on a steady diet of **snapshots** of
the game state. The authoritative server holds the one true world and must keep dozens or hundreds of machines
agreeing about it, many times a second. The naive plan is to mail everyone the entire world on every tick. With
a few entities that's fine. With hundreds of players, projectiles, and pickups, it's a bandwidth firehose
nobody can afford. The way out is simple and powerful: most of the world doesn't change each tick, so **send
only what did**.

<Callout type="intuition" title="The one-sentence idea">
**State synchronization** keeps every machine's copy of the world in agreement by streaming snapshots; **delta
compression** sends only the entities that changed since a known baseline instead of the whole world every time.
</Callout>

## Intuition: don't resend what hasn't moved

You met the **client-server** model: one authoritative server owns the truth and broadcasts it. The question is
*how* to broadcast efficiently.

- **Full snapshot.** Every tick, serialize and send all $N$ entities. Dead simple and robust — a fresh arrival
  is instantly correct — but the cost is fixed at "the whole world" no matter how little happened.
- **Delta.** Send only the entities that differ from a baseline the receiver already has. If just five of two
  hundred entities moved, you ship five, not two hundred.

Deltas need a shared reference point. The receiver acknowledges the last snapshot it fully holds, and the server
diffs against *that*. To stay robust against dropped packets, the server periodically sends a full **keyframe**:
a complete snapshot that re-grounds everyone, so a momentary loss can't corrupt the world forever.

There is a second axis of savings hiding in each entity: you rarely need full precision. A position can be
**quantized** to a few bits, an angle to a single byte. Delta compression decides *which* entities to send;
quantization shrinks *each* one. Together they turn the firehose into a trickle.

## The formal definition

With $N$ entities at $B$ bytes each, a **full snapshot** costs

$$
S_{\text{full}} = H + N\,B,
$$

for a small header $H$. If a fraction $c$ of entities change in a tick, a **delta** costs

$$
S_{\Delta} = H + cN\,(B + b_{\text{id}}),
$$

where $b_{\text{id}}$ is the per-entity index needed to say *which* ones changed. The fractional bandwidth saved
is

$$
\text{saved} = 1 - \frac{S_{\Delta}}{S_{\text{full}}} \approx 1 - c\left(1 + \frac{b_{\text{id}}}{B}\right),
$$

so low churn $c$ means big savings. Keyframes every $k$ ticks add back one full snapshot's cost periodically,
amortized as $S_{\text{full}}/k$ extra per tick.

```python
# delta against the last snapshot the client acknowledged
delta = [(i, e) for i, e in enumerate(world) if e != baseline[i]]
packet = header(ack_id) + serialize(delta)   # only the changed entities
if tick % keyframe_interval == 0:            # periodic full re-ground
    packet = header(ack_id) + serialize(world)
```

<Callout type="why">
"Store a baseline, transmit only the difference" is the same instinct behind sending **gradients** instead of
whole models in distributed training, and behind **sparse updates** that touch only the parameters that moved.
Communicating the change rather than the whole state is a core efficiency pattern far beyond games.
</Callout>

## The same idea, four different worlds

<ExampleCard domain="Video compression" icon="🎥">
A video isn't a pile of full images. Codecs send occasional complete **keyframes** (I-frames) and, between them,
**delta frames** (P-frames) that encode only what moved since the last one. A still scene compresses to almost
nothing — identical to a calm game world producing tiny deltas.
</ExampleCard>

<ExampleCard domain="Backups" icon="💾">
A full backup copies everything; an **incremental backup** copies only the files changed since the last one. You
take a full baseline occasionally and stream cheap diffs in between — the keyframe-plus-delta rhythm applied to
storage instead of game state.
</ExampleCard>

<ExampleCard domain="Version control" icon="🔀">
Git doesn't store a fresh copy of your project per commit; it records the **diff** from the previous state and
deduplicates unchanged content. A thousand commits over one big repo stay small because each one transmits only
the change — delta compression as the foundation of a tool.
</ExampleCard>

<ExampleCard domain="Financial data" icon="📈">
A market data feed doesn't resend the whole order book on every update. It pushes **incremental** ticks — this
price changed, that quantity updated — against a periodic full snapshot, so millions of subscribers stay in sync
on a modest pipe. Same delta-plus-keyframe design.
</ExampleCard>

<Callout type="note">
Deltas trade robustness for size: lose a delta and your baseline is wrong until the next keyframe, so engines
balance keyframe frequency against bandwidth. More keyframes recover faster from loss but cost more — exactly
the slider you nudged in the demo.
</Callout>
