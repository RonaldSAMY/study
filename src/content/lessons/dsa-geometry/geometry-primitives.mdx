## The hook: which way does the road bend?

A self-driving car is tracing a route. At each corner it must answer a deceptively simple question:
am I about to turn **left**, turn **right**, or keep going **straight**? You could reach for
`atan2`, subtract angles, and worry about the seam where the angle wraps around. Or you could compute
**one subtraction-heavy number** whose *sign* answers the question exactly — no angles, no division,
no floating-point trigonometry. That number is the 2-D cross product, and it is the atom that every
other geometry algorithm in this track is built from.

<Callout type="intuition" title="The one-sentence idea">
The signed area of the parallelogram spanned by AB and AC is positive when C is to the **left** of the
ray A→B, negative when it is to the **right**, and zero when the three points are **collinear**.
</Callout>

## Intuition: turn the vectors you already know into a compass

You already met the dot product (how much two vectors agree) and the cross product (how much they
disagree, and in which rotational sense) in the vectors lesson and in Game Dev's dot & cross product.
Here we specialise the cross product to the plane. Given three points A, B, C, form two vectors that
start at A:

- $\vec{AB} = B - A$ — the direction you are travelling,
- $\vec{AC} = C - A$ — the direction to the third point.

Their cross product in 2-D is a single scalar (the z-component of the 3-D cross product):

$$
\operatorname{cross}(A, B, C) = (B_x - A_x)(C_y - A_y) - (B_y - A_y)(C_x - A_x).
$$

Geometrically it is the **signed area** of the parallelogram on those two vectors. The *magnitude*
tells you how far C sits from the line; the *sign* tells you which side. Sweep C from one side of the
line to the other and the number glides through zero exactly as C crosses — that zero is collinearity.
Because the whole test is additions, subtractions and multiplications on the coordinates, it is exact
for integer inputs: no rounding, no `atan2`, no special cases at the angle wrap-around.

## How it works

Take A = (0, 0), B = (4, 0), and a moving C. The base vector is $\vec{AB} = (4, 0)$, so the formula
collapses to $4 \cdot C_y$ — the sign is simply the sign of C's height:

```
A=(0,0)  B=(4,0)
C=(1, 2):  cross = (4)(2) - (0)(1) =  8   > 0  → left  (counter-clockwise)
C=(1,-2):  cross = (4)(-2) - (0)(1) = -8  < 0  → right (clockwise)
C=(2, 0):  cross = (4)(0) - (0)(2) =  0        → collinear
```

Reduce the raw value to just its sign and you get the **orientation** predicate, the workhorse of
computational geometry:

  return (a.x - o.x) * (b.y - o.y) - (a.y - o.y) * (b.x - o.x);

function orientation(o, a, b) {
  const v = cross(o, a, b);
  return v > 0 ? 1 : v < 0 ? -1 : 0;
}

function triangleArea(a, b, c) {

## The real implementation

Every file in this track opens with the same primitive. Here it is verbatim from the convex-hull
source, with the sign convention spelled out:

```ts
// /dsa/23-geometry/convex-hull.ts
/**
 * Cross product of vectors OA and OB.
 * Returns positive if OAB makes a counter-clockwise turn,
 * negative if clockwise, zero if collinear.
 */
export function cross(O: Point, A: Point, B: Point): number {
  return (A.x - O.x) * (B.y - O.y) - (A.y - O.y) * (B.x - O.x);
}

export function distanceSquared(a: Point, b: Point): number {
  return (a.x - b.x) ** 2 + (a.y - b.y) ** 2;   // compare distances without a sqrt
}
```

Run the full file: `npx tsx 23-geometry/convex-hull.ts`.

Two habits worth stealing: keep distances **squared** whenever you only need to compare them (a
square root is slow and irrational), and treat `cross` as the single source of truth for "which side."

## Complexity

Each primitive is a fixed handful of arithmetic operations:

$$
T_{\text{cross}} = O(1), \qquad T_{\text{area}} = O(1), \qquad T_{\text{orientation}} = O(1).
$$

| Primitive | Formula | Time | Notes |
|-----------|---------|------|-------|
| Cross product | $(B-A)\times(C-A)$ | $O(1)$ | exact for integers |
| Orientation | sign of cross | $O(1)$ | left / right / straight |
| Triangle area | $\lvert\text{cross}\rvert / 2$ | $O(1)$ | Shoelace, one triangle |
| Distance² | $(a-b)\cdot(a-b)$ | $O(1)$ | avoids the square root |

The magic is not the speed of one call — it is that hulls, intersections, and polygon tests are just
**many** of these $O(1)$ tests stitched together.

## Common pitfalls & interview tips

- **Floating-point zero is a lie.** With `double` coordinates the cross product is rarely *exactly* 0
  for near-collinear points. Compare against a small epsilon (the source uses `1e-9`), or use integer
  coordinates so the test is exact.
- **Sign flips with the y-axis.** The formula gives positive = counter-clockwise when y points **up**
  (math convention). On a screen where y points **down**, the same number reads clockwise. Pick one
  convention and hold it across the whole codebase.
- **Order matters.** `cross(A, B, C)` and `cross(A, C, B)` are negatives of each other — swapping the
  last two points flips the answer. Fix the argument order to the "pivot, from, to" pattern.
- **Overflow on big integers.** Coordinates in the millions squared can exceed 32-bit range; in
  JavaScript stay under $2^{53}$, or reach for BigInt when multiplying large fixed-point values.
- **Interview cue:** if a problem says "left turn," "same side," "counter-clockwise," or "is the point
  above the line," reach for the cross product's sign before you even think about slopes.

## Practice

1. **Easy:** Compute `cross((0,0),(2,0),(2,2))` by hand and state the orientation.
2. **Easy:** Show that `cross(A,B,C) = -cross(A,C,B)` for any three points.
3. **Medium:** Use only orientation tests to decide whether point P is inside triangle ABC (hint: it
   must be on the same side of all three edges).
4. **Medium:** Given a polygon's vertices in order, compute its signed area and report whether they are
   listed clockwise or counter-clockwise.
5. **Hard:** Sort a set of points by polar angle around a pivot using `cross` as the comparator, with no
   call to `atan2`, breaking ties by distance.
