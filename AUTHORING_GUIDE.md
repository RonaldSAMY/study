# Lesson Authoring Guide (read this fully before writing)

You are authoring interactive lessons for an Astro + Preact + MDX + KaTeX learning site
("Mathematics for Machine Learning", beginner → advanced). Match the established pattern EXACTLY.

## Reference files to read first (do this before writing anything)
- `src/content/lessons/linear-algebra/vectors.mdx` — the canonical lesson. Copy its structure, tone, and MDX usage.
- `src/components/demos/VectorPlayground.tsx` — the canonical interactive island. Copy its canvas conventions.
- `src/components/Callout.astro`, `ExampleCard.astro`, `KeyTakeaways.astro`, `Quiz.tsx` — shared components & their props.
- `src/data/curriculum.ts` — the full list of tracks, lesson ids, titles, and summaries. Use it to pick correct
  `prereqs`/`leadsTo` slugs. **DO NOT EDIT THIS FILE.**

## Where lessons live & how "ready" works
- A lesson file goes at `src/content/lessons/<trackId>/<lessonId>.mdx`.
- A lesson automatically becomes "ready" (renders, loses its "soon" badge) the moment its MDX file exists.
  You therefore NEVER edit any shared file — you only CREATE new files. This is what lets many authors work in parallel.

## Frontmatter (YAML) — required exactly these fields
```yaml
---
title: <Lesson Title>            # must match the title in curriculum.ts
track: <trackId>                 # e.g. algebra-1, calculus-3, linear-algebra
summary: <one sentence>          # can match/improve the curriculum summary
difficulty: beginner            # beginner | intermediate | advanced (progress within the track)
estMinutes: 12
prereqs:                         # slugs "<trackId>/<lessonId>" of lessons this builds on
  - <trackId>/<lessonId>
leadsTo:                         # slugs of the natural next lesson(s)
  - <trackId>/<lessonId>
---
```
Set `prereqs` to the previous lesson(s) in the same track and any cross-track foundation it relies on.
Set `leadsTo` to the next lesson in the track. The FIRST lesson of a track may use the last lesson of the
prior foundational track as a prereq (keep the global flow connected). Use REAL slugs that exist in curriculum.ts.

## Imports (relative paths from a lesson file are THREE levels up)
```jsx
import Callout from '../../../components/Callout.astro';
import ExampleCard from '../../../components/ExampleCard.astro';
import KeyTakeaways from '../../../components/KeyTakeaways.astro';
import Quiz from '../../../components/Quiz.tsx';
import YourDemo from '../../../components/demos/YourDemo.tsx';
```

## Required lesson structure (same order as the reference)
1. `## The hook: ...` — a vivid, concrete real-world opening that motivates the concept.
2. `<Callout type="intuition" title="The one-sentence idea">` — the concept in ONE sentence.
3. `## Intuition` — plain-language explanation, building on prior lessons. Reference what the learner already knows.
4. `## Play with it` — embed your interactive island `<YourDemo client:visible />` + a guiding sentence + a `<Callout type="tip">`.
5. `## The formal definition` — KaTeX math, introduced gently. Include a `<Callout type="why">` connecting it to machine learning.
6. `## The same idea, ...` — **THREE to FOUR `<ExampleCard domain icon>`**, each from a DIFFERENT real-life domain.
7. `## Quick check` — `<Quiz client:visible questions={[...]} />` with 3 questions (mix conceptual + one small computation).
8. `<KeyTakeaways points={[...]} />` — 5 bullet strings (HTML like `<strong>` allowed).
9. A one-line closing that points to the next lesson.

## Component APIs (do not invent props)
- `<Callout type="intuition|tip|warning|why|note" title="optional">...</Callout>`
- `<ExampleCard domain="Domain name" icon="🌍">...</ExampleCard>`
- `<KeyTakeaways points={["<strong>x</strong> ...", "..."]} />`
- `<Quiz client:visible questions={[{ q: "...", options: ["a","b","c","d"], answer: 1, explain: "..." }]} />`

## The example-variety rule (IMPORTANT — the user cares about this)
- Every lesson shows 3–4 real-world examples from DISTINCT domains (navigation, biology, finance, music, sports,
  cooking, gaming, medicine, climate, social media, logistics, art, etc.).
- NEVER reuse the same scenario across lessons. Across the whole site, examples should feel fresh each time.
  The vectors lesson already used: navigation/GPS, physics forces, RGB color, recommender systems — avoid repeating those exact ones.

## Interactivity requirement (every lesson needs a demo)
- Every lesson MUST have a working interactive element in "Play with it". Prefer a small Preact island
  (canvas or SVG) that lets the learner drag/slide/simulate and see the concept change live.
- Build islands in `src/components/demos/`. Name them uniquely and descriptively (e.g. `QuadraticExplorer.tsx`,
  `BayesGridDemo.tsx`). Avoid generic names that could collide with other tracks.
- Reuse the canvas conventions from VectorPlayground: `devicePixelRatio` scaling for crispness, responsive
  sizing on `window.resize`, pointer events with `class="touch-none"`, redraw-on-state via `useEffect`,
  clean up any `requestAnimationFrame` on unmount. Colors: indigo `#4f46e5`, sky `#0ea5e9`, emerald `#10b981`.
  Surrounding UI uses the Tailwind CSS-variable classes: `bg-surface`, `bg-surface-2`, `border-border`,
  `text-muted`, `text-text`, `bg-brand`, `text-white`, `bg-brand-soft`. Must work on mobile (touch) and resize cleanly.
- If two lessons in your track share a natural visualization, you MAY create one parameterized island and use it in both.

## KaTeX rules
- Inline math: `$ ... $`. Display math: `$$ ... $$` on its own lines.
- Use standard LaTeX. In matrices/aligned use `\\` for row breaks. Keep it valid — invalid KaTeX breaks the build.

## Book citations (do NOT add these yourself)
Mapping lessons to the "Mathematics for Machine Learning" book is handled centrally in
`src/data/bookMap.ts`, and the "📖 In the book" badge is rendered automatically by LessonLayout.
Do NOT add any book reference to lesson frontmatter or body — just write the lesson.

## Hard rules
- Do NOT run `npm run build`, `astro build`, `astro sync`, or `astro dev` — a parallel process handles builds; concurrent runs race.
- Do NOT edit `curriculum.ts` or any file outside the new lesson MDX files and new demo islands you create.
- Keep difficulty progressing within the track (early lessons `beginner`, later ones `intermediate`/`advanced`).
- Write genuinely clear, correct, friendly explanations. Intuition before formalism, always.
